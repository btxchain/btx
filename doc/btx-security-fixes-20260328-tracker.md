## Disposition Key

- `fixed`: implemented in this branch
- `partial`: some concrete hardening landed, but the full report claim still needs follow-up
- `deferred-consensus`: real or plausible issue, but a safe complete fix still needs activation plumbing, a fork rule, or more protocol work
- `architectural`: design/privacy/storage tradeoff, not a bounded patch for this PR
- `not-reproduced`: the report claim did not hold as stated after code review, or the live consensus path already blocks it
- `source-incomplete`: the RTF grouped or counted findings without enough per-item detail to map one-to-one

## TDD Execution Framework

The remainder of this tracker is now used as the implementation control document for the branch. Every unresolved row must fit one of the following execution lanes before it can move to `fixed`.

- `lane-a-consensus`: add a failing consensus or proof-check test first, gate the fix at `61000`, then add `60999` / `61000` / `61001` boundary coverage plus reorg coverage if state is consumed or created.
- `lane-b-policy`: add a failing mempool / miner / wallet policy test first, gate policy at `61000` where practical, and add miner-template plus disconnected-block / reaccept coverage where applicable.
- `lane-c-state-redesign`: add a deterministic reproducer plus restart / disconnect / rebuild coverage first; do not mark `fixed` until the state machine is transactional or crash-recoverable on every touched store.
- `lane-d-source-repro`: add a minimal reproducer or source-level disprover first; do not patch consensus based only on a grouped RTF claim.

Common TDD loop for all remaining items:

1. Reproduce the claim from the RTF against current `btx-node`.
2. Land a failing unit / functional / restart test that isolates the issue.
3. Implement the smallest code change that closes the failing surface.
4. Add activation-boundary coverage at `60999`, `61000`, and `61001` when any rule changes after launch.
5. Add reorg / restart / snapshot rebuild coverage for any stateful shielded store transition.
6. Only then update the disposition in this tracker and PR notes.
