# s1 — Standard C/C++ dependency supply-chain audit (boost, libevent, zeromq, sqlite, leveldb, zlib, crc32c)

Scope: full per-dependency provenance + version-pinning + CVE-reachability for BTX's STANDARD
C/C++ library dependencies. Read-only on repo. No production assets touched. Builds on c3.md
(prior inventory) — this report DRILLS DOWN on the 7 named standard libs, adds leveldb/zlib/crc32c
(c3 only mentioned in passing), and RESOLVES the open sqlite-CVE-backport and Alpine-version questions.
All CVEs cited from primary sources; none fabricated. Today = 2026-06-02.

---

## 0. The TWO build stacks (which actually ships?)

- **`depends/`** (Bitcoin-Core cross-compile, SHA256-pinned tarballs) — Guix/release-style builds ONLY.
- **`contrib/docker/Dockerfile`** = **THE SHIPPED IMAGE** (per contrib/docker/README.md "headless node";
  `alpine:3.22` builder + final, `apk add boost-dev libevent-dev sqlite-dev zeromq-dev`). Links **Alpine 3.22 apk
  versions**, NOT depends/. ZMQ ON. zlib NOT linked (no `-lz`, no `find_package(ZLIB)` — confirmed).
- **`contrib/docker/Dockerfile.static`** = a STATIC variant (not the documented ship image). It is the ONLY
  build that explicitly pulls **`zlib-static`** and links **`-lz`** (transitive, for static libevent/sqlite).
  ZMQ OFF here.
- **`contrib/docker/Dockerfile.tests`** = CI test image (`BUILD_TESTS=ON`, ZMQ OFF). Not shipped.

> Conclusion: the SHIPPED daemon's standard-lib versions = **Alpine 3.22 apk** (sqlite 3.49.2, boost 1.84,
> libevent 2.1.12, zeromq 4.3.5) + **in-tree leveldb 1.22 / crc32c 1.1.0** (compiled from src/, version-independent
> of any package manager). depends/ versions ship NOWHERE in the container path.

## 2. KNOWN-CVE CHECK (exact shipped versions)

### sqlite — RESOLVES c3's open question
- **Shipped (Alpine `sqlite-libs 3.49.2-r1`): the CVE-2025-6965 fix IS backported.** Snyk's Alpine 3.22 advisory
  states CVE-2025-6965 affects "versions **<3.49.2-r1**" ⇒ 3.49.2-r1 is the FIXED Alpine build. (Upstream fix is
  3.50.2, but Alpine backported it into the -r1 rebuild.) Also clears CVE-2025-29087 (concat_ws, fixed upstream
  3.49.1) and CVE-2025-7458 (sz*nBig, fixed 3.49.1). ⇒ **SHIPPED sqlite is CLEAN for all 2025 CVEs.**
- **depends/ `sqlite 3460100` (3.46.1) is STILL VULNERABLE** to CVE-2025-6965 (CVSS 9.8/critical per NVD; was
  7.2 in c3 — NVD now scores it 9.8) and CVE-2025-29087. depends/ only matters for Guix/depends release builds,
  NOT the container. Reachability: SQLite is the descriptor-**wallet** DB — LOCAL, attacker would need to control
  SQL run against the wallet (not network/consensus data) ⇒ **LOW exploitability** even on the depends path.
  Newer note: CVE-2025-70873 (zipfile-extension info-disclosure, ≤3.51.1) is N/A — BTX builds sqlite with
  extensions/rtree/fts disabled (sqlite.mk `--disable-dynamic-extensions`); zipfile not compiled.
- Sources: https://www.sqlite.org/cves.html ; https://nvd.nist.gov/vuln/detail/CVE-2025-6965 ;
  https://security.snyk.io/vuln/SNYK-ALPINE322-SQLITE-10872094

### zlib — NEW (c3 did not assess zlib)
- **Shipped image links NO zlib** (dynamic Dockerfile: no `-lz`, no `find_package(ZLIB)`, only Qt internal zlib
  which is GUI/N-A). The **static** Dockerfile pulls Alpine **zlib 1.3.2-r0**, which is PAST both relevant CVEs:
  CVE-2022-37434 (inflate heap over-read, fixed 1.2.13) and CVE-2023-45853 (MiniZip integer overflow, fixed 1.3.1).
  ⇒ **CLEAN.** (BTX wouldn't use MiniZip anyway.) The `zlib.h` hits in src/test/smile2_*adversarial_tests.cpp are
  TEST files, not the daemon.
- Sources: https://nvd.nist.gov/vuln/detail/CVE-2023-45853 ; https://www.sentinelone.com/vulnerability-database/cve-2022-37434/

### leveldb 1.22.0 — NEW assessment
- **No CVE affects the core google/leveldb C++ library at 1.22.** (The "leveldb CVEs" in databases —
  CVE-2026-25747 / CVE-2021-25915 — are the **Apache Camel `camel-leveldb` Java** wrapper's Java-deserialization
  bug, a different product; N/A to the C++ library.) leveldb 1.22 = the same pin Bitcoin Core ships; it reads
  ONLY node-local chainstate/blockindex it wrote itself — not attacker-controlled serialized data. **CLEAN.**
  Reachability of the DB engine: consensus-adjacent (chainstate) but inputs are self-produced ⇒ not a remote
  vector. Source: https://github.com/google/leveldb/security/advisories (no advisories);
  https://security.snyk.io/vuln/SNYK-JAVA-ORGAPACHECAMEL-15353482 (the Java false-cognate).

### crc32c 1.1.0 — NEW assessment
- Pure checksum dispatcher (Bitcoin Core subtree, used for block/index integrity). **No CVE history.** Stable,
  no parsing of untrusted structured data. **CLEAN.**

### boost 1.81 (depends) / 1.84 (shipped) — confirmed
- BTX uses only header/light components (multi-index, signals2, system, filesystem, program_options — see final
  Dockerfile `boost-system/filesystem/program_options`). The only notable Boost CVE, CVE-2022-21658
  (Boost.Filesystem `remove_all` TOCTOU), was fixed in **1.80** ⇒ both 1.81 and 1.84 are past it. No
  high-severity CVE reachable from BTX's usage at either version. **CLEAN.** Bumping depends 1.81→1.84 = hygiene,
  not a fix. Source: https://www.cvedetails.com/product/13033/Boost-Boost.html

### libevent 2.1.12 (both stacks) — confirmed
- All historic evdns/parser CVEs (CVE-2016-10195/-10196/-10197, and the pre-2.1.6 `evutil_parse_sockaddr_port`
  stack overflow) are fixed by **2.1.6-beta** ⇒ 2.1.12 includes them. **No CVE published against 2.1.12.** It is
  the **last upstream release (Jul-2020)** — there is no newer version to bump to; Alpine 2.1.12-r8 carries distro
  patches. Reachability is moot (no live CVE). **CLEAN; track upstream only.**
  Source: https://www.cvedetails.com/vulnerability-list/.../Libevent ; libevent ChangeLog.

### zeromq 4.3.5 (both stacks) — confirmed
- The libzmq RCE/overflow/DoS CVEs (CVE-2019-13132 UAF, CVE-2021-20234/-20235/-20236) are all fixed **<4.3.3** ⇒
  4.3.5 unaffected. ZMQ in BTX is an OPTIONAL **outbound** pub/sub notification interface (operator binds it
  locally), not a consensus/inbound-peer path; also OFF in the static + tests builds. **CLEAN.**
  Source: https://www.cvedetails.com/product/52489/

## 4. RECOMMENDATIONS (prioritized by reachability)

1. **[LOW-MED, hygiene] Bump depends/ sqlite 3.46.1 → ≥3.50.2** (sqlite.mk: version `3460100`→`3500200`, new
   sha256). Closes CVE-2025-6965/-29087 for any Guix/depends release. Low priority because the SHIPPED container
   is already patched (Alpine 3.49.2-r1) and the path is wallet-local. RE-PIN.
2. **[LOW-MED, reproducibility — carried from c3] Pin Alpine apk versions in the Dockerfiles**
   (`apk add boost-dev=1.84.0-r3 libevent-dev=2.1.12-r8 sqlite-dev=3.49.2-r1 zeromq-dev=4.3.5-r2 …` + a pinned
   repo snapshot). Today these float: the shipped binary's standard-lib versions are decided at `docker build`
   time, so a future bad Alpine package (or a silent downgrade of the sqlite -r1 backport) is auto-pulled and the
   build is non-reproducible. This is the single most impactful supply-chain item for the SHIPPED artifact.
   VERIFY-HASH/RE-PIN.
3. **[LOW] Record upstream commit + SHA256 for the in-tree leveldb (1.22.0) and crc32c (1.1.0) subtrees**
   (and secp256k1/minisketch). The squashed "Publish BTX snapshot" history means a reviewer cannot diff the
   vendored copy against canonical google/leveldb @ the 1.22 tag to detect tampering. Add a SHA256SUMS / commit
   note like dilithium has. VERIFY-HASH.
4. **[LOW, optional hygiene] Bump depends/ boost 1.81 → 1.84** to match the shipped Alpine and drop stale headers.
   No CVE driver. BUMP.
5. **[INFO] libevent is the only NETWORK-facing standard lib AND is EOL (2.1.12, last release 2020).** No open CVE
   today, but it has the highest reachability if one ever lands ⇒ put it on a CVE watch-list; there is no newer
   upstream to bump to, so monitoring is the only action.

### Highest-risk item (this report's scope)
**#2 — unpinned Alpine apk versions in the shipped Dockerfile.** It is the only item touching the ACTUALLY-SHIPPED
binary: the daemon's sqlite/boost/libevent/zeromq versions are non-deterministic and silently track upstream
Alpine, so the (currently-clean) shipped state is not guaranteed and not reproducible. Everything else is either
already-patched-as-shipped, not-linked, not-compiled, or wallet-local. (Note: the absolute highest BTX
supply-chain risk overall remains c3's bespoke unaudited matmul/shielded crypto + the fail-open PQ RNG — but those
are OUT of this report's "standard C/C++ library" scope.)
