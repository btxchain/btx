# d4 — Miner-targeted economic attacks + the BTX-NOVEL MatMul SERVICE-CHALLENGE subsystem (G5)

Empirical regtest on `btx-node:pr214` (default regtest = MatMul PoW active, chain_guard OFF,
future-drift cap OFF). Namespace `rtD4-*`, `/tmp/rtD4/`. Protected production / pf3-*/bft-*/evx*/btxn*
assets untouched. No git commit.

## TL;DR
- **What the service-challenge subsystem IS:** an *off-chain* proof-of-MatMul-work *anti-spam / rate-limit
  gate* (a Hashcash/CAPTCHA replacement), NOT an on-chain reward/issuance mechanism. `getmatmulservicechallenge`
  issues a challenge envelope; a client runs `solvematmulservicechallenge` (real matmul nonce-grind) to produce
  (nonce64, digest); `redeemmatmulserviceproof` verifies the proof and *atomically consumes* the challenge_id in
  a persistent registry so it can only be redeemed once. **No BTX coins are minted or paid** — so the classic
  "steal block rewards / claim rewards without work" framing does not apply. The economic value at stake is the
  operator's *rate-limit/anti-spam guarantee*, not coin.
- **Core mechanism is SOUND.** Double-redeem (single node), batch self-double-redeem, restart-replay, proof
  forgery, and difficulty-downgrade tampering are all correctly REJECTED (empirically). The challenge_id binds
  every field; seeds/target are re-derived; redeem is mutex+file-lock atomic and persists across restart.
- **NEW finding (med): registry eviction-DoS.** The registry is an LRU capped at 10,000 entries, evicting the
  **lowest `issued_at` regardless of redeemed/pending state**. An attacker who triggers ~10,000 challenge
  issuances (≈2.4 min, no PoW required by the issuer) evicts ALL older *pending* (unredeemed) challenges of
  legitimate users — their already-solved proofs then return `unknown_challenge` and are denied. Confirmed
  end-to-end. This is an availability/grief DoS on the service, not a double-spend.
- **NEW finding (low): `verify` vs `redeem` API footgun.** `verifymatmulserviceproofs` returns
  `valid:true,reason:ok` for an *already-redeemed* proof (it reports cryptographic validity, exposing redemption
  state only in a separate `redeemed` field). An app that gates on `valid` instead of using `redeem` (or checking
  `redeemed`) is trivially replayable. Documentation/contract hardening.
- **NEW finding (low): issuance not chain-guard-gated but createNewBlock-coupled.** `getmatmulservicechallenge`
  builds a full block template (`createNewBlock()`) per call — couples the off-chain anti-spam service to
  chain/mining health and makes each issuance ~110 ms (template build dominates, not registry I/O).
- Mission items 2/3/4 (orphan amplification, chain_guard griefing, difficulty-profitability): re-confirm b5/a7;
  no new *profitable* variant found. Details below.

## PART A — MatMul SERVICE-CHALLENGE subsystem (primary target)

### Mechanism (code: src/rpc/mining.cpp ~1640–6700, src/init.cpp:516)
- `challenge_id = SHA256(domain || binding_hash || salt || anchor_hash || anchor_height || issued_at ||
  expires_at || target_ms || validation_ms || propagation_ms)` (`ComputeMatMulServiceChallengeId`, :2921).
  `salt = GetRandHash()` per issuance → every issuance is unique & unpredictable (blocks precompute).
- `seed_a/seed_b = SHA256(domain || label || challenge_id || anchor_hash)` and `merkleroot` likewise derived
  from `challenge_id` → the matmul problem is fully bound to the challenge; you cannot reuse a proof under a
  different challenge.
- Proof verification (`EvaluateMatMulServiceProof`, :4084): `matmul::VerifyCommitment` (digest≤target) +
  `matmul::Verify` (Freivalds transcript). This is the SAME matmul soundness b1 audited (REFUTED forgery,
  2^-62 sound). Solve = real `matmul::Solve` nonce grind.
- Anti-replay = the persistent registry `std::map<uint256 challenge_id, {issued_at,expires_at,redeemed,
  redeemed_at}>` on disk (`matmul_service_challenges.dat`), guarded by an in-process mutex AND an OS file lock
  (`AcquireMatMulServiceChallengeRegistryFileLock`, 200×25 ms retries). Default scope = node-local file; a
  shared `-matmulservicechallengefile` lets a cluster share it.
- Redeem (`BuildMatMulServiceRedeemResult`, :4167): recompute challenge_id → must be `issued_by_local_node`
  (present in registry) → not already `redeemed` → not expired → proof `transcript_valid` →
  `RedeemMatMulServiceChallenge` flips `redeemed=true` under the locks and fsyncs.

### Vectors attempted
- **Double-redeem (single node), same proof twice** — REJECTED. 1st: `valid:true reason:ok redeemed:true`.
  2nd: `valid:false reason:already_redeemed`. (empirical)
- **Batch self-double-redeem** (`redeemmatmulserviceproofs` with the same proof ×3 in one call) — REJECTED.
  Result: `by_reason {ok:1, already_redeemed:2}` — each item takes the lock sequentially, so only the first
  consumes. (empirical)
- **Restart-replay** (redeem, restart node, re-redeem) — REJECTED. After `docker restart`, the redeemed flag is
  loaded from disk: `reason:already_redeemed redeemed:true`. Persistence is correct. (empirical)
- **Difficulty-downgrade forgery** (tamper `solve_time_target_s` to inflate the target / lower work) — REJECTED.
  Parse recomputes the expected difficulty from the anchored chain state: `-8 invalid service challenge:
  difficulty_resolution does not match the anchored network conditions`. Also `challenge_id` mismatch would
  re-randomize the seeds, invalidating any precomputed proof. (empirical)
- **Cross-binding replay** (reuse a login-proof for a different purpose/resource/subject) — REJECTED by
  construction: binding text feeds `binding_hash`→`challenge_id`→seeds; any change re-randomizes the problem.
  (code + tamper-test corroborated)
- **Proof forgery without work** — REJECTED. Solve/verify use the audited matmul path; b1 confirmed Freivalds
  is 2^-62 sound and the challenge is Fiat-Shamir bound to C'. (code; relies on b1's empirical 0/800k forgeries)
- **Concurrent-redeem race (single node), 16 parallel redeems of one solved proof** — REJECTED (atomic).
  Result: exactly **1×`valid:true,ok` and 15×`already_redeemed`**. The in-process mutex + OS file lock serialize
  load→check→write correctly; no TOCTOU double-redeem. (empirical)
- **O(n²) prune / 570 KB-rewrite resource amplification at the 10k cap** — REFUTED as a DoS. Injected a
  10,001-entry registry; a single issuance still took ~125 ms vs ~113 ms baseline (prune+rewrite adds ~12 ms).
  std::map iteration is cheap at 10k; cost is dominated by `createNewBlock()`. Not an amplification lever.
  (empirical)

### ★ NEW: registry eviction-DoS (med) — CONFIRMED end-to-end
`PruneMatMulServiceChallengeRegistry` (mining.cpp:2694) when `size > 10,000` repeatedly erases the entry with
the **minimum `issued_at`**, with **no regard for whether it is redeemed or still pending**. Issuance
(`RememberIssuedMatMulServiceChallengeLocked`, :2709) calls prune before each insert.

Consequence: an attacker who can cause ~10,000 challenge issuances (in the normal anti-spam flow this is just
hitting the protected endpoint 10k times — issuance requires NO proof-of-work from the requester) sweeps every
older still-pending legitimate challenge out of the registry. A victim who already solved their challenge then
gets `issued_by_local_node:false → reason:unknown_challenge` and is **denied**, despite holding a valid proof.

Repro (empirical, minimal):
1. Inject a registry at the 10,000-entry cap (9,999 far-future junk + 1 victim issued at real-now, so the
   victim is the global min `issued_at`). Victim status pre-flood: `issued_by_local_node:true` (redeemable).
2. Attacker issues ONE challenge → registry prunes the oldest = victim.
3. Victim redeem now: `issued_by_local_node:false, reason:unknown_challenge, valid:false`. PROVEN.

Cost vs damage:
- Attacker cost: ~71 issuances/s measured (8 concurrent RPCs) ⇒ **~10,000 issuances ≈ 2.4 min**; no PoW, no coin.
  In a real deployment the attacker just spams the operator's "request a challenge" endpoint.
- Damage: every legitimate user whose challenge was issued before the flood and not yet redeemed loses their
  solved work (denied service); high-difficulty (long solve_time) challenges are exactly the ones most painful
  to lose. Sustained flooding keeps the registry churning so any challenge older than ~10k issuances is unsafe
  to hold.
- NOT a double-spend: an evicted *redeemed* entry becomes `unknown` (rejected), so eviction cannot resurrect a
  redeemable state. Confirmed.

Mitigations:
1. **Evict redeemed/expired entries before pending ones** (prefer expiry-time ordering; never evict an
   unredeemed, unexpired challenge to make room for a new issuance — instead reject/queue issuance or raise the
   cap). The current min-`issued_at` LRU silently drops live obligations.
2. Make the cap configurable and large; or shard by `expires_at` so pending entries persist until they actually
   expire (expiry-pruning already exists and is the correct primary bound — the hard count cap is the problem).
3. Operationally: rate-limit `getmatmulservicechallenge` per client, and use short `expires_in_s` so honest
   pending entries clear quickly (reduces the window, doesn't fix the eviction-of-pending bug).

### ★ NEW: `verify` accepts already-redeemed proofs (low) — CONFIRMED
`verifymatmulserviceproofs` on an already-redeemed proof returns `valid:true, reason:ok` (with
`local_registry_status_checked:false` it doesn't even consult the registry; with it true it still says
`valid:true` and surfaces `redeemed:true` only as a side field). `valid` reflects cryptographic proof validity,
not consumption. An application that authorizes on `valid` (rather than calling `redeemmatmulserviceproof`, the
atomic primitive, or checking `redeemed`) is **replay-vulnerable**. This is an API-contract footgun, not a core
break — `redeem` is correct. Mitigation: document that only `redeem` provides anti-replay; consider renaming
`valid`→`proof_valid` and requiring callers to read `redeemed`; or have `verify` default to surfacing a clear
`already_redeemed` reason even when `valid` is true. (empirical)

### NEW: issuance coupling (low)
`getmatmulservicechallenge` → `BuildMatMulServiceChallengeResponse` calls `EnsureMining(node).createNewBlock()`
(:3084) but, unlike `getmatmulchallenge` (:5532), does NOT call `EnsureMiningChainGuardOrThrow`. So issuance is
not *directly* guard-gated, but it depends on a buildable block template — coupling the off-chain anti-spam
service to chain tip / IBD / mining health, and costing ~110 ms/issue (template build). If `createNewBlock`
itself refuses under guard/IBD (not testable on default-OFF regtest; code-reasoned), an a7-style outbound-peer
eclipse would also halt challenge *issuance*, a broader DoS than a7's mining-only pause. Worth confirming on a
guard-ON build. Mitigation: decouple issuance from full template build (only the prev-block header/anchor +
next difficulty are actually needed), and decide explicitly whether the anti-spam service should fail when
mining is paused.

### Refuted / robust (don't repeat)
- Double-redeem / batch-dup / restart-replay / difficulty-downgrade / cross-binding / proof-forgery — all
  correctly rejected (above).
- O(n²) prune + 570 KB rewrite at cap is not a resource-amplification DoS (~12 ms overhead at 10k).
- The registry being node-local does NOT enable cross-node double-redeem: each issuance has a random salt ⇒
  unique challenge_id ⇒ a given proof is redeemable only against the one node/cluster that issued it; the
  shared-file mode serializes redemption via the OS file lock (atomic by construction).
- **Crash-corruption-replay REFUTED:** persist uses write-`.new` + `Commit()` (fsync) + atomic `RenameOver`
  (mining.cpp:2615-2663), so a crash mid-write cannot corrupt the live file. And even if the file were
  externally corrupted, `LoadMatMulServiceChallengeRegistryLocked` quarantines it and CLEARS the in-memory map
  ⇒ entries become `unknown_challenge` (denied), never replayable. Corruption is a DoS-on-pending at worst,
  not a double-redeem.

### Open leads
- Confirm `createNewBlock` behaviour under chain_guard-ON / IBD for the issuance-DoS coupling (needs guard-ON build).
- Shared-file mode on a multi-host cluster: verify the file lock holds under NFS/network FS (FileLock semantics
  vary); a lock that silently no-ops on NFS would allow a cross-host redeem race. (Code uses fsbridge::FileLock;
  test on the intended shared medium.)
- The `redeemed`-blind eviction also means a flood of *attacker self-redeemed* long-expiry challenges pins slots
  until expiry — combine with item-1 to size the cap/expiry policy.

## PART B — Miner-targeted attacks (items 2–4): re-confirmation, no new profit

### Item 2 — Orphan/stale amplification (extends b5/D3)
Re-confirmed b5: selfish-mining steady-state revenue is the Eyal–Sirer hashrate-share function (threshold ≈1/3
at γ=0, →0 as relay/eclipse advantage γ→1); ASERT's fast retarget only shortens the loss-making transient
(marginally helps attacker), it does not change the threshold. PR#214's future-drift cap is NEUTRAL to selfish
mining (doesn't depend on future-timestamp freedom). No timestamp trick orphans honest miners without actually
out-producing blocks. NO new profitable variant beyond b5. (code-reasoned + b5's empirics)

### Item 3 — chain_guard griefing (extends a7)
a7 stands: eclipsing a victim's OUTBOUND peers (guard counts outbound only, default min 2) trips
`should_pause_mining` ⇒ `getblocktemplate`/`getmatmulchallenge` throw and mining halts (cheap, sustained,
local mining-availability DoS; guard ON by default mainnet, OFF on test chains). NEW adjacency (Part A): the
service-challenge *issuance* path is createNewBlock-coupled but NOT guard-gated — so the same eclipse may also
degrade challenge issuance via template-build failure (confirm on guard-ON build). No cheaper/novel trip than
a7's outbound-peer stall was found. (code + a7)

### Item 4 — Difficulty/profitability griefing
Re-confirmed b4/b5/c1: difficulty-raising (minimal-gap, ×1.017/block, ~40 blocks to ×2) is self-funded (attacker
pays full PoW at the rising difficulty) and self-healing (~50 honest blocks); the future-drift cap throttles the
*drop* lever to ×1.97/block and the 2 h wall-clock bound caps cumulative drift. No profitable
"force-difficulty-up-then-abandon" variant: the attacker burns proportional PoW and the chain self-heals.
Template/fee manipulation to make honest miners build suboptimal blocks: not reproduced as a profitable lever
(out of scope of the novel subsystem; generic and pre-existing). NO new finding.
