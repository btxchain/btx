# e2 — SHIELDED ZK SOUNDNESS / FROZEN-HEART (smile2 ct_proof, matrict range/balance)

Red-team cryptographer audit of BTX's bespoke shielded NIZK proof systems on the
consensus verification path. Focus: weak Fiat-Shamir ("Frozen Heart" class),
range-proof soundness, commitment binding, soundness-error/challenge-space.
Method: deep code review of the verifiers + an ASan/UBSan unit-test harness
(rte2_frozenheart_tests) that drives ProveCT/VerifyCT and the matrict verifier
directly (bypasses the c5 z_sendmany wallet segfault). Builds on c5 (impl-level
monetary checks PRESENT & correct), c4 (crash-surface), c3 (bespoke-crypto audit).
No production assets touched; rtE2- namespace, /tmp/rtE2, isolated docker.

## TL;DR
- **No Frozen-Heart / weak-FS forgery found on the consensus path — CONFIRMED
  empirically (ASan).** Both proof systems hash the FULL public statement AND every
  prover commitment/announcement into the challenge before responses are committed.
  Empirically (hardened smile2 CT): tampering output coin, value_balance/fee,
  anon_set pk, A matrix, or input coin-ring at verify time — and lifting a proof
  onto another statement — is EACH REJECTED by VerifyCT (e2_a1..a5, e2_b3 all pass,
  clean under ASan/UBSan). [§1, §2, EMPIRICAL]
- **MatRiCT (mainnet height <61000 path) is sound on all 4 targets:** complete
  nested Fiat-Shamir, range proof bound to the same value commitment as balance,
  huge challenge space (fixed-Hamming-weight ternary, |C| ≫ 2^200). [§2]
- **★ NEW (med): smile2 CT does NOT bound shielded note amounts to MAX_MONEY.**
  CONFIRMED empirically — a BALANCED CT proof with an output (or input) amount of
  2^60 sat (≈550× MAX_MONEY) VERIFIES on the active hardened path (e2_c1/c2). The
  base-4 32-slot encoding allows up to 2^64-1; neither ProveCT nor VerifyCT checks
  MAX_MONEY, and the SMILE path carries no separate range proof (v2_proof.cpp:942).
  NOT a standalone inflation break (the FIRST >MAX_MONEY value can't enter — shield
  ingress value_balance is MoneyRange-checked, egress likewise, pool = legit
  Σdeposits, per c5) but a real RANGE-INVARIANT GAP + latent risk; the matrict path
  enforces VALUE_BITS=51<MAX_MONEY, the smile path should match. [§4]
- **smile2 CT opening challenge is a MONOMIAL `±X^k`, |C| = 2·POLY_DEGREE = 256**
  (vs the membership proof's large ternary). Code analysis: this governs only the
  BDLOP commitment-OPENING binding; the aggregated value relation is pinned to
  negligible error by the UNCONDITIONAL pre/post-h2 binding digests (committed
  pre-challenge) + high-entropy alpha/gamma/opening challenges, so it is NOT a
  ~1/256-grindable inflation hole. LOW/defense-in-depth note for external audit.
  (E2-B2 grind = 0 forgeries/100, but it only exercised the prover-refusal path —
  see §3 caveat.) [§3]

## Targets & findings

### 1. ★ Frozen Heart / weak Fiat-Shamir — smile2 CT (active path)
Verifier (src/shielded/smile2/ct_proof.cpp `VerifyCT`, ~3083-3690) rebuilds the
transcript and re-derives every challenge. Absorbed BEFORE the first challenge
(`public_transcript`, 3204-3232): anon_set (`AppendAnonSetTranscript`; in the
HARDENED path also the shared A matrix), coin-ring digest (`AppendCoinRingDigest`
— all input commitments + account leaves + pks), all OUTPUT coin commitments
(t0 compressed + t_msg), the public_fee (= value_balance), num_inputs,
num_outputs. The prover's `fs_seed` must equal this recomputation (3220-3230) —
any change to a public input changes the seed ⇒ reject.

Per-round prover commitments are absorbed before their challenge:
- round-1 aux commitment (`AppendCtRound1Commitment`, 3266) + serial numbers ⇒
  tuple/coin-opening challenges, then `wi` (recovered round-1 responses) and a
  prover `round1_aux_binding_digest` ⇒ `seed_c0` ⇒ `c0_chal` (3290-3300).
- z0 + key-binding A·z0 + tuple responses + serial bindings + coin-opening
  binding ⇒ `gamma1` challenges; then `pre_h2_binding_digest`, `h2`, and
  `post_h2_binding_digest` ⇒ `seed_c` ⇒ final `c_chal` (3401-3456).

The prover-supplied binding **digests** (`round1_aux`, `pre_h2`, `post_h2`) are
NOT trusted: the verifier RECOMPUTES the underlying committed polynomials from
`aux_ck`, `proof.z`, `proof.aux_commitment.t0`, and `c_chal`
(`RecoverFullAuxMessageCommitments`) and re-checks each digest UNCONDITIONALLY
(3478, 3531, 3665). Because each digest was absorbed into the transcript BEFORE
`c_chal` is derived, the prover cannot adaptively choose responses after seeing
the challenge — the digest pins the round-1/relation values to the challenge that
hashes them. This closes the adaptive-soundness gap.

**`SeedIsPresent`/`!IsZero()` conditionals (c5's flagged suspects) — RE-CONFIRMED
NOT soundness holes:**
- `c0_chal`/`c_chal` are derived from the RECOMPUTED `seed_c0_check`/`seed_c_check`
  (3300, 3456); when `proof.seed_*` is present it is checked equal to the
  recomputation (3291, 3447) and rejected on mismatch ⇒ either way the challenge
  is the genuine recomputed value, never attacker-chosen.
- Setting `proof.omega=0` skips the weak-opening semantic check (3579) and
  `framework_omega=0` skips the `lhs==framework_omega` check (3675), BUT the
  UNCONDITIONAL `post_h2_binding_digest` check (3665-3671) passes the
  VERIFIER-computed `lhs` into the digest; with framework_omega=0 the prover's
  committed digest = Digest([psi_tmsg, 0]) ⇒ the check then forces `lhs == 0`,
  i.e. the relation must vanish — a condition the masked relation cannot satisfy
  for an unbalanced witness without effectively proving the true relation. The
  binding digest, not the omega equality, is the real enforcer (matches c5).

A subtlety I reviewed and judged SOUND: in the live-m1 layout the verifier uses
the (not-yet-validated) `proof.seed_c` to reconstruct `wi` (3279-3287) which feeds
the EARLIER `seed_c0` ⇒ a forward reference creating a hash fixed-point
`seed_c == f(... seed_c ...)`. The honest prover is a consistent fixed point; an
attacker cannot freely choose `c_chal` because they must satisfy the random-oracle
fixed-point equation. Not exploitable for challenge selection.

**Legacy SMILE path (`bind_anonset_context=false`) omits the A matrix from the FS
transcript** (`AppendAnonSetTranscript` line 193 `continue`). NOT consensus-
reachable: `bind_smile_anonset_context = IsShieldedMatRiCTDisabled(height)`
(validation.cpp:1154) and `nShieldedMatRiCTDisableHeight = 61000` (mainnet),
`=0` (regtest) ⇒ on every live network SMILE runs HARDENED (A bound) and below
61000 MatRiCT (not legacy-SMILE) is active. The "no mixed A" structural check
(3157) further constrains it. Flagged as a defense-in-depth note only.

### 2. MatRiCT range/balance/ring (mainnet height <61000) — SOUND
- **Range proof** (ringct/range_proof.cpp): per-bit OR-proof; each bit challenge
  hashes bit_index + bit_commitment + value_commitment + BOTH announcements a0,a1
  (`ComputeBitChallengeHash` 218-232); verifier recomputes a0,a1 from responses,
  challenge, statement and checks `c0+c1 == H(...)` over the full 256-bit digest
  (593-608). Statement = `value_commitment − Σ 2^i·bit_commitment_i` checked to be
  a zero-commitment; per-bit ∈{0,1}; `static_assert 2^VALUE_BITS > MAX_MONEY`,
  VALUE_BITS=51. No negative, no wraparound. Range proof is verified against the
  SAME `proof.output_commitments[i]` used by the balance proof (matrict.cpp:447-
  459) ⇒ binding not splittable.
- **Balance proof** (ringct/balance_proof.cpp): FS binds nonce + statement +
  input/output commitments + fee + tx_binding_hash + static lattice params;
  verifier recomputes (283-291).
- **Top-level** `ComputeProofChallenge` binds ring-sig seed + balance transcript +
  all range transcripts + output commitments + fee + tx_binding (matrict.cpp:53-
  64); `== proof.challenge_seed` (462). Each sub-transcript independently
  re-derived & checked, so the top-level binds verified values.
- **Challenge space**: `SampleChallenge` = fixed-Hamming-weight ternary,
  BETA_CHALLENGE=60 nonzero of POLY_N=256 ⇒ |C| = C(256,60)·2^60 ≫ 2^200 ⇒
  soundness error negligible. (The per-bit OR-proof's lattice scalar challenge
  ∈[−60,60] is small, but the OR-proof binding is over the full 256-bit hash, the
  standard MatRiCT design.)

### 3. smile2 CT challenge space — PARAMETER CONCERN (feeds external audit)
`HashToMonomialChallenge` (ct_proof.cpp:354-369): challenge = ±X^k,
k = hash[0] % POLY_DEGREE, sign = hash[1]&1 ⇒ **|C| = 2·128 = 256** for c0_chal
and c_chal. By contrast the membership proof uses `HashToTernaryChallenge`
(membership.cpp:77, P(0)=1/2,P(±1)=1/4 over 128 coeffs ⇒ huge |C|). The monomial
is deliberate (norm control: ||c·r||_∞≤1, ct_proof.h:104-110). The open question:
does the VALUE-BINDING relation inherit a ~1/256 knowledge-soundness error
(grindable by ~256 forgery attempts, each a fresh ProveCT re-randomizing the
transcript)? My read is that the relation soundness is amplified to negligible by
(a) the unconditional binding digests committing the relation value pre-challenge
and (b) the high-entropy alpha (DeriveRhoChallenges/HashToPoly), gamma
(HashToSlotChallenge), and opening (DeriveOpeningChallenges) challenges that make
the aggregated relation an identity over a large space — so 256 governs only the
commitment-opening binding, not the relation. EMPIRICAL grind test E2-B2 settles
it. Defense-in-depth: if the monomial is the sole relation gate, parallel-repeat
or widen it.

### 4. smile2 amount range — BINDING UNKNOWN (feeds external audit; the key
monetary-soundness gap for the active path)
- SMILE send path carries NO separate range proof; `V2SendWitness::IsValid`
  requires native `output_range_proofs.empty()` for the smile branch
  (v2_proof.cpp:942) and only checks output-coin DIMENSIONS
  (`HasCanonicalSmileOutputCoin` = t0.size==RAND_DIM, t_msg.size==1; 216-220) —
  NOT the committed amount's range.
- Amount is a base-4 vector over 32 NTT slots (2 bits/slot ⇒ 64-bit encodable
  range). `EncodeAmountToSmileAmountPoly` rejects amount<0 and >64-bit but has NO
  MAX_MONEY check; ProveCT enforces amount≥0 and Σ-balance but NOT
  amount≤MAX_MONEY (MAX_MONEY ≈ 2.1e15 ≈ 2^51 ≪ 2^64).
- The verifier's framework relation enforces SELECTOR binarity (one-out-of-many
  ring selection: `f[sel]²+c·f[sel]==0`, ct_proof.cpp:3648-3652) but I found NO
  per-output amount-digit ∈{0,1,2,3} constraint in VerifyCT. So whether the
  lattice relation forces each amount slot into a valid base-4 digit (and forbids
  >MAX_MONEY notes / modular-wrap amounts) is a SOUNDNESS property of the bespoke
  ZK that cannot be settled by code reading. E2-C1 empirically tests whether a
  balanced >MAX_MONEY note verifies at the CT layer.
- NOTE: c5 confirmed the TRANSPARENT-facing value_balance is MoneyRange-checked
  and the pool can't go negative — so a >MAX_MONEY note is only exploitable if it
  can be unshielded/realized; that egress is gated by value_balance+pool checks.
  This bounds (does not eliminate) the concern; it is the residual unknown c3/c5
  defer to external audit, now localized to "amount-digit range binding".

## NEW / unmitigated findings
- **smile2 CT notes are NOT bounded to MAX_MONEY (CONFIRMED empirically) | severity
  MED (range-invariant gap + latent inflation risk; not standalone-exploitable
  given c5's value_balance/pool checks) | §4.** repro: rte2_frozenheart_tests
  e2_c1 — `CTTestSetup::Create(16,1,1,{2^60},{2^60},...)`, ProveCT (hardened),
  VerifyCT ⇒ TRUE; the active SMILE path has no standalone range proof
  (v2_proof.cpp:942) and no per-output digit-range / MAX_MONEY constraint in
  VerifyCT, and ProveCT lacks a MAX_MONEY check. A >MAX_MONEY note is constructible
  internally (e.g. combine many MAX_MONEY notes via z→z). Bounded by: shield-in &
  unshield-out value_balance both MoneyRange-checked (c5), pool = legit Σdeposits.
  Mitigation: add an explicit per-output amount ≤ MAX_MONEY (and digit∈{0,1,2,3})
  constraint to the smile2 CT relation/validation — match the matrict path's
  VALUE_BITS=51 + static_assert 2^VALUE_BITS>MAX_MONEY. Also flag for c3/c5 +
  external audit (confirm the lattice relation can't be coaxed into realizing a
  >MAX_MONEY note's value, e.g. via overflow in any downstream int64 sum).
- **smile2 CT monomial challenge |C| = 256 | severity LOW (defense-in-depth) | §3.**
  Code analysis: the 256-space monomial governs only BDLOP commitment-opening
  binding; the aggregated value relation is pinned to negligible error by the
  UNCONDITIONAL pre/post-h2 binding digests + high-entropy alpha/gamma/opening
  challenges ⇒ NOT a ~1/256 grindable forgery. Mitigation (optional): widen /
  parallel-repeat the CT challenge, or have external audit confirm the relation
  soundness bound. (Empirical 256-soundness test needs a hand-built malicious
  proof; deferred.)

## Refuted / robust (so others don't repeat)
- Frozen-Heart / weak-FS on BOTH proof systems: full statement + all
  commitments/announcements absorbed before responses; digests recomputed &
  rechecked unconditionally; cross-statement lift impossible. (§1, §2)
- MatRiCT range/balance/ring soundness, commitment binding, challenge space: sound
  by construction (§2).
- smile2 `SeedIsPresent`/`omega`/`framework_omega` conditionals: not soundness
  holes (challenges recompute from transcript; relation pinned by UNCONDITIONAL
  binding digests). Re-confirms c5.

## z_sendmany SIGSEGV (c5) — characterization via the ASan build
The ASan/UBSan build is usable (built test_btx; the daemon was not needed for the
soundness work). Across 100s of ProveCT/VerifyCT calls on the hardened CT path
(1-in-1-out and multi-in/out via the existing smile2_ct suite, all under
ASan+UBSan) I observed **NO memory error in the core CT prove/verify relation
logic** — the path is memory-clean. ⇒ the c5 z_sendmany SIGSEGV is NOT in the
ProveCT/VerifyCT lattice relation itself; it is reached only through the WALLET
bridge / spend-path assembly (smile2/wallet_bridge.cpp + v2_send.cpp building a
real z→z bundle with a ≥16-note pool on an encrypted wallet), which the test
harness does not exercise. To root-cause it, run `z_sendmany` against an
ASan-instrumented DAEMON build (BUILD_DAEMON=ON + ENABLE_WALLET, same
-DSANITIZERS=address,undefined; force HAVE_SSE41=OFF/HAVE_AVX2=OFF/HAVE_X86_SHANI=
OFF to avoid the ASan-vs-asm SHA256 self-test SEGV seen here — a harness artifact,
not a node bug). Lead handed to c5/c4. (NOTE for builders: the SHA256 SSE4/SHANI
asm faults under ASan during the kernel::Context SHA256AutoDetect self-test; disable
those HAVE_* cache vars for any ASan build of this codebase.)
