# d2 — MatMul backend non-determinism → targeted consensus split (catalog G3)

Mission: can two BTX nodes on DIFFERENT matmul backends (CPU/Metal/CUDA/MLX) or different
CPUs/SIMD levels/build flags compute a DIFFERENT validity for the SAME block → split the
network along hardware lines? Answer: **NO — consensus verification is deterministic by
construction (single pinned pure-integer CPU path), empirically confirmed.** No HIGH/CRIT.
One LOW documentation/defense-in-depth note.

## Vectors attempted

- **Backend dispatch reaches consensus verification?** Traced every `Check*` entry from
  validation.cpp. validation.cpp (7995, 8263, 8399, 8531, 8552, 8561, 8585) calls ONLY the
  pure-CPU `CheckMatMulProofOfWork_{Phase1,Phase2,Phase2WithPayload,Freivalds,ProductCommitted}`
  + `CheckMatMulPreHashGate` in src/pow.cpp. Those call `matmul::transcript::CanonicalMatMul`,
  `matmul::freivalds::Verify`, `matmul::transcript::ComputeProductCommittedDigest`,
  `matmul::SharedFromSeed`, `matmul::noise::Generate` DIRECTLY. None of them ever touch
  `matmul::accelerated::*` / `backend::Kind` / GPU.
  RESULT: REJECTED(attack) — consensus is backend-agnostic; GPU path is unreachable from validation.

- **Where the backend (CPU/METAL/CUDA/MLX) dispatch actually lives.** `accelerated::ComputeMatMulDigest`
  / `SubmitMatMulDigestPreparedBatchForMining` / `ResolveMiningBackendFromEnvironment`
  (BTX_MATMUL_BACKEND env) are called ONLY from `SolveMatMul` (mining, pow.cpp 2696/2966) and
  `btx-genesis.cpp` (genesis mining tool). "MLX" is just a lowercase alias of METAL
  (backend_capabilities.cpp ParseKind, line 86). METAL/CUDA are `#if defined(BTX_ENABLE_METAL)`
  / `BTX_ENABLE_CUDA_EXPERIMENTAL` compile-gated (default-off on this Linux host → "disabled_by_build").
  RESULT: REJECTED(attack) — backend selection is mining-only, never consensus.

- **Mining self-defends: GPU result is CPU-confirmed before a block is emitted.** In SolveMatMul,
  a winning accelerated candidate is recomputed on the CPU canonical path
  (`ComputeDigestCpuFromPreparedInputs` / `CanonicalMatMul` / `ComputeProductCommittedDigest`,
  pow.cpp 3030, 3066–3076) using the SAME prepared inputs, and **`block.matmul_digest = accepted_digest`
  is the CPU digest** (3099/3107). Even if `BTX_MATMUL_CPU_CONFIRM=0` disables that confirm
  (ShouldCpuConfirmSolvedMatMulCandidates, pow.cpp 972), the emitted block must still pass the
  pure-CPU consensus `Check*` on every node *including the miner's own*. A buggy GPU backend can
  therefore only make the miner waste work / self-orphan — it can NEVER make a block that one
  backend accepts and another rejects.
  RESULT: REJECTED(attack) — GPU divergence is miner-local, not network-splitting.

- **Float anywhere in consensus?** `grep -E '(float|double|__m128|__m256|_mm_|fma)'` over
  field.cpp, matrix.cpp, transcript.cpp, freivalds.cpp, noise.cpp → ZERO hits. Field is pure
  integer mod p = 2^31−1 (field.h MODULUS=0x7FFFFFFF). No `-ffast-math` / `-march=native` /
  `-Ofast` in CMakeLists/configure. ASERT path (DeriveTarget/aserti3-2d) is `__int128`/256-bit
  integer per b4 LEARNINGS. No float in any consensus comparison.
  RESULT: REJECTED(attack) — no cross-platform FP non-determinism surface.

- **SIMD/reassociation/overflow divergence in the dot product.** field::dot has TWO compile-time
  paths: `ScalarDot` (x86, active here) and `NeonDot` (ARM `__ARM_NEON`). They use DIFFERENT
  accumulation groupings (Scalar: reduce every 4; Neon: 4-wide load, reduce per group, then
  `add(reduce64(acc), ScalarDot(tail))`). reduce64 is a double-Mersenne fold safe for all uint64
  (verified reduce64(UINT64_MAX)=3). Because every intermediate is reduced to a canonical residue
  mod p and field add is associative/commutative, grouping order is irrelevant to the result.
  EMPIRICAL: 2,000,000 random differential trials (lengths 1–600) + worst-case all-(MOD−1)
  len 1–1024 → ScalarDot == NeonDot == per-term reference, **0 mismatches**
  (/tmp/rtD2/dot_determinism.cpp). MultiplyBlocked is multi-threaded but partitions OUTPUT ROWS
  across workers (disjoint cells) and the per-cell k-accumulation order is fixed regardless of
  thread count.
  RESULT: REJECTED(attack) — x86 and ARM consensus dot paths are bit-identical.

- **Cross-build differential (EMPIRICAL).** Extracted the exact consensus integer arithmetic
  (reduce64, fadd, fmul, fdot, naive operator*, MultiplyBlocked with tiles 4/8 × workers 1/4/8/16)
  into /tmp/rtD2/consensus_arith.cpp and built 5 ways: `-O0 -march=x86-64`, `-O3 -march=x86-64`,
  `-O3 -march=native`, `-Ofast -march=native -ffast-math`, `-O2 -funroll-loops`. For n∈{8,16,24,32,64}
  random matrices, blocked(all tiles/workers) == naive in every case, and the combined digest was
  IDENTICAL (`52a6145589d676fb`) across all 5 builds and across 5 repeated runs each (thread-schedule
  fuzz). (Full node not built — heavy + shared 32 cores; the consensus arithmetic is the load-bearing
  part and is self-contained.)
  RESULT: REJECTED(attack) — no optimization/arch/FP-flag/thread-order divergence.

- **Existing in-tree determinism tests** corroborate: src/test/matmul_field_tests.cpp covers
  reduce64 edge/overflow cases (comments "gpu_reduce64_matches_cpu", "gpu_reduce64_max_uint64")
  and worst-case dot (n512, large); src/test/matmul_metal_tests.cpp has
  `metal_digest_matches_cpu_across_supported_dimensions` / `metal_product_digest_matches_cpu`
  / `metal_batch_digest_matches_single` asserting BACKEND digest == CPU digest (so the GPU kernels
  are held to the CPU canonical result as ground truth — exactly the right invariant).

## NEW / unmitigated findings
- (none of severity ≥ med). The consensus matmul path is deterministic-by-construction.

- **LOW (defense-in-depth / docs):** the cross-backend determinism guarantee currently rests on
  (a) the compile-time `__ARM_NEON` Scalar/Neon equivalence and (b) the mining CPU-confirm. Both
  are correct today (proven above), but there is NO consensus-level test that pins
  `NeonDot == ScalarDot` as a regression guard (field tests only exercise the *active* build's
  path), and the metal/cpu parity test is `#if BTX_ENABLE_METAL` (skipped on CI without Metal).
  Proposed mitigation: add a build-independent unit test that computes both Scalar and Neon dot
  (compile both kernels unconditionally for the test) over the worst-case corpus and asserts
  equality, so a future SIMD-kernel change can't silently introduce a cross-arch consensus split.
  Pure hardening; not a live vuln.

## Refuted / robust (so others don't repeat)
- **G3 backend non-determinism split — REFUTED (robust by construction).** Consensus verification
  is a single pinned pure-integer CPU code path (mod 2^31−1, no float, no SIMD-float, no GPU
  dispatch, no `-matmulbackend`-style runtime switch reaching validation — `BTX_MATMUL_BACKEND` in
  init.cpp 1195 is LOG-ONLY). The CPU/Metal/CUDA backend choice affects MINING SPEED only; the
  emitted block always carries the CPU-canonical digest and is re-verified on the pure CPU path by
  every node. SIMD (Neon) and scalar dot kernels are differentially proven bit-identical (2M trials
  + worst case). Cross-build (-O0/-O3/-march=native/-ffast-math) digests identical. Parallel
  MultiplyBlocked is order-independent (disjoint output cells + associative field add).
  Endianness pinned (`static_assert std::endian::native == little`, field.cpp 23). No way for an
  attacker to craft a block valid on backend A but invalid on B.
