### A3. Is ml-kem/Kyber USED/reachable, and is its unpinned provenance a risk?

- **USED:** yes — shielded note encryption (`src/shielded/note_encryption.cpp`, `bundle.cpp`,
  `v2_egress/v2_send`) and bridge view-grants; ChaCha20-Poly1305 AEAD keyed by the ML-KEM shared secret.
- **CONSENSUS-REACHABLE: NO.** `validation.cpp`, `src/consensus/`, and `src/script/` contain **zero**
  references to `mlkem::` or `note_encryption.h`. All `Decaps`/`TryDecrypt` callers are wallet-side
  (`shielded_wallet`, `shielded_rpc`, `bridge_wallet`, `v2_egress/v2_send` note-scan/egress) — i.e. a
  validator NEVER decrypts notes; it verifies the bespoke lattice ZK proofs (the c5/e-wave surface),
  which do not run ML-KEM. ⇒ a bug/weakness in ml-kem can leak a **recipient's** note plaintext to that
  recipient's own wallet path or break their decryption, but **cannot fork the chain, forge value, or be
  triggered remotely against a validator.**
- **RISK of the unpinned provenance:** LOW. It is faithful PQClean (verified above), documented, and
  additionally hardened; it is wallet-only. The exposure is purely **hygiene/auditability** — there is no
  recorded commit/SHA, so a future re-sync or a silent in-tree edit can't be diffed against a named
  baseline. REMEDIATION: record the PQClean commit + a source-tree SHA256SUMS (mirror what
  dilithium/sphincs get in the README), and note the two intentional constant-time deltas so they survive
  a future re-vendor.

## PART B — BUILD / RELEASE SUPPLY CHAIN

### B1. depends/ fetch integrity — ALL SHA256-pinned to canonical mirrors (CLEAN)

- Every package in `depends/packages/*.mk` carries a `_sha256_hash`; the lone file without one is
  `packages.mk` (the package-list aggregator, performs no fetch). `funcs.mk` runs `sha256sum -c` on every
  fetch (confirms c3). Download paths are canonical: boost→archives.boost.io, sqlite→sqlite.org,
  libevent/zeromq→their GitHub release URLs, native_libmultiprocess→bitcoin-core/libmultiprocess
  (pinned commit + hash). FALLBACK_DOWNLOAD_PATH = bitcoincore.org/depends-sources. **No unpinned fetch.**
- No `.gitmodules` ⇒ no submodule-takeover vector (vendored code is in-tree). CONFIRMED.
- **Residual:** `depends/packages/sqlite.mk` is still `version=3460100` (**3.46.1**) → carries
  CVE-2025-6965 (CVSS 7.2) / CVE-2025-29087, fixed in ≥3.50.2 (c3). **This matters MORE than c3 implied:
  the canonical release path is Guix→depends/ (see B4), so the *released* binaries link sqlite 3.46.1**;
  the Alpine 3.49.2 mitigation c3 cited applies only to the convenience Docker, NOT to attested releases.
  Exploitability stays LOW (local descriptor-wallet SQL, not network/consensus). REMEDIATION: bump
  depends/ sqlite ≥3.50.2 (+ re-pin hash). [expat 2.4.8 / freetype 2.11.0 GUI-only per c3 — N/A headless.]

### B2. Dockerfile(s): apk packages UNPINNED (confirmed; risk re-scoped DOWN by B4)

- `contrib/docker/{Dockerfile, Dockerfile.static, Dockerfile.tests}` all do
  `apk add --no-cache <pkgs>` with **no `=version-rN` pin and no repository-snapshot pin**; base is the
  **floating `alpine:3.22` minor tag** (not a `@sha256:` digest). ⇒ the container's dep versions (and base
  layer) float with whatever Alpine 3.22 serves at build time → **non-reproducible**; a future bad/yanked
  Alpine package or a poisoned mirror is auto-pulled. (This is why the shipped sqlite is 3.49.2 *today* but
  could silently change.)
- **Severity re-scoped to LOW-MED:** the Docker image is a **convenience/dev/run container, NOT the
  attested release artifact** — the official release is Guix-built (B4). So this is reproducibility/dev-hygiene,
  not the released-binary trust root. REMEDIATION: pin `apk add pkg=ver-rN` (or `--repository` snapshot)
  and pin the base by digest `alpine:3.22@sha256:…`; add `apk add --no-cache` + a checksum'd repo snapshot
  for reproducible container builds.

  operator runs) **DOES verify SHA256 by default** (`require_snapshot_sha256`, mismatch → hard abort;
  bypass only via explicit `--allow-missing-snapshot-sha256`). Well-designed; no action.

### B4. contrib/guix reproducible-build setup — PRESENT, COMPLETE, and IS the release trust root

- Full Bitcoin-Core-derived Guix bootstrappable/reproducible framework is present and wired:
  `guix-build / guix-attest / guix-verify / guix-codesign / manifest.scm / cuda-packages.scm`, multi-host
  (`x86_64/aarch64-linux-gnu, riscv64, ppc64(le), mingw32, apple-darwin`), `SOURCE_DATE_EPOCH` from git,
  GPG signer attestations via `guix.sigs`.
- **Release DOES rely on reproducibility:** `doc/release-process.md:68` + `doc/btx-github-release-automation.md`
  + `scripts/release/cut_release.py --build-with-guix` show the **canonical release path is Guix** →
  multi-platform outputs → `collect_release_assets.py` → signed `SHA256SUMS.asc` + signer attestations.
  A `--native`/non-Guix CLI preview track exists but is **explicitly labeled** "does not claim Guix
  reproducibility or signer attestation coverage." ⇒ the trusted release is reproducible+attested.
- Guix builds from **depends/** (the SHA256-pinned stack), NOT the Alpine Docker → this is why B1's
  sqlite 3.46.1 reaches releases and why B2's unpinned-apk Docker is NOT the release artifact.
- CUDA redistributables (`cuda-packages.scm`) are **SHA256-pinned** (`content-hash … sha256`) from
  NVIDIA's canonical `developer.download.nvidia.com/compute/cuda/redist/`. Good provenance.
- **Net:** the reproducible-build chain is the strong part of BTX's supply chain. The weak points are the
  *inputs that flow into it* (depends/sqlite stale) and the *parallel unattested* paths (Docker apk, CI).
