# e4 — CRYPTOGRAPHER: PQ signature (ML-DSA/Dilithium, SLH-DSA) + RNG attack

Goal: forge a signature, recover a PQ private key, or break the PQ auth protecting coins.
Builds on c3 (fail-open RNG in `ml_dsa/utils.c`) and c5. Method: deep crypto code review of
`src/libbitcoinpqc` + `src/pqkey.cpp` + `src/script/{pqm,interpreter}.cpp`, diff vs upstream
pq-crystals @ pinned commit, plus EMPIRICAL build of the vendored dilithium ref + BTX wrapper with
three custom harnesses (keygen entropy / signing-nonce / rhoprime-derivation). Regtest namespace
rtE4-, /tmp/rtE4. No production assets touched, no git commit.

## HEADLINE
- **The c3 RNG defect does NOT lead to key recovery or forgery via SIGNING.** BTX Dilithium signing
  is the FIPS-204 *hedged* construction: the per-signature nonce seed is
  `rhoprime = CRH(secret_key ‖ rnd ‖ mu)` (mu = H(tr‖pre‖msg)). Even with the RNG fully
  zeroed/attacker-controlled (`rnd = 0`), `rhoprime`/`y` stays bound to the **secret key** and the
  **message** ⇒ no ECDSA-style nonce-reuse key leak is possible. EMPIRICALLY CONFIRMED below.
- **The c3 fail-open `/dev/urandom`→memset(0) branch is UNREACHABLE on every production path.** All
  keygen/sign callers (`CPQKey::MakeNewKey/MakeDeterministicKey/Sign`, `pqkey.cpp`) ALWAYS pass a
  non-NULL ≥128-byte `random_data` sourced from Core's `GetRandBytes`, which is **fail-CLOSED**
  (`ProcRand`→`RandFailure()`→`std::abort()`, random.cpp:55-58). So `custom_randombytes_impl` always
  takes the injected-entropy branch; the fopen-fallback memset(0) is dead in production. ⇒ **c3's
  HIGH should be RE-RATED to LOW (code-quality / latent), not a realistic key-compromise.** Fix it
  anyway (fail-closed) for robustness and to protect any future caller that passes NULL.
- **Dilithium ref code is FAITHFUL to pq-crystals @ pinned commit `444cdcc` (verified vs upstream
  `ref/sign.c`; `unpack_sig` is the SUF-CMA-hardened upstream version).** No security-relevant
  deviation. Parameter set = ML-DSA-44 (Dilithium MODE 2), the standardized NIST level-2 set.
- **Signature is strictly length-encoded + SUF-CMA non-malleable; sighash is domain-separated**
  (epoch byte = 2, distinct from Taproot's 0; CSFS uses tag `CSFS/btx`). No malleability/replay break.

## Vectors attempted (HOLD / BREAK)
- **Dilithium nonce-reuse → key recovery (the ECDSA analogue), RNG zeroed/controlled** — HOLD (SAFE).
  RESULT: empirical. `rhoprime` (the seed for `y`) is `CRH(key‖rnd‖mu)`; zeroing `rnd` does not
  collapse `y` across different messages (different `mu`) and cannot be predicted without `key`.
  Two distinct sigs sharing the same `y` (the precondition for solving `z = y + c·s1`) is impossible
  because same-`mu` ⇒ same challenge `c` ⇒ identical signature (leaks nothing); different-`mu` ⇒
  different `y`. Evidence:
  ```
  [nonce] zero-rnd diff-msg rhoprime(=>y seed) equal? NO  => y independent (SAFE)
  [nonce] zero-rnd same-msg rhoprime reproducible? YES (deterministic, expected)
  [nonce] zero-rnd: rhoprime depends on secret key?  YES (unpredictable without sk)
  [sign]  zero-rnd different-msg sigs identical(reused nonce?): no (msg bound via mu)
  [sign]  zero-rnd sigs verify: m1=OK m2=OK
  ```
  Cryptographic argument: FIPS-204 §3.4 hedged signing is *defined* to be secure with rnd=0 (that IS
  the deterministic mode); the only thing the RNG buys is side-channel/fault hedging, not EUF/SUF.
- **Predictable-key via zero/weak keygen entropy (build on c3)** — HOLD in production (path gated),
  CRITICAL only if the fail-open branch is ever reached. RESULT: empirical — feeding all-zero 128B
  entropy to `ml_dsa_44_keygen` yields a fixed, reproducible keypair:
  ```
  [keygen] strong vs zero-entropy pubkeys differ: YES
  [keygen] zero-entropy reproducible (attacker can recompute): YES (CRITICAL if reachable)
  ```
  But production never feeds zero/short entropy (Core RNG fail-closed + 128B always supplied), so the
  attacker cannot *cause* zero entropy remotely. Severity realistic = LOW-MED, must-fix-for-robustness.
- **Faithfulness / malleability of the Dilithium impl (Target 3,4)** — HOLD. `ref/sign.c`,
  `unpack_sig` byte-match upstream. `crypto_sign_verify_internal` re-derives `c2` and checks all
  `CTILDEBYTES` (full c~), checks `‖z‖∞ < γ1−β` (line 314), and `unpack_sig` rejects non-canonical
  hints ("ordered for strong unforgeability", "extra indices zero") ⇒ SUF-CMA, no hint/z malleability.
  At the script layer the sig must be EXACTLY `2420` bytes (or `2421` with a trailing hashtype byte);
  any other length → `SCRIPT_ERR_SIG_MLDSA` (interpreter.cpp:1189-1197). No alternate valid encoding.
- **Cross-context / cross-tx signature replay (Target 2)** — HOLD. PQ sigs are over a BIP341-style
  P2MR sighash (`SignatureHashSchnorr`, sigversion P2MR) that binds tx version, locktime, all
  prevouts (incl. the exact txid:vout spent), spent amounts/scripts/sequences, outputs, input index,
  AND the tapleaf hash (which commits the pubkey/script). Domain separation: epoch byte = 2 (vs
  Taproot 0) + `ext_flag`. CSFS uses a *separate* tagged hash `TaggedHash("CSFS/btx")` so a
  CSFS-signed blob can't be lifted into a tx sighash. ⇒ no cross-tx / cross-opcode replay.
- **Parameter / out-of-norm acceptance (Target 5)** — HOLD. MODE 2 = ML-DSA-44 (NIST level 2),
  γ1=2^17, β=78, ω=80, CTILDEBYTES=32 — the standardized set. Verification rejects out-of-norm z and
  oversized/duplicate hints (above).
- **SLH-DSA (SPHINCS+) RNG / signing** — HOLD. Same fail-open pattern in `slh_dsa/utils.c:86-95`
  (also unreachable: callers supply strong entropy). SPHINCS+ optrand = `SHAKE256(sk‖m)` then mixes
  injected randomness ⇒ sk+message-bound and secure even fully deterministic. SLH-DSA-SHAKE-128s is
  the standardized SLH-DSA parameter set.

## NEW / unmitigated findings
- **[LOW, code-quality — RE-RATE of c3 HIGH] Fail-open RNG fallback is real but unreachable in prod.**
  `src/libbitcoinpqc/src/ml_dsa/utils.c:57-72` (and `slh_dsa/utils.c:84-99`): on `/dev/urandom`
  open-fail or short read, `memset(out,0)` instead of aborting. NOT reached today (all callers pass
  non-NULL strong entropy; Core RNG is fail-closed). Repro: only by a future/test caller invoking
  `ml_dsa_44_keygen`/`sign` with `random_data==NULL`. **Fix (still do it):** delete the zero-fill
  branches — on RNG failure, `abort()` (matches Core's `RandFailure`); never silently zero a key/seed
  buffer. (Aligns with LEARNINGS fix-list item 8, but severity is LOW-realistic, not HIGH.)
- **[LOW, defense-in-depth] Injected-entropy wrap/reuse in `custom_randombytes_impl` (lines 79-97).**
  When more bytes are requested than supplied, the offset wraps and re-emits bytes. In the ACTUAL
  call pattern this is benign: signing supplies a 64-byte `signing_seed` and consumes only
  RNDBYTES=32 (no wrap); keygen supplies 128B and consumes SEEDBYTES=32 (no wrap). But the wrap is a
  latent footgun. **Fix:** return an error / abort if `outlen > g_random_data_size` rather than
  cycling. (Defense-in-depth; not currently exploitable.)
- **[INFO, not a break] P2MR sighash has no explicit genesis/fork-id binding.** It relies on prevout
  (txid:vout) uniqueness for chain separation — identical to BIP341 Taproot. A pre-fork UTXO could be
  replay-spent on a later split chain (the standard Bitcoin replay caveat). The CSFS tag *is*
  chain-namespaced (`CSFS/btx`). **Optional hardening:** fold `hashGenesisBlock` into the P2MR epoch
  domain (mirrors what c5 reports the SHIELDED chain-bound digest already does). Low priority.

## Refuted / robust (so others don't repeat)
- **PQ signing cannot leak the private key via the RNG defect.** Hedged construction binds `y` to
  `sk` and `mu`; zero/controlled `rnd` is the secure deterministic mode (FIPS-204). No nonce reuse
  across distinct messages; same-message reuse is just the deterministic identical signature.
- **The c3 fail-open `/dev/urandom` memset(0) branch is dead code in production** — every keygen/sign
  caller supplies non-NULL strong entropy; Core's `GetRandBytes` aborts on failure. (Downgrades c3's
  realistic key-compromise exposure to LOW; the FIX is still warranted, as code quality.)
- **Dilithium ref impl is faithful (no malleability/forgery deviation), ML-DSA-44 params standard,
  verify enforces z-norm + canonical hints (SUF-CMA), script enforces strict sig length.** No forgery
  or malleability primitive found.
- **Domain separation present**: P2MR epoch=2 ≠ Taproot epoch=0; CSFS tag `CSFS/btx` ≠ `TapSighash`.
