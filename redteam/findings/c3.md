# c3 — Supply-chain / dependency audit (BTX, branch redteam/comprehensive-hardening off pr-214)

Scope: dependency inventory + known-CVE check + vendored/custom-crypto provenance + build/release
supply-chain. Read-only on repo. No production assets touched. CVE claims cited; none fabricated.

NOTE on build paths: there are TWO dependency stacks.
- **`depends/`** (Bitcoin-Core-style cross-compile, SHA256-pinned tarballs) — used for release/Guix-style builds.
- **`contrib/docker/Dockerfile*`** — builds against **Alpine 3.22 apk packages** (boost/libevent/sqlite/zeromq),
  NOT depends/. So the *actually-shipped* container uses Alpine's versions, which differ from depends/.
Both are reported below.

## 2. KNOWN-CVE CHECK (external deps)

- **sqlite 3.46.1 (depends/ only) — CVE-2025-6965 (CVSS 7.2, memory corruption / possible RCE, fixed 3.50.2)
  and CVE-2025-29087 (concat_ws heap overflow, fixed 3.49.1).** Reachable only via attacker-controlled SQL;
  in Bitcoin Core the SQLite wallet DB is LOCAL (descriptor wallet), not network/consensus data ⇒
  **LOW exploitability**. The shipped Alpine container already uses sqlite **3.49.2-r1** (carries the
  CVE-2025-29087 fix; CVE-2025-6965 fix is 3.50.2 so Alpine 3.49.2 is still pre-fix for -6965 unless
  backported). REMEDIATION: bump depends/ sqlite to ≥3.50.2; verify Alpine backport for -6965.
  Source: https://www.sqlite.org/cves.html , https://nvd.nist.gov/vuln/detail/CVE-2025-6965

- **libevent 2.1.12 — CLEAN for the historic evdns CVEs** (CVE-2016-10195/-10196/-10197 were all fixed by
  2.1.6-beta; 2.1.12 includes them). Caveat: 2.1.12 (Jul-2020) is the **last upstream release** — no newer

  Source: https://libevent.org / changelog 2.1.12.

- **zeromq 4.3.5 — CLEAN.** The known libzmq RCE/overflow/DoS CVEs (CVE-2019-13132, -2021-20234/20235/20236)

## 3. VENDORED / CUSTOM CRYPTO RISK (highest supply-chain category)

### 3a. libbitcoinpqc — forked from an EXPERIMENTAL personal repo, self-declared "not production-hardened"
- Upstream is **cryptoquick/libbitcoinpqc** (Hunter Beast / surmount.systems), a BIP-360 experiment.
  The Cargo.toml claims `repository = https://github.com/bitcoin/libbitcoinpqc` — **that URL is 404**
  (no such repo under the `bitcoin` org). This is a **mislabeled / non-canonical origin** — a reviewer
  cannot pin the true upstream from the manifest. (Supply-chain provenance hygiene issue.)
- README explicitly states: *"The implementations are based on reference code from the NIST PQC
  standardization process and are **not production-hardened**."* This code is **consensus-reachable**
  (PQ signature verification in script). Shipping un-hardened reference PQ crypto in consensus = HIGH
  inherent risk (timing/side-channel + correctness), independent of any specific CVE.
- Pinned dilithium `444cdcc` = pq-crystals master as of **2024-11-01**; current master HEAD is 2025-10-08
  ("add GPL license"). The only intervening commit is the license add — **no security/correctness fix is
  being missed** by the pin. ⇒ dilithium pin is effectively current. (Source: github.com/pq-crystals/dilithium/commits/master)
- sphincsplus `7ec789ace` — verify against sphincs/sphincsplus master before release; no public CVE found,
  but confirm no post-pin correctness/fault-resilience commit is missed (academic fault attacks exist on
  SPHINCS+ but require physical fault injection, not remote). LOW-remote / track upstream.

### 3b. ★ Embedded custom RNG has TWO key-generation flaws (src/libbitcoinpqc/src/ml_dsa/utils.c)
`custom_randombytes_impl()` (the `randombytes` provider for dilithium/sphincs keygen):
1. **Silent zero-fill on RNG failure (HIGH if hit):** if `fopen("/dev/urandom")` fails OR `fread` under-reads,
   it `memset(out,0,...)` and returns — i.e. it generates keys from **all-zero / partially-zero entropy**
   instead of aborting. Predictable PQ keys ⇒ key compromise. (Lines 57-70.)
2. **Entropy wrap-around / reuse (MED):** when caller-supplied entropy is shorter than requested, the loop
   wraps `g_random_data_offset = 0` and **re-emits the same bytes** rather than failing — reduced-entropy /
   correlated key material. (Lines 93-96.)
- Reachability: keygen is WALLET-side (creating PQ keys), not block/consensus verification, so this is NOT a
  remote network exploit — but it is a real crypto-quality defect in vendored consensus-adjacent crypto.
  REMEDIATION: replace the fallback with a hard failure/abort (never zero-fill a key buffer); on insufficient
  supplied entropy, return an error rather than wrapping; prefer Bitcoin Core's `GetStrongRandBytes()` over a
  raw `/dev/urandom` fopen. SEVERITY: HIGH (defect class), realistic-exposure MED (local keygen only).

### 3c. ★ matmul/ and shielded/ are 100% BESPOKE, UNAUDITED, consensus-critical (HIGHEST risk)
- `src/matmul/*` (Freivalds verifier, noise generator, field, transcript, matmul_pow) and
  `src/shielded/{smile2,ringct(matrict),lattice}/*` are all original "Copyright (c) 2026 The BTX developers"
  code with **no upstream, no third-party audit, no CVE history to check** — by definition the largest
  supply-chain unknown. There is no "bump to fixed version" remediation; soundness rests entirely on the
  in-house implementation. (Freivalds soundness itself was separately analyzed by agents b1/b2 — REFUTED as
  forgeable — but the broader bespoke shielded ZK stack (smile2 BDLOP/CT-proofs, matrict ring/range proofs,
  lattice NTT/sampling) has NOT been third-party audited.) RECOMMENDATION: external cryptographic audit of
  shielded/ + matmul/ before relying on them economically; this is the #1 supply-chain exposure for BTX.
- `shielded/lattice/ntt` reuses the Dilithium reference NTT (provenance noted in lattice/ntt.h) — ties the
  shielded path's correctness to the same dilithium pin as §3a.
- `crypto/ml-kem-768` (Kyber) used for shielded note encryption is also a reference-style in-tree impl with
  no upstream pin recorded — should carry a version/commit + SHA256SUMS like dilithium does.

## 4. BUILD / RELEASE SUPPLY CHAIN

GOOD:
- `depends/funcs.mk` enforces **SHA256 verification on every fetch** (fetch_file_inner runs `sha256sum -c`),
  with a recipe-hash build-id. No `curl|sh`, no unverified tarball execution in depends/.
- FALLBACK_DOWNLOAD_PATH = `https://bitcoincore.org/depends-sources` (canonical Bitcoin Core mirror) — not a
  personal host. native_libmultiprocess pulled from `bitcoin-core/libmultiprocess` (canonical), pinned commit + hash.
- No git submodules at all (`.gitmodules` absent) ⇒ no submodule-takeover vector; vendored code is in-tree.
- Docker bases are `alpine:3.22` (a release tag, not `:latest`). CI tool pins exist (shellcheck v0.8.0,
  mlc v0.19.0, llvm llvmorg-21.1.1, iwyu pinned).

GAPS / NOTES (LOW, CI/dev-only — not the shipped artifact):
- **CI installs unverified binaries (LOW):** `ci/lint/04_install.sh:58,64` does
  `curl -sL <github release> | tar` for shellcheck and `curl -sL ... -o /usr/bin/mlc; chmod +x` for mlc —
  **no SHA256 / signature check**. A compromised release asset or MITM would execute in CI. Also
  `git clone --depth=1 pyenv` (04_install.sh:23) and `llvm-project`/`iwyu` clones are unpinned-to-HEAD of a
  branch/tag (mutable). REMEDIATION: pin + checksum the shellcheck/mlc downloads; pin pyenv to a commit.
- **Alpine apk packages are unpinned in Dockerfiles (LOW-MED):** `apk add --no-cache boost-dev libevent-dev
  sqlite-dev zeromq-dev ...` with no `=version` pin and no pinned repo snapshot ⇒ the shipped binary's
  dependency versions float with whatever 3.22 serves at build time (non-reproducible; a future bad Alpine
  package is auto-pulled). REMEDIATION: pin apk versions (`pkg=ver-rN`) and/or a repository snapshot for
  reproducible container builds. (This is also why the shipped sqlite is 3.49.2 today but could change.)
- `libbitcoinpqc` Cargo.toml repository field is a 404 (`bitcoin/libbitcoinpqc`) — fix provenance label (§3a).

## PRIORITIZED VULNERABILITY LIST
1. **HIGH (crypto-quality defect) — silent zero-fill / entropy-wrap in embedded PQ RNG** (3b,
   src/libbitcoinpqc/src/ml_dsa/utils.c). Keygen from zero/repeated entropy on RNG failure. Local keygen
   reach ⇒ realistic MED, but must-fix (never zero-fill a key buffer; fail closed; use GetStrongRandBytes).
2. **HIGH (inherent) — bespoke, unaudited, consensus-critical matmul/ + shielded/ crypto** (3c). No upstream,
   no audit, no CVE to bump. Recommend external cryptographic audit before economic reliance. #1 supply-chain unknown.
3. **MED — un-hardened reference PQ crypto in consensus** (3a): libbitcoinpqc self-declares "not production-
   hardened" yet is consensus-reachable; fork origin is mislabeled (404 repo). Track sphincs/dilithium pins
   (dilithium pin is effectively current; verify sphincs pin) + fix provenance label.
4. **MED — depends/ sqlite 3.46.1 (CVE-2025-6965 / CVE-2025-29087)** for Guix/depends release builds; bump
   to ≥3.50.2. (Shipped Alpine container already on 3.49.2 — partially mitigated.)
5. **MED (GUI-only) — depends/ expat 2.4.8 and freetype 2.11.0** carry multiple CVEs incl. exploited
   CVE-2025-27363 (freetype); bump expat≥2.6.4, freetype≥2.13.3. N/A to headless daemon.
6. **LOW — pin ml-kem-768/Kyber to a recorded upstream commit+hash** (no version recorded, unlike dilithium).
7. **LOW (CI) — unverified shellcheck/mlc downloads + unpinned apk in Dockerfiles** (§4): add checksums /
   version pins for reproducibility.
