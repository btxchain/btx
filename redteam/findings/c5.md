# c5 — MONETARY SOUNDNESS (shielded value forgery / inflation / double-spend) crown-jewel audit

Branch `redteam/comprehensive-hardening` off `pr-214`, image `btx-node:pr214`. Method: deep
consensus-path code review + partial regtest experiment (rtC5- namespace, /tmp/rtC5). Scope: can an
attacker FORGE shielded value, INFLATE supply, DOUBLE-SPEND, or steal via an IMPLEMENTATION gap (a
missing/incorrect/forgeable check) — NOT a full crypto break. Builds on c3 (bespoke unaudited crypto →
recommend external audit) and c4 (shielded crash-surface cleared). No production assets touched.

## Headline
- **No implementation-level inflation / double-spend / value-forgery gap found.** Every monetary
  invariant (value conservation, range, nullifier double-spend, membership/anchor, bridge mint, proof
  binding) is enforced by a present, unconditional, by-construction-correct check on the consensus
  ConnectBlock path. The residual exposure is the SOUNDNESS of the bespoke lattice ZK primitives
  themselves (smile2 VerifyCT, matrict) — a crypto-break question, NOT an exploitable code gap → c3's
  external-audit recommendation stands.
- **NEW (med, stability/DoS, NOT consensus): reproducible SIGSEGV in the wallet shielded-send prover.**
  `z_sendmany` (z→z) crashed the node twice on two fresh regtest builds (exit 139, not OOM, no
  assert/terminate message) — a raw segfault in the SMILE-CT prove path. Local wallet-side (the sender
  builds its own proof), not remotely triggerable, so not a network DoS and not a money bug — but it
  means shielded sends are non-functional/unstable on this build and points at a memory-safety defect
  in the bespoke `src/shielded/smile2` prover (matches c3 "bespoke unaudited" + c4's deferred sanitizer
  build). Worth a sanitizer build + fix before relying on shielded sends.

## What the SHOULD-exist checks are, and proof they exist & are correct (cite file:line)

### 1. Value conservation / balance — PRESENT & CORRECT
- **Transparent side (the money equation):** `Consensus::CheckTxInputs` (src/consensus/tx_verify.cpp
  :203-227) folds the shielded `value_balance` into Bitcoin's input/output conservation:
  `adjusted_in = nValueIn + value_balance`; rejects `adjusted_in < value_out`; `txfee = adjusted_in -
  value_out`. `MoneyRangeSigned(value_balance)` + `CheckedAdd` overflow guards (206-213). This runs
  UNCONDITIONALLY for every non-coinbase tx in ConnectBlock (src/validation.cpp:6364).
- **`value_balance` is the value the shielded pool releases to the transparent side**; its TRUTH is
  bound to and proven by the shielded proof, not taken on the attacker's word:
  - V2_SEND (active SMILE path): `payload.value_balance` is passed as `public_fee` into
    `VerifySmile2CTFromBytes` (src/shielded/v2_proof.cpp:2515) → `VerifyCT` enforces the homomorphic
    relation `Σ input_note_values == Σ output_note_values + public_fee` via the **h2 balance check**
    (src/shielded/smile2/ct_proof.cpp:3122-3133: first SLOT_DEGREE coeffs of h2 must be 0; unbalanced ⇒
    non-zero ⇒ reject). `public_fee` (= value_balance) is also bound into the Fiat-Shamir transcript
    (3211-3215) so it can't be swapped post-hoc.
  - Legacy/recovery MatRiCT path: `VerifyMatRiCTProof(... value_balance ...)` →
    `VerifyBalanceProof(input_commitments, output_commitments, fee=value_balance, tx_binding_hash)`
    (src/shielded/ringct/matrict.cpp:453-459).
- **Pool can never go negative:** every shielded tx applies its state-delta to the running pool and
  rejects on underflow — `next_pool_balance.ApplyValueBalance(*value_balance)` →
  "shielded-pool-balance-negative" (src/validation.cpp:6040-6049), unconditional in ConnectBlock.
- **Structural floor:** `SendPayload::IsValid` requires `value_balance >= fee`, `fee>=0`,
  `MoneyRangeSigned(value_balance)` (src/shielded/v2_bundle.cpp:720-735), enforced via
  `CheckTransaction → CheckStructure` (src/consensus/tx_check.cpp:28) for EVERY tx unconditionally.
- I looked for a shielded→transparent / transparent→shielded / bridge path where the declared
  value_balance bypasses the proof: NONE. `GetShieldedTxValueBalance` (src/shielded/bundle.cpp:1091)
  and `TryGetShieldedStateValueBalance` (1035) derive the amount only from proof-bound payload fields
  for every family; ingress/egress/rebalance route through their settlement-witness verification (see
  §5).

### 2. Range proofs (no negative / no >MAX wraparound) — PRESENT & CORRECT
- MatRiCT: every output commitment is range-proven — `VerifyRangeProof(output_range_proofs[i],
  output_commitments[i])` for all i (src/shielded/ringct/matrict.cpp:447-451). The proof is per-bit
  (bit challenges, range_proof.cpp:225) and bounds value to `[0, 2^VALUE_BITS)` with a static_assert
  `2^VALUE_BITS > MAX_MONEY` (range_proof.cpp:59-60); `value<0 → false` (116); `value <
  2^VALUE_BITS` (120). No negative, no wraparound.
- SMILE: amounts are encoded as a base-4 digit vector across NUM_NTT_SLOTS=32 slots, 2 bits/slot ⇒ a
  64-bit range; `EncodeAmountToSmileAmountPoly` rejects `amount<0` and any value whose base-4 rep
  overflows 32 slots (src/shielded/smile2/ct_proof.cpp:2037-2057). The verifier never sees cleartext;
  the digit/selector binary constraints are enforced inside the lattice relation (selector binary
  check ct_proof.cpp:3648-3652; framework relation bound by the **unconditional**
  `post_h2_binding_digest`, see §6). NOTE: this is the smile-specific encoding; whether the lattice
  relation fully forces every amount digit into {0,1,2,3} is a SOUNDNESS property of the bespoke ZK,
  not something I could break at the code level — flagged for external crypto audit (c3).

### 3. Nullifier soundness (double-spend) — PRESENT & CORRECT (consensus path)
- **Within-block duplicate:** `block_nullifiers.insert(nullifier).second` → "bad-shielded-nullifier-
  duplicate" (src/validation.cpp:5938-5942).
- **Cross-block already-spent:** `m_chainman.IsShieldedNullifierSpent(nullifier)` →
  "bad-shielded-nullifier-spent" (src/validation.cpp:5943-5946); backed by a persisted NullifierSet
  (`m_shielded_nullifiers->Contains`, src/validation.cpp:12411-12416). Both run UNCONDITIONALLY for
  every shielded tx in ConnectBlock (gated only by `enforce_shielded_consensus`, which the code
  comments explicitly note is "not relaxed by assumevalid", src/validation.cpp:5873-5876).
- **Nullifier ↔ note binding & non-malleability:** the nullifier is bound to the spent note inside the
  proof (MatRiCT `VerifyRingSignatureNullifierBinding`, matrict.cpp:432; SMILE serial-number → nullifier
  hash, validation.cpp:936-942) and re-checked against the payload nullifier by `CShieldedSpendAuthCheck`
  (validation.cpp:1717-1730: proof-bound nullifier must == payload nullifier, else
  "bad-shielded-spend-auth-nullifier-mismatch"). intra-tx uniqueness also in SendPayload::IsValid
  (v2_bundle.cpp:744) and CheckTransaction (tx_check.cpp:37). ⇒ two distinct-looking nullifiers for one
  note would require breaking the binding (crypto), and a replayed nullifier is rejected.

### 4. Membership / commitment-tree soundness — PRESENT & CORRECT
- **Anchor must be a real historical root:** `IsShieldedAnchorValid` checks the bundle's anchor against
  the tracked `m_shielded_anchor_roots` history, fails closed if state uninitialized
  (src/validation.cpp:12305-12322). Enforced for EVERY anchor a bundle references, in ConnectBlock
  (5954-5971) AND mempool (3246-3261). ⇒ you cannot prove membership against a fake/forged root.
- **Membership proof verifies against the chain-provided tree snapshot**, not an attacker-claimed set:
  ring members are reconstructed from `m_tree_snapshot` (the node's own
  `GetShieldedMerkleTree()` snapshot taken in ConnectBlock, validation.cpp:5858-5864) at the spend's
  claimed positions (`BuildV2SendSmileRingMembers` / `BuildV2SendRingMembers`, validation.cpp:1409/1442),
  then the lattice ring/membership proof verifies. Output-commitment duplicates across the block are also
  rejected (5978-5985, "bad-shielded-commitment-duplicate"). ⇒ spending a note never added to the tree,
  or forging membership against a wrong/old root, is blocked.

### 5. Bridge peg (mint/burn) — PRESENT & CORRECT (no mint-without-burn / replay found)
- INGRESS (mint into pool): the pool credit `value_balance = -(total_amount+fee)` is taken from
  `witness.header.statement.total_amount`, but that witness is verified: `VerifyV2IngressProof` +
  settlement-witness (signed-receipt / proof-receipt threshold, external-anchor) verification runs
  inside `CShieldedProofCheck` (validation.cpp:1468-1541). I CHASED a suspected gap here:
  `NeedsShieldedProofCheck` (validation.cpp:1228) does NOT list V2_INGRESS_BATCH explicitly and relies
  on `HasShieldedInputs()`; for an ingress that = `consumed_spends.size()`. **A zero-consumed-spend
  ingress would skip proof verification — BUT `IngressBatchPayload::IsValid` REQUIRES
  `!consumed_spends.empty()` (src/shielded/v2_bundle.cpp:861), enforced unconditionally by
  CheckTransaction→CheckStructure before any state change.** So consumed_spends ≥ 1 always ⇒
  HasShieldedInputs() true ⇒ proof always checked. The gap is CLOSED by the structural invariant — not
  exploitable, but FRAGILE (see hardening note). EGRESS (burn out): debit
  `value_balance=-statement.total_amount` (bundle.cpp:1071) gated on a matured, single-use settlement
  anchor that must already exist on-chain (`GetShieldedSettlementAnchorState` + maturity + single-use
  consume, validation.cpp:6244-6290) and an imported-receipt bundle bound to the external anchor
  (`VerifyV2EgressImportedReceiptBundle`, validation.cpp:1572). Replay of a mint/anchor is blocked by
  single-use settlement-anchor consumption (6278+) and nullifier/anchor uniqueness.

### 6. Proof binding / replay / malleability — PRESENT & CORRECT
- **Each proof is bound to THIS exact tx.** `ComputeV2SendStatementDigest` hashes the WHOLE stripped
  transaction `TX_WITH_WITNESS` (vin/vout + full v2 bundle incl. spends, outputs, value_balance, fee),
  with proof_payload + statement_digest zeroed to avoid circularity (src/shielded/v2_proof.cpp
  :1140-1190); the verifier RECOMPUTES it from the actual tx and requires the bundle's envelope to match
  (`ProofEnvelopesMatch`, v2_proof.cpp:1868). Change any output ⇒ digest changes ⇒ envelope mismatch ⇒
  reject. Chain-bound variant additionally folds `hashGenesisBlock` + fork height (1182-1185) ⇒
  cross-chain replay blocked. ⇒ a proof cannot be lifted from another tx.
- **Conditional-check audit (the prime "implementation gap" suspects) — all REFUTED:** smile2 VerifyCT
  has several checks gated on prover-supplied fields:
  `if (SeedIsPresent(proof.seed_*) && ...)` (ct_proof.cpp:3221/3291/3447/3719) and
  `if (!proof.omega.IsZero())` / `if (!proof.framework_omega.IsZero())` (3579/3675). I verified these
  are NOT soundness holes: (a) the consensus-critical Fiat-Shamir challenges are ALWAYS derived from the
  **recomputed transcript** (`seed_c_final = SeedIsPresent ? proof.seed : seed_c_check`, 3446; c0_chal
  from recomputed `seed_c0_check`, 3300; aux_ck/account/alpha challenges from the recomputed
  public_transcript) — zeroing a seed just falls back to the genuine recomputed value, it does NOT let
  the attacker choose a challenge; (b) the omega / framework_omega equalities are REDUNDANT with the
  **unconditional** binding digests `ComputeCtBindingDigest("...PRE_H2...")` (3531-3542) and
  `("...POST_H2...")` (3665-3674), which tightly bind the verifier-computed `lhs` to the prover's
  committed `framework_omega` (CollectLiveCtPostH2BindingPolys feeds `framework_omega` on prove,
  ct_proof.cpp:944-960/2932-2939, and `lhs` on verify, 3665-3671 — so digest-match == `lhs==
  framework_omega` regardless of the `if(!IsZero())`). ⇒ no free-challenge / skip-the-relation
  primitive. (Defense-in-depth: make those equalities unconditional / drop the seed-absent fallback —
  not required for soundness.)

## Vectors attempted
- Inflate via lying `value_balance` (shielded→transparent): REJECTED by construction — value_balance is
  proof-bound (h2/balance check) AND folded into transparent conservation (CheckTxInputs). No bypass path.
- Negative / >MAX output value (range): REJECTED — per-output range proofs (MatRiCT) / 64-bit base-4
  encode + lattice constraints (SMILE); no negative, no wraparound.
- Double-spend (replay nullifier, two nullifiers/one note): REJECTED — within-block + cross-block
  nullifier-set checks, proof-bound nullifier binding, intra-tx uniqueness, all unconditional in
  ConnectBlock.
- Spend a note not in tree / forge membership vs wrong root: REJECTED — anchor must be a tracked
  historical root (fail-closed), membership verified vs the node's own tree snapshot.
- Bridge mint without burn / zero-consumed-spend ingress to skip the proof: REFUTED — IsValid forces
  consumed_spends≥1 ⇒ proof always runs; egress gated on matured single-use on-chain anchor.
- Lift a proof onto another tx (replay/malleability): REJECTED — statement digest = hash of whole
  stripped tx + (chain-bound) genesis/fork; envelope-match enforced.
- Skip the ZK relation by zeroing prover seeds / omega / framework_omega: REFUTED — challenges fall back
  to the recomputed transcript; the relation is pinned by unconditional binding digests.
- EMPIRICAL inflation/double-spend test on a live mutated tx: BLOCKED by the prover segfault below
  (could not construct a spendable shielded-send tx to mutate). Shield/unshield (z_shieldfunds) worked;
  z→z send (z_sendmany) crashes the node.

## NEW / unmitigated findings
- **Wallet shielded-send prover SIGSEGV | severity MED (stability/DoS, NOT consensus/money) | repro:**
  regtest `btx-node:pr214`; createwallet+encrypt; build a ≥16-note anonymity pool via repeated
  `z_shieldfunds 5`; then `z_sendmany [{z-addr,5.0}] 0.001`. Node exits 139 (SIGSEGV; OOMKilled=false;
  no assert/terminate line) inside the SMILE-CT prove path. Reproduced on TWO independent fresh
  builds. **Impact:** local sender only (builds its own proof) ⇒ no remote/network DoS, no inflation;
  but shielded sends are non-functional and a segfault in consensus-adjacent bespoke crypto is a red
  flag. **Mitigation:** ASan/UBSan build of `src/shielded/smile2` ProveCT/serialize and fix the
  memory-safety defect; add a fuzz target for the prover; (matches c4's deferred sanitizer-build lead
  and c3's bespoke-crypto external-audit recommendation).
- **Fragile reliance on a structural invariant (LOW, hardening):** `NeedsShieldedProofCheck`
  (validation.cpp:1228) omits V2_INGRESS_BATCH/V2_REBALANCE and depends on
  `HasShieldedInputs()`≡`consumed_spends≥1` (enforced elsewhere by IngressBatchPayload::IsValid). Safe
  today, but a future edit that relaxes the `consumed_spends.empty()` rule, or a new value-bearing
  family not listed, would silently skip proof verification ⇒ inflation. **Mitigation:** make
  `NeedsShieldedProofCheck` return true for ANY family with a non-zero state value-balance / any
  bridge-mint family explicitly, rather than inferring it from input count.

## Refuted / robust (so others don't repeat)
- Shielded value conservation, range, nullifier double-spend, membership/anchor, bridge mint/burn,
  proof tx-binding/replay: all enforced, present, correct, UNCONDITIONAL on ConnectBlock — no
  implementation gap (citations in §1-§6). The smile2 conditional `SeedIsPresent`/`!IsZero()` checks are
  NOT soundness holes (challenges recompute from transcript; relation pinned by unconditional binding
  digests).
- The ONLY residual money risk is the SOUNDNESS of the bespoke lattice ZK (smile2 VerifyCT h2/range/
  framework relations; matrict balance/range/ring) — a crypto-break question, out of "implementation
  gap" scope ⇒ DEFER to external crypto audit (c3). I did not find, and do not claim, a primitive break.
