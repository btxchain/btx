## How the nullifier/membership scheme actually works (mapped, with file:line)

There are TWO note/nullifier derivations in-tree; only the SMILE one is consensus-live post-fork.

1. **Classical SHA256 note (note.cpp)** — `cm = SHA256(tag||inner||rho||rcm)`,
   `nf = SHA256("BTX_Note_Nullifier_V1"||spending_key||rho||cm)` (note.cpp:22-62). Domain-separated,
   collision-resistant, nullifier bound to (spending_key, rho, cm) ⇒ note-unique. This is the legacy/
   reference shape (the header comment); the LIVE serial differs (below).

2. **SMILE serial-number nullifier (LIVE / DIRECT_SMILE)** — the spend proves a lattice one-of-many.
   - Per-note spending key: `s = SampleTernary(seed)` where `seed = DeriveNoteSubKey(note,
     "BTX-SMILE-V2-NOTE-TO-PK")` = SHA256 over note.rho||rcm||recipient_pk_hash||value
     (wallet_bridge.cpp:43-59,181-190). ⇒ **`s` is PER-NOTE** (fresh per distinct rho/rcm), not
     per-account — so the serial is note-unique (resolves the "all of an account's notes share one
     nullifier ⇒ lock-up" concern; it does NOT arise).
   - Public key `pk = A·s`; serial `sn = ⟨b_sn, s⟩` (ct_proof.cpp:2096-2110, ComputeSerialNumber).
     Nullifier = `ComputeSmileSerialHash(sn)` = SHA256("BTX_SMILE2_Serial_Nullifier_V1" || sn_coeffs
     reduced mod q) (verify_dispatch.cpp:227-245). The serial is a KEY-IMAGE of `s` (MatRiCT/RingCT
     style), bound to the note only via `s`'s note-derivation.
   - The merkle leaf binds the spend's three public objects together:
     `account_leaf_commitment = ComputeShieldedAccountLeafCommitment(BuildShieldedAccountLeaf({pk,
     public_coin}, note_commitment, DIRECT_SEND))` (wallet_bridge.cpp:121-137; account_registry.cpp
     HashTaggedObject/Pair with distinct `BTX_SMILE_ACCOUNT_*_V1` tags). Membership proves this leaf is
     in the tree at a valid anchor.
   - **Serial↔key binding is a Schnorr/Fiat-Shamir relation:** `z0 = y0 + c0·s`
     (ct_proof.cpp:2680-2692). The verifier appends BOTH `A·z0` (key relation, lines 3408-3417) AND
     `⟨b_sn, z0⟩` (serial relation, lines 3424-3431) to the transcript that derives the final challenge,
     and binds the recovered first-round commitments via the unconditional `pre_h2`/`post_h2`/`round1_aux`
     binding digests (3478-3542). Because both use the SAME prover `z0[inp]` and the SAME challenge,
     the relation forces `sn = ⟨b_sn, s⟩` for the SAME `s` with `A·s = pk_selected`. ⇒ a prover cannot
     present a serial decoupled from the spent key (the double-spend-via-fresh-nullifier route is closed
     **iff** the underlying lattice opening is sound).

## Vectors attempted

- **Two distinct valid nullifiers for one note (double-spend):** REFUTED at impl level. `sn` is a
  deterministic linear function of the note-derived `s`; `ComputeSmileSerialHash` canonicalizes coeffs
  via `mod_q` into [0,Q) (verify_dispatch.cpp:232) so wire-malleable representations of the same
  `sn mod q` hash identically. A *different* valid serial would require a different valid `s` opening the
  same coin/leaf — that is the lattice binding (soundness), not a code gap. Intra-proof duplicate serials
  are rejected with a domain-separated hash (`BTX_SMILE2_Dup_Serial_V1`, verify_dispatch.cpp:203-221).

- **Nullifier swap / replay (declare a payload nullifier ≠ the one the proof commits):** REFUTED.
  `CShieldedSpendAuthCheck` requires `proof_bound_nullifier == payload nullifier`
  ("bad-shielded-spend-auth-nullifier-mismatch", validation.cpp:1717-1730); the proof-bound value is
  re-extracted from the serial (validation.cpp:927-942). Cross-block/within-block spend caught by the
  NullifierSet (Insert/Contains, nullifier.cpp; checks at validation.cpp:5938-5946 per c5). Null serial
  rejected ("bad-smile2-proof-null-serial-number"). Proof is tx-bound (statement digest over the whole
  stripped tx + genesis/fork, per c5 §6) ⇒ a proof cannot be lifted to another tx.

- **Forge membership of a never-added note / prove vs wrong-or-old root:** REFUTED at impl level.
  Anchor must be a tracked historical root, fail-closed (IsShieldedAnchorValid, validation.cpp per c5
  §4); ring members reconstructed from the node's own tree snapshot, not attacker-claimed. Merkle hashing
  is domain-separated SHA256 (`BTX_Shielded_Branch_V1` / `_Empty_Leaf_V1`, merkle_tree.cpp:224-257) ⇒
  collision/second-preimage resistant; Append/Root/Witness follow standard Zcash-style frontier. Forging
  a leaf that collides with a real one requires SHA256 second-preimage. (Whether the lattice membership
  proof FORCES the same hidden index across membership+key+coin+serial is the one-of-many SOUNDNESS — see
  below.)

- **Ring / linkability (legacy MatRiCT, recovery path):** key image `KI = s·H_point(ring_member_cm)`,
  `nf = SHA256("BTX_MatRiCT_RingSig_Nullifier_V1" || KI)` (ring_signature.cpp:701-741); bound into the
  signed message `RingSignatureMessageHash` (input/output commitments, fee, nullifiers, tx_binding,
  line 685-699) AND `ComputeProofChallenge` (matrict.cpp:461). KI uniqueness + null/zero-norm rejection
  enforced (ring_signature.cpp:648-767). This path is **disabled post-fork** (IsShieldedMatRiCTDisabled);
  downgrade/replay to it is blocked — `ValidateSmile2Proof` requires exact wire_version match
  (LEGACY vs M4_HARDENED, verify_dispatch.cpp:119-125), `RejectMismatchedV2WireFamilyForFork` +
  `RejectRetiredMatRiCTEnvelopeAfterDisable` + `RejectDirectSendEncodingForFork` (validation.cpp:1157-235)
  gate it. No legacy-path replay surface.

- **Privacy / de-anon (#4):** note encryption is KEM-DEM with ML-KEM-768: fresh `kem_seed` per note
  (`GetStrongRandBytes`), HKDF-SHA256 → ChaCha20Poly1305 AEAD key+nonce derived from the per-encapsulation
  shared secret, KEM ct as AEAD AD (note_encryption.cpp:457-527). **No nonce reuse** (nonce is a
  deterministic fn of the unique-per-ct shared secret), no static-ECDH reuse (it's a KEM with fresh
  randomness). 8-bit `view_tag = sha3_256(ct||pk)[0]` is the intended Orchard-style trial-decrypt
  optimization (1/256 FP), not a leak. Decoy selection (ring_selection.cpp) is the post-Monero-audit
  redesign: shifted-Pareto age sampler, age-spread bins, exclusion-window forcing, Fisher-Yates shuffle,
  randomized real_index on duplicate occurrence (lines 452-475) — mitigates the known newest-bias /
  first-occurrence / excluded-window deanon attacks. SMILE path uses full-tree membership (no decoys) ⇒
  stronger. No de-anon primitive found.

- **EMPIRICAL double-spend / forged-membership submission:** BLOCKED by the prover SIGSEGV (below) — same
  blocker c5 hit. Could not build a spendable z→z tx to mutate (value_balance / second nullifier /
  forged anchor / lifted proof) and submit. The §arguments are therefore code-reasoned by-construction.

## Residual SOUNDNESS unknowns (crypto-break scope — DEFER to external audit, not code gaps)

These are the irreducible lattice-soundness questions my review surfaces precisely (so an auditor can
target them); I did NOT find, and do not claim, a break:

1. **One-of-many index consistency:** does the smile2 CT relation FORCE the same hidden ring index across
   (a) the key relation `A·s=pk`, (b) the coin/amount opening, (c) the account-leaf membership, and (d)
   the serial `⟨b_sn,s⟩`? The binding is via shared `z0`/`c`/`c0` + the recovered-aux/pre_h2/post_h2
   binding digests and the `account_challenges/alpha/gamma` row mixing (ct_proof.cpp:2346-2421,
   1952-1993, 3478-3588). MatRiCT/Esgin-style one-of-many proofs have historically had soundness
   subtleties (challenge space, rejection-sampling slack, knowledge-soundness of the recursive
   index decomposition `DecomposeIndex`). If (d) could be satisfied with an `s'` ≠ the `s` opening
   (a)-(c), a fresh nullifier for an already-spent note ⇒ CRIT double-spend. UNVERIFIED (needs the
   prover fixed + a forging harness, or a formal extractor argument).
2. **Norm-bound slack vs soundness:** the verifier checks `‖z0‖²<β0²`, `‖z‖²<β²` etc.
   (ct_proof.cpp:3310-3398) but the binding only constrains `z0 = y0+c0·s` in zero-knowledge; the
   extractor's `s` short-ness (hence serial well-definedness and key-image binding) depends on these
   bounds being tight relative to the challenge difference invertibility. Parameter soundness
   (Q=2^32−959, POLY_DEGREE=128, SLOT_DEGREE=4, KEY_ROWS=5, KEY_COLS=4) is auditor territory.
3. **Amount-digit range binding** (flagged by c5 too): whether the lattice relation forces every amount
   digit into {0,1,2,3} across 32 slots (no negative/overflow) — VerifyCT never sees cleartext.

## Refuted / robust (so others don't repeat)
- SMILE serial nullifier is canonicalized (mod_q) before hashing ⇒ no wire-malleability → distinct
  nullifier; intra-proof + within-block + cross-block duplicate detection all present; payload-nullifier
  must equal proof-bound nullifier (no swap); proof is whole-tx + chain bound (no lift/replay).
- Merkle accumulator hashing is domain-separated, collision-resistant SHA256; anchor history is tracked &
  fail-closed; membership verified against the node's own snapshot — no forged-membership at impl level.
- Note encryption (ML-KEM-768 + HKDF + ChaCha20Poly1305) has no nonce/KEM reuse; view-tag is intended;
  decoy selection is post-audit-hardened — no de-anon primitive.
- Legacy MatRiCT key-image path binds the nullifier into the signed message + Fiat-Shamir challenge and is
  fork-disabled with exact-wire-version + envelope + encoding gates ⇒ no downgrade/replay.
