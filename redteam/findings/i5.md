## 1. What the fix would have to change (the two-site prover/verifier challenge)

The monomial Fiat–Shamir opening challenge is produced by **one** function and consumed identically
by prover and verifier — that part the mission describes correctly:

- `HashToMonomialChallenge(data, len, domain)` — `ct_proof.cpp:354-369`: `c.coeffs[hash[0]%128] =
  ±1`. Challenge space = `2·128 = 256 = 2^8`. (Confirmed f3/e2.)
- PROVER derives `c0_chal` / `c_chal` from the recomputed transcript hash at `ct_proof.cpp:2678`
  (`seed_c0`) and `ct_proof.cpp:2943` (`seed_c`).
- VERIFIER derives them identically at `ct_proof.cpp:3300` (`c0_chal` from `seed_c0_check`) and
  `ct_proof.cpp:3456` (`c_chal` from `seed_c_final`). Round-trip requirement #1 (identical derivation
  prover↔verifier) is satisfied by construction *if* one only swaps the sampler — but that is the
  easy 1% of the change. The hard 99% is below.

A naive swap = replace those `HashToMonomialChallenge` calls with a Hamming-weight ternary sampler
(the SMILE2 ring already ships one: `HashToTernaryChallenge`, `membership.cpp:77-110`, P(0)=½,
P(±1)=¼ over 128 coeffs ⇒ `|C| ≈ 2^{0.81·128} ≫ 2^128`; or a fixed-weight variant). **That naive
swap does not compile-correctly into a sound proof, and at runtime it throws / rejects every honest
proof.** Here is why, with the exact blocking sites.

## 2. BLOCKER A — the verifier *divides by the challenge*; a non-monomial challenge makes that
   operation ill-defined and the honest path throws.

The SMILE2 CT extractor/verifier does not just compute `c·(stuff)` (linear, swap-safe). It
**recovers committed messages by INVERTING the challenge polynomial**:

- `DivideByMonomialChallenge(scaled, c) = NttMul(InvertMonomialChallenge(c), scaled)`
  (`ct_proof.cpp:371-374`).
- `InvertMonomialChallenge` (`ct_proof.cpp:66-100`) **throws `std::invalid_argument` unless `c` has
  exactly one nonzero coefficient** (lines 82-86: `if (monomial_index >= POLY_DEGREE ||
  nonzero_count != 1) throw`). It returns a *monomial* inverse (`a^{-1}X^0` or `−a^{-1}X^{d−k}`).

This is called **on the consensus verify path**, not just in the extractor proof:
- `RecoverFullAuxMessageCommitments` → `recovered[slot] = DivideByMonomialChallenge(scaled_commitment,
  c_chal)` (`ct_proof.cpp:1149`), invoked by `VerifyCT` at `ct_proof.cpp:3458`.
- `BuildFullAuxResidues` path (`ct_proof.cpp:1169` via `ComputeAuxMessageResidueFromCommitment`),
  invoked at `ct_proof.cpp:3544`.
- Compressed-W0 recovery: `DivideByMonomialChallenge(weighted_response − w0_residue_acc, c_chal)`
  (`ct_proof.cpp:3497`), in the per-input loop of `VerifyCT`.
- `DivideByMonomialChallenge(scaled_binding_acc, c0_chal)` in the coin-opening proof
  (`ct_proof.cpp:1941`), reached from `VerifyCombinedCoinOpeningProof` (`ct_proof.cpp:3686`).

**Consequence of the swap:** a Hamming-weight ternary challenge has ~64 nonzero coeffs ⇒
`InvertMonomialChallenge` immediately throws on the FIRST honest verify, so **every post-activation
shielded tx fails to verify** (round-trip BROKEN). To make it work you must replace
`InvertMonomialChallenge` with a **full ring inverse** in `Z_q[X]/(X^128+1)` (invert each of the 32
degree-4 CRT slots, `ntt.h:12-16`). That is doable *mechanically* (the ring is a field-product over
the NTT slots, so a uniformly-random / high-weight challenge is invertible w.h.p.), **but**:

1. The inverse is now a **dense degree-127 polynomial**, not a monomial. Every `recovered[...]` term
   that was previously a clean single-shifted message becomes a dense convolution. Whether the
   downstream binding-digest checks (`ComputeCtBindingDigest` PRE_H2/POST_H2/ROUND1_AUX,
   `ct_proof.cpp:3478/3531/3665`) and the weak-opening accumulator (`ComputeCtWeakOpeningAccumulator`,
   `ct_proof.cpp:3569`) still recover the *same* committed polynomials is **NOT obvious and is not
   preserved by a sampler swap** — it depends on the algebraic identities the whole proof was
   designed around the monomial for. This is an extractor re-derivation, not a parameter tweak.
2. A *fixed-weight* ternary challenge is **not guaranteed invertible** (a ternary poly can be zero in
   a CRT slot). The legacy MatRiCT path sidesteps this by **never inverting the challenge** (see §3);
   SMILE2 cannot, so a swap must also prove/ensure invertibility of every sampled challenge AND of
   every *challenge difference* `c−c'` (needed for the 2-transcript extractor), or restrict the
   sampler to an invertible-difference set — exactly the MatRiCT+ "partition-and-sample" machinery
   that does **not** exist in `src/shielded/smile2/`.

## 3. BLOCKER B — the "reuse the legacy `SampleChallenge` pattern" premise is FALSE for this proof
   system. The legacy fix worked because MatRiCT is *linear* in the challenge; SMILE2-CT is *not*.

The mission says to reuse the proven legacy fix. I checked it directly. The legacy
`SampleChallenge` lives in a **different ring and a different proof system**:

- `lattice::SampleChallenge` → `Poly256` (degree **256**, q=**8380417** Dilithium prime, fixed
  Hamming weight `BETA_CHALLENGE=60`), `lattice/sampling.cpp:91-...`, `lattice/params.h:27`. SMILE2 is
  `SmilePoly` (degree **128**, q=2^32−959, BDLOP). The types are not interchangeable.
- More importantly, the MatRiCT balance/range verifiers are **affine in the challenge**: e.g.
  `range_proof.cpp:50` `RESPONSE_NORM_BOUND = GAMMA_RESPONSE − BETA_CHALLENGE·SECRET_SMALL_ETA` and
  `balance_proof.cpp:97` "contribution c·b (whose inf-norm is at most BETA_CHALLENGE·||b||_inf)". The
  verifier recomputes announcements as `a = A·z − c·t` and checks a *hash* equality — it **never
  divides by `c`**. So widening `c` there only *subtracts more* from a norm bound (cheap, local) and
  never needs an inverse. That is why the legacy 23-bit→200-bit swap was a clean, self-contained fix.
- SMILE2-CT is **quadratic in the challenge and inverts it.** The framework relation
  (`ct_proof.cpp:3641-3663`) literally checks a polynomial identity in `c`:
  `selector_bin = f² + c·f` (binarity), `c_fg = c·f[G]`, `c_sq = c·c`, `c2h = c²·h2`,
  `bracket = framework_sum − c·f[G] − c²·h2`, plus the message recovery via `c^{-1}` (§2). Special-
  soundness here extracts the witness from **2–3 accepting transcripts with distinct challenges whose
  difference is invertible** — the monomial was chosen *specifically* so that distinct monomials
  always have invertible differences and `‖c·r‖∞ ≤ 1`. Swapping the sampler **changes the extractor**,
  the invertible-difference guarantee, and the norm algebra simultaneously. There is no "reuse"
  available; this is original cryptographic design + a security proof.

## 4. BLOCKER C — norm/parameter re-derivation is coupled and unproven (the explicit §2 requirement)

The σ and z-norm checks are hard-tuned to `‖c·r‖∞ ≤ 1`:

- `params.h:90-100`: *"With monomial challenge c = ±X^k, ‖c·r‖∞ ≤ 1 for ternary r, so σ only needs
  to hide a ±1 shift. σ = 55 (SIGMA_MASK) / 31 (SIGMA_KEY)"*; responses are **7-bit centered**
  (`Z_COEFF_BITS = Z0_COEFF_BITS = 7`).
- Verifier z-norm bounds: `β0² = SIGMA_KEY²·2·KEY_COLS·d` (`ct_proof.cpp:3316`), `β² =
  SIGMA_MASK²·2·rand_dim·d` (`ct_proof.cpp:3346`), plus per-vector Gaussian bounds (`ct_proof.cpp:
  3367-3368`). Prover rejection sampling is tuned to these σ (`ct_proof.cpp:2697, 2730`,
  `ComputeCtRejectionLogAccept`, `ct_proof.cpp:43-64`).

With a Hamming-weight-`w` ternary challenge, `‖c·r‖∞ ≤ w·‖r‖∞` (here `w ≈ 64` for the membership-
style sampler, or the chosen fixed weight). To keep the prover from aborting forever you must inflate
`σ_MASK, σ_KEY` by ≈`w` (and the centered-coeff width from ~7 bits to ~7+log₂w bits **per
coefficient**), then **re-derive `β0², β²` and every Gaussian bound consistently**, AND re-verify
that the inflated σ still gives M-SIS/M-LWE soundness at the live ring (the larger responses move the
SIS/LWE operating point). That coupled re-derivation (σ ↔ rejection rate ↔ norm bound ↔ extractor
slack ↔ BKZ cost) is **exactly the missing `C-002` security estimate**
(`doc/btx-audit-reassessment-remediation-plan-2026-03-23.md` item C-002, cited in f3.md). I can state
the *direction* of the change but I cannot *prove the resulting params are sound*, and shipping
guessed lattice parameters into a consensus inflation-bearing proof is unacceptable.

**Proof-size impact estimate (so the magnitude is on record):** responses dominate the CT proof.
Going monomial→weight-`w` ternary inflates σ by ~`w` (worst case) or ~`√w` (with a tighter infinity-
to-2-norm argument), i.e. centered-coeff width grows from 7 bits to ≈ `7 + log₂ w` bits. For `w≈64`
that is ≈13 bits/coeff (~+85% on the response vectors) before any extra parallel-repetition; with a
smaller deliberately-chosen weight (e.g. `w=16` like Dilithium-ish, IF invertible-difference can be
guaranteed) ≈11 bits (~+57%). Either way it is a material, not negligible, proof-size increase — and
it is *contingent on the extractor re-derivation succeeding at all*, which it does not here.

## 5. Could a *different* conservative fix avoid the inversion entirely?

I considered two genuinely-conservative alternatives that would NOT require rewriting the extractor:

- **(a) Parallel repetition of the monomial opening (κ rounds).** Soundness error `2^{-8κ}`; κ=16 ⇒
  `2^{-128}`. This *is* sound and reuses the existing monomial machinery verbatim (no inversion change,
  no σ change per round). BUT it multiplies the CT proof size and verify cost by ≈16× — on the active
  shielded path that is a large consensus-relevant blowup, and it is still a hard-fork bilateral change
  needing the same height gate, AND it changes the wire format / `serialize.cpp` layout. It is a
  *defensible* fix but it is NOT "small/conservative", it needs its own param + perf review, and most
  importantly **the per-round monomial soundness still rests on the same unproven M-SIS reduction
  (`C-002`)** — parallel repetition amplifies *challenge-entropy* error but does nothing for the
  *reduction-is-loose* failure mode f3 actually flags (per the cited Journal-of-Cryptology result in
  f3 §Sources, small-challenge FS soundness is only inverse-polynomial and does NOT amplify cleanly
  under FS without a one-shot negligible-error guarantee). So κ-repetition does not even cleanly close
  the stated gap. Rejected.
- **(b) Widen only the membership/relation challenges, leave the opening monomial.** Already done —
  the relation challenges ARE high-entropy (`DeriveSlotChallenges`, `DeriveRhoChallenges` ≈2^128,
  e2 §3). The residual `C-002` gap is specifically the *opening* (extractor) challenge, so (b) changes
  nothing. Rejected.

Neither alternative is a safe, verifiable, soundness-improving conservative swap I can ship today.
