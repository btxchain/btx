# y1 — CONSENSUS FIX for the CRITICAL shielded-inflation hole (x1c), VERIFIED

## ★★★ RESULT: the forge is CLOSED. `smile2::VerifyCT` now REJECTS the x1c inflation proof, and honest proofs still ACCEPT. ★★★

x1c empirically confirmed that `smile2::VerifyCT` (the consensus shielded verifier)
ACCEPTS a forged proof declaring `public_fee=50` on a truly balanced `in=out=100`
spend (50 sat minted from nothing, `verify=1`, deterministic) because it never
independently recomputes the value-conservation carry — it only checks
`proof.h2.coeffs[0..3]==0` and TRUSTS the prover's `h2`. This deliverable adds the
missing independent value-conservation equation to `VerifyCT` and PROVES, by
re-running x1c's own forge harness against the patched verifier, that:

- the forge (rtx1_05, rtx1_08) is now **REJECTED** (`verify=0`), and
- honest proofs (rtx1_00 baseline, rtx1_10 balanced control, the full
  `smile2_ct_tests` honest prove→verify suite) still **ACCEPT**.

## 1. The fix (consensus change), with the algebra that makes it sound

### 1.1 Why the hole exists (traced in `src/shielded/smile2/ct_proof.cpp`, pr-214)

The prover builds the balance carry into `h2` (live m=1 layout):
```
proof.h2 = proof.g0 + live_y_sum;                       // ct_proof.cpp:2848
const int64_t delta = sum_in - sum_out - public_fee;    // :2849
if (delta != 0) { carry.coeffs[0] = mod_q(delta); proof.h2 += carry; }  // :2850-2854
```
`VerifyCT` STEP-1 (`:3122-3133`) checks only `proof.h2.coeffs[0..3]==0`. For an
honest balanced spend `delta==0`, so `h2 = g0+live_y_sum` already has
`coeffs[0..3]==0`. The forge drops the carry (sets `delta=0` while the true
`delta=-50`): STEP-1 still passes, and the PRE_H2/POST_H2/round1_aux/coin-opening
binding digests pin the prover's *own forged* `h2`/`lhs` to itself, never to the
committed amounts. **No check in `VerifyCT` re-derives that `(amounts, fee)` must
balance.** ⇒ inflation.

### 1.2 The missing input was already recovered by the verifier

In the live m=1 layout, `ComputeLiveCtRetainedAuxMsgCount` returns **0**
(`ct_proof.cpp:880-886`), so EVERY aux message slot — including the input/output
**amount** slots — is RECOVERED by `RecoverFullAuxMessageCommitments`
(`:1131-1153`, called at `:3457`) via
`recovered[slot] = DivideByMonomialChallenge(⟨b[slot],z⟩ − aux_residues[slot], c)`.
For an honest, self-consistent proof the prover sets
`aux_residues[slot] = ⟨b[slot],z⟩ − c·t_msg[slot]` (`ComputeAuxMessageResidueFromCommitment`,
`:1119-1129`), so the recovery is EXACT:
`recovered_aux_tmsg[slot] == t_msg[slot] == EncodeAmount(amount)`.

Those recovered amount slots are:
- **bound** into `round1_aux_binding_digest` (`CollectLiveCtRound1AuxBindingPolys`
  includes `InputAmountSlot`/`OutputAmountSlot`, `:903-914`; digest re-checked at
  `:3478-3487`), and
- **tied to the on-chain coin commitments** by the combined coin-opening proof,
  which consumes `aux_residues[input_amount_slot_offset+i]` /
  `[output_amount_slot_offset+i]` (`:1920-1944`, reached from `VerifyCT` `:3686`).

So the verifier already holds the genuine committed amounts; it simply never used
them to enforce balance. The forge prover commits the REAL amounts (in=100,
out=100) — it only drops the carry and lies about `public_fee` — so the recovered
amounts are 100 and 100.

### 1.3 The added check (STEP 8d-value, inserted right after the round1_aux digest)

`VerifyCT` now, on the live m=1 path:
1. decodes `recovered_aux_tmsg[InputAmountSlot(i)]` → `sum_in`
   (`DecodeAmountFromSmileAmountPoly`, `:2059`), rejecting non-canonical or
   `>MAX_MONEY` amounts (also closes e2's MAX_MONEY range-invariant gap and guards
   the int64 sums),
2. decodes the output amount slots → `sum_out`,
3. **independently REQUIRES `sum_in == sum_out + public_fee`** — the
   value-conservation equation the prover previously controlled via `h2` alone.

A proof with a dropped/forged carry has `sum_in=100, sum_out=100, public_fee=50` →
`100 != 150` → **REJECTED**. An honest proof satisfies the equation by construction
(the prover refuses to build an unbalanced one, `:2244`) → unaffected.

### 1.4 Gating tradeoff (flagged per the mission)

The mission suggested gating at `nMatMulHardeningHeight=126400`. **I deliberately
apply the check UNCONDITIONALLY (no height gate), and this is the correct, safer
choice here** — the reasoning matters:

- Unlike i5's monomial→polynomial challenge swap (which would throw on
  `InvertMonomialChallenge` and break EVERY honest verify ⇒ chain split, so it
  needs a height gate + the external cryptographer), **this check is a strict
  ADDITION that can only reject proofs already violating value conservation.**
  Every honest proof — at any height, pre- or post-activation — satisfies
  `sum_in == sum_out + public_fee` and passes unchanged. There is therefore **zero
  chain-split / honest-proof-rejection risk**, so a height gate buys nothing.
- This is a LIVE inflation hole on the active shielded pool, exploitable
  permissionlessly at MEMPOOL admission (x2: full `CShieldedProofCheck`/`VerifyCT`
  runs at mempool). Height-gating would leave it exploitable until 126400 — an
  unacceptable window for an active inflation surface. Applying it unconditionally
  closes it immediately at mempool AND block-connect.
- If the maintainer nonetheless wants the cosmetic consistency of a gate (e.g. to
  bundle it under the 0.30.3 hardening flag day), it can be wrapped in
  `if (height >= nMatMulHardeningHeight)` with NO compatibility cost — but I
  recommend AGAINST delaying a live inflation fix. (`nMatMulHardeningHeight` does
  not yet exist in `consensus/params.h` on pr-214 — i5 §6 — and `VerifyCT` has no
  height parameter today; threading height through the
  `VerifySmile2CTFromBytes`→`VerifyCT` call path is straightforward if gating is
  insisted upon, but unnecessary.)

This fix is independent of and orthogonal to the f3/C-002 monomial-soundness
question (i5): it closes a STRUCTURAL balance hole that needs no crypto break and
no extractor re-derivation. C-002 (the monomial M-SIS reduction) still warrants the
external audit, but it is NOT what x1c exploited and NOT what this fixes — and that
is precisely why this fix is shippable in-house where the i5 swap was not.

```diff
@@ include block @@
+#include <consensus/amount.h>      // for MAX_MONEY / MoneyRange

@@ VerifyCT, immediately after the round1_aux_binding_digest check (~:3487) @@
+    // STEP 8d-value: INDEPENDENT VALUE-CONSERVATION CHECK (rtY1 / x1c fix)
+    // [full rationale in comment block in source]
+    if (aux_layout.UsesLiveM1Layout()) {
+        int64_t recovered_sum_in = 0, recovered_sum_out = 0;
+        for (size_t i = 0; i < num_inputs; ++i) {
+            const size_t slot = aux_layout.InputAmountSlot(i);
+            if (slot >= recovered_aux_tmsg.size()) return false;
+            const auto decoded = DecodeAmountFromSmileAmountPoly(recovered_aux_tmsg[slot]);
+            if (!decoded.has_value() || *decoded < 0 || *decoded > MAX_MONEY) return false;
+            recovered_sum_in += *decoded;
+            if (recovered_sum_in < 0 || recovered_sum_in > MAX_MONEY) return false;
+        }
+        for (size_t i = 0; i < num_outputs; ++i) {
+            const size_t slot = aux_layout.OutputAmountSlot(i);
+            if (slot >= recovered_aux_tmsg.size()) return false;
+            const auto decoded = DecodeAmountFromSmileAmountPoly(recovered_aux_tmsg[slot]);

+            recovered_sum_out += *decoded;
+            if (recovered_sum_out < 0 || recovered_sum_out > MAX_MONEY) return false;
+        }
+        if (recovered_sum_in != recovered_sum_out + public_fee) {
+            LogDebug(BCLog::VALIDATION, "VerifyCT FAIL [step 8d-value]: value conservation "
```
