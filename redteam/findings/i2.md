## FIX 1 — Fail-closed RNG (c3/e4, defense-in-depth)  [IMMEDIATE, non-consensus]
**Files:** `src/libbitcoinpqc/src/ml_dsa/utils.c`, `src/libbitcoinpqc/src/slh_dsa/utils.c`

**Change.** In both `custom_randombytes_impl` / `custom_slh_randombytes_impl`:
- `/dev/urandom` open-fail → was `memset(out,0); return` → now **`abort()`** (never fabricate a key
  from zeros).
- short `fread` → was zero-fill remainder → now **`abort()`**.
- injected-entropy path that previously WRAPPED/REUSED bytes when `outlen > remaining` → now
  **`abort()`** (no wrap/reuse); on success it copies the next `outlen` bytes verbatim and advances
  the offset (no modulo cycling).

**Why.** Matches LEARNINGS/e4: not production-reachable today (Core RNG is fail-closed and callers
pass ≥128B strong entropy; Dilithium signing is FIPS-204 hedged) — but a fail-OPEN RNG that
silently produces zero/repeated key material is a latent crypto footgun. Fail-closed mirrors Core's
`RandFailure()`→`abort()`. LOW severity, defense-in-depth, **immediate** (no consensus gate; wallet/
keygen-side only).

**Verification (standalone C harness against the REAL patched .c files, /tmp/rtI2/rng_harness.c,
slh_harness.c):**
```
[sufficient-entropy-no-wrap] aborted=0 first-32-bytes-match=1   (copies verbatim, no wrap)
[exact-entropy]              aborted=0 first-32-bytes-match=1
[insufficient-entropy-aborts] aborted=1                          (was: wrapped & reused — now ABORTS)
[offset-advance-no-reuse]    first_ok=1 second_aborted=1         (2nd call has too few bytes -> ABORT, no reuse)
RESULT: ALL PASS
[slh sufficient] aborted=0 match=1
[slh insufficient] aborted=1
[slh offset-no-reuse] first=1 second_aborted=1
RESULT: ALL PASS
```
Confirms: verbatim copy with no wrap, abort on insufficient entropy, offset advances without reuse.

## FIX 2 — Freivalds rounds 2→3, GATED at nMatMulHardeningHeight=126400 (b1)  [CONSENSUS, GATED]
**Files:** `src/consensus/params.h`, `src/kernel/chainparams.cpp`, `src/kernel/chainparams.h`,
`src/chainparams.cpp`, `src/chainparamsbase.cpp`, `src/pow.cpp`

**Change.**
- New consensus params: `nMatMulFreivaldsRoundsHardened{3}`, `nMatMulHardeningHeight{INT32_MAX}` +
  helpers `IsMatMulHardeningActive(h)` and `MatMulFreivaldsRoundsForHeight(h)` (returns the hardened
  count at/after the height, else the legacy `nMatMulFreivaldsRounds`).
- Both consensus Freivalds verify sites in pow.cpp (`CheckMatMulProofOfWork_Freivalds`,
  `...ProductCommitted`) now use `MatMulFreivaldsRoundsForHeight(block_height)` instead of the raw
  `nMatMulFreivaldsRounds`.
- Activation: **mainnet `nMatMulHardeningHeight = 126400`** (per PR_PLAN); test nets + regtest +
  shielded-v2-dev activate **from genesis (0)**; regtest CLI override
  `-regtestmatmulhardeningheight=<n>` added (mirrors the existing digest/binding overrides).

**Why.** k=2 ⇒ 2^-62 per-attempt forgery bound; k=3 ⇒ 2^-93 headroom. One extra O(n²) Freivalds
pass, negligible vs O(n³) mining. The round count is purely VERIFIER-side: an honest C'=A'·B' payload
passes any number of rounds, so the change does not alter the miner/payload format — but it IS a
consensus rule (a pre-fork node verifying with k=2 vs a post-fork node with k=3 could disagree on a
forged-but-2^-62-lucky block), hence **gated** so pre-126400 blocks are verified with exactly k=2.

**Verification (regtest, image rti2-pr214):**
- Mined 130 blocks with hardening active from genesis (regtest default) → all accepted (honest
  k=3-verified blocks): `height=130`, single valid tip. Confirms k=3 does not reject honest blocks.
- Built a node with `-regtestmatmulhardeningheight=100`; the arg was accepted and the node mined 120
  blocks **across the boundary** (blocks 1–99 verified k=2, 100+ verified k=3) — all accepted. Confirms
  the gate is height-conditioned and both sides accept honest blocks.
- Existing unit-test invariant preserved: `matmul_freivalds_tests.cpp` asserts mainnet
  `nMatMulFreivaldsRounds == 2` — still true (the base field is unchanged; only the height-gated
  effective count is 3).

## FIX 3 — MatMul payload deserialization bound (b2)  [IMMEDIATE, hardening]
**File:** `src/primitives/block.h`

**Change.** New `UnserializeBoundedMatMulPayload()` reads each MatMul payload vector
(`matrix_a/b/c_data`) by reading the compact-size count, rejecting `count > MATMUL_PAYLOAD_MAX_WORDS`
(= 2048² = 4,194,304 words = the largest legal n×n matrix) with `ios_base::failure
("matmul-payload-too-large")` BEFORE any bulk allocation, then `resize`+read. Used on the read path
for all three payload vectors; the write path is unchanged ⇒ on-wire format is byte-identical (same
compact-size prefix + LE32 words).

**Why.** b2 nicety: previously a junk length prefix could drive a transient allocation up to the
~16MB transport cap before the cheap dim check rejected. Bounding at deserialize removes that
transient. `MATMUL_PAYLOAD_ABS_MAX_DIM=2048` matches `nMatMulMaxDimension` on every network (no
override), so a legitimate block never trips the bound. Immediate (serialization hardening, not a
gated consensus rule — it only rejects payloads that were already invalid/over-long).

**Verification.** Full node builds; block (de)serialization compiles into consensus + node. Mining
130 regtest blocks (which serialize/deserialize the C' payload every block) succeeds, and no
`matmul-payload-too-large` log line appears for honest blocks (the n=64 payload = 4096 words ≪ bound).

## FIX 4 — c4 shielded refactor-proofing  [IMMEDIATE, hardening]
**Files:** `src/shielded/v2_bundle.h` (+ helper), `bundle.cpp`, `v2_bundle.cpp`, `v2_proof.cpp`,
`v2_send.cpp`, `v2_egress.cpp`, `v2_ingress.cpp`, `smile2/ct_proof.cpp`, `smile2/membership.cpp`

**Change (a) checked variant access.** New `shielded::v2::CheckedGet<T>(payload)` (const + non-const
overloads) does `std::get_if<T>` and throws a clean catchable `std::ios_base::failure
("shielded-payload-variant-mismatch")` on mismatch instead of `std::bad_variant_access` (which can
escape to `std::terminate`). ALL family-keyed `std::get<XPayload>(...)` on shielded `FamilyPayload`
variants were replaced with `CheckedGet` (tuple `std::get<N>` accesses were left untouched). This
makes the safety LOCAL/refactor-proof: today the gets are safe only because the switch key is derived
from the active alternative (`GetPayloadFamily`); if a future edit keys on the wire `family_id`, a
mismatch now rejects cleanly rather than terminating.

**Change (b) release-safe asserts.** The verify-reachable size/bounds `assert()`s c4 flagged
(ct_proof.cpp `SetAuxCommitmentSlot` slot bound, `ComputeCtWeakOpeningAccumulator` opening/aux/w0
counts; membership.cpp `ComputeFinalPublicXValues` `gamma_final.size()>=2`) are converted to real
runtime checks (`SmileVerifyRequire(...)` → throw `ios_base::failure`) that run under NDEBUG too. The
throw is caught at the shielded tx-accept boundary (`catch(std::exception)` in v2_proof.cpp), so a
violation rejects the tx instead of indexing out of bounds in release.

**Why.** Matches c4/LEARNINGS item 10/14: no current exploit, but the by-construction safety should
not depend on the NDEBUG-stripped asserts or a global derived-tag invariant.

**Verification.** Full node + bitcoin_shielded library builds green (ct_proof.cpp, membership.cpp,
all v2_*.cpp compiled). Mining/validation + a shield (z_shieldcoinbase-class) path exercises these
sites; regtest node ran without `variant-mismatch` rejects on honest traffic.

## FIX 5 — NeedsShieldedProofCheck explicit family (c5)  [IMMEDIATE, hardening]
**File:** `src/validation.cpp`

**Change.** `NeedsShieldedProofCheck` no longer infers "needs proof" from `HasShieldedInputs()`
(which for ingress is just `consumed_spends.size()`). New `ShieldedFamilyNeedsProofCheck(family)`
returns true for EVERY value-bearing / bridge-mint family explicitly — V2_SEND,
V2_SPEND_PATH_RECOVERY, **V2_INGRESS_BATCH**, V2_EGRESS_BATCH, **V2_REBALANCE**, V2_SETTLEMENT_ANCHOR,
V2_LIFECYCLE — and only V2_GENERIC returns false; the `default` (unknown/future family) **fails
closed (returns true)**. `HasShieldedInputs()` is kept as a belt-and-suspenders OR.

**Why.** c5: the old form was safe only via a structural invariant elsewhere
(`IngressBatchPayload::IsValid` forcing `consumed_spends≥1`). A future relaxation, or a new
value-bearing family, could silently skip proof verification ⇒ inflation. Now inference is by FAMILY
and fails closed.

**Verification.** Builds (after dereferencing the `std::optional<TransactionFamily>` return). The
previously-omitted V2_INGRESS_BATCH/V2_REBALANCE are now explicitly proof-checked; the only family
that skips is V2_GENERIC (value-less).

## FIX 6 — d3 shielded-rebuild DoS (MED→HIGH)  [IMMEDIATE, DoS]
**File:** `src/validation.cpp`

**Change.** The O(chain) self-repair (`RebuildShieldedState` ≈ 25-min full genesis→tip replay, and the
sibling O(chain) anchor-history rebuild) is **removed from the MemPoolAccept path entirely**:
- The recoverable-reject → `TryAutoRebuildShieldedState("MemPoolAccept", …)` retry block is deleted;
  a recoverable shielded-proof reject in the mempool now just `state.Invalid(...)` (rejects the tx).
- The `bad-shielded-anchor` → `TryAutoRepairShieldedAnchorHistory("MemPoolAccept", …)` retry is
  likewise deleted (an anchor the chain never created is peer misbehavior, not node-state corruption).
- The now-dead `shielded_state_repair_attempted` / `shielded_anchor_repair_attempted` locals in
  MemPoolAccept are removed.
Self-repair (both kinds) remains on the **PoW-gated `Chainstate::ConnectBlock` path and at startup**,
where the node's own state — not a peer's unconfirmed tx — is the input.

**Why.** d3: a cheap, never-mined mempool tx hitting a recoverable reject (notably the structural
`bad-shielded-ring-member-position` on attacker-controlled ring_positions) could force a ~25-min
full-chain rebuild, re-armable per shielded block. Restricting self-repair to block-connect/startup
closes the lever; dropping the structural "position absent" reject from the mempool recoverable path
is implied (the rebuild it fed is gone). This also satisfies d3 mitigation (c): a mempool tx
referencing a nonexistent ring position/anchor is a plain peer-misbehavior reject.

**Verification (code-structural — the decisive property; a malicious-bundle repro is blocked by the
c5 z→z prover SIGSEGV, noted by d3/c5):**
```
TryAutoRebuildShieldedState(...)        -> only at validation.cpp 6223, 6514  (both in Chainstate::ConnectBlock)
TryAutoRepairShieldedAnchorHistory(...) -> only at validation.cpp 5972        (in Chainstate::ConnectBlock)
grep '"MemPoolAccept"'                  -> ZERO occurrences (context string gone from rebuild/repair)
```
So no mempool path can reach the O(chain) rebuild. Regtest startup still performs the legitimate
genesis-init rebuild (height=-1) and normal mining never triggers a rebuild.

## FIX 7 — d4 service-challenge eviction-DoS + verify/redeem footgun (MED/LOW)  [IMMEDIATE, DoS/API]
**File:** `src/rpc/mining.cpp`

**Change (d4-1 never-evict-pending).** `PruneMatMulServiceChallengeRegistry` now, when over the 10k
count cap, evicts **only REDEEMED entries (oldest-redeemed first)** — it NEVER evicts an unredeemed,
unexpired challenge. (Expiry-pruning still runs first as the correct primary bound.) If every slot is
a live pending obligation, it leaves the registry over the soft cap rather than dropping a live
challenge. New `MatMulServiceChallengeRegistryFullOfPending()` detects that state; issuance then
**refuses** (`RPC_CLIENT_NODE_CAPACITY_REACHED`) instead of evicting a victim's solvable challenge.

**Change (d4-1 rate-limit).** A token bucket (5 issuances/s, burst 50), enforced under the registry
lock in `RememberIssuedMatMulServiceChallengeLocked`, bounds issuance rate so a flood (the attack
relied on ~71/s) cannot churn the registry. Over-rate issuance throws `RPC_CLIENT_NODE_CAPACITY_REACHED`.

**Change (d4-2 verify/redeem footgun).** `verifymatmulserviceproofs` now reports
`proof_valid` (crypto validity) AND `valid = proof_valid && !already_redeemed`; when the local
registry shows the challenge already redeemed, `reason = "already_redeemed"` and `valid=false`. The
result doc is updated to tell callers to authorize on `redeemmatmulserviceproof`, not on `valid`.

**Why.** d4: 10k cheap issuances (~2.4 min, no PoW) swept out legit PENDING challenges (proven
end-to-end by d4); and `verify` returning `valid:true` for an already-redeemed proof is a replay
footgun for apps that gate on `valid`.

**Verification (regtest, image rti2-pr214):**
- **Rate-limit:** a parallel burst of issuances → 25 refused with
  `"matmul service challenge issuance rate limit reached; retry shortly"` (RPC -34); after the bucket
  refilled, issuance resumed. (Serial cli calls are slower than 5/s so they never trip — the limit
  only bites a genuine flood, as intended.)
- **d4-2 verify-after-redeem:** issued → solved (`solvematmulservicechallenge`) → redeemed
  (`redeemmatmulserviceproof` → `valid:true, redeemed:true`) → then `verifymatmulserviceproofs`:
```
"proof_valid": true,
"valid": false,
"reason": "already_redeemed",
"redeemed": true,
(batch summary "valid": 0)
```
  Previously this returned `valid:true, reason:ok` — the footgun is closed.
- **d4-1 never-evict-pending:** code-level — the prune loop's victim selection skips any
  `!redeemed` entry and `break`s when no redeemed entry exists; the issuance path refuses when
  `MatMulServiceChallengeRegistryFullOfPending()`. (Reaching the 10k cap empirically is impractical
  in a quick regtest run; the eviction logic is provably pending-safe.)
