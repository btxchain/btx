# f3 — SHIELDED lattice/ZK cryptanalysis (SMILE2 / MatRiCT): the e1 tactic applied to the shielded scheme

Method: deep code review of `src/shielded/{smile2,ringct,lattice}`, cross-referenced to the
academic source papers (online research), plus the project's own internal audit docs under
`doc/`. No regtest forging prover was built — a *soundness* gap of this class is "the security
rests on an unproven/un-audited reduction," not a check that a mutated regtest tx can trip, so
the deliverable is an assessment + the exact hardness assumption, per the mission's "if sound,
state the assumption and flag for external audit." No production assets touched; no git commit.

---

## 1. Scheme identification (code ↔ paper)

BTX's shielded pool ships **two** distinct lattice ZK subsystems, both consensus-reachable:

### A. ACTIVE path — "SMILE2" CT proof (`src/shielded/smile2/`)
- Consensus-active proof kind for `V2_SEND` is `ProofKind::DIRECT_SMILE`
  (`v2_proof.cpp:1267,1307-1308`), verified by `smile2::VerifySmile2CTFromBytes`
  (`v2_proof.cpp:2510`) → `smile2::VerifyCT` (`ct_proof.cpp:3083`).
- Paper lineage: **SMILE — "Set Membership from Ideal Lattices…" (Lyubashevsky, Nguyen,
  Seiler, CRYPTO 2021, ePrint 2021/564)**, built on the **BDLOP commitment** (Baum–Damgård–
  Lyubashevsky–Oechsner–Peikert) — confirmed by `smile2/bdlop.h:19` and
  `doc/btx-smile-implementation-spec.md:75-92`. The set-membership / one-out-of-many proof
  (`smile2/membership.cpp`) is the SMILE log-size proof; balance is the MatRiCT-style "h2 carry"
  balance proof (`ct_proof.cpp:3591-3593`).
- Ring: `R_q = Z_q[X]/(X^128+1)`, **d=128**, **q = 2^32−959 ≈ 2^32** (`params.h:18,24`), split
  into l=32 NTT slots of degree 4. M-SIS/M-LWE module ranks **α=β=10** (`params.h:74-75`),
  key dims k=5/ℓ=4, ternary secrets, σ_mask=55 / σ_key=31, anonymity set 2^15.

### B. LEGACY path — MatRiCT/MatRiCT+ (`src/shielded/ringct/`, `src/shielded/lattice/`)
- Reachable only as `V2_SPEND_PATH_RECOVERY` / legacy-direct (`validation.cpp:870,1207`;
  `VerifyMatRiCTProof` in `ringct/matrict.cpp`).
- Paper lineage: **MatRiCT (Esgin–Zhao–Steinfeld–Liu–Sakzad, CCS'19, ePrint 2019/1287)** +
  **MatRiCT+ (Esgin–Steinfeld–Liu, S&P'21, ePrint 2021/545)** — the latter's
  "partition-and-sample" optimal challenge space with invertible challenge differences is the
  design `lattice/params.h:27 BETA_CHALLENGE=60` targets.
- Ring: `R_q`, **d=256**, **q=8380417** (the Dilithium prime), MODULE_RANK=4, VALUE_BITS=51,
  Hamming-weight-60 ternary polynomial challenges.

## 2. CRYPTANALYTIC FINDINGS

### F3-1 — **[HIGH, soundness-assumption]** Active SMILE2 CT proof gates commitment openings with a 256-value monomial Fiat-Shamir challenge; consensus soundness rests on an UN-VALIDATED M-SIS reduction that BTX's own auditors flagged (`C-002`) and deferred WITHOUT producing the required security estimate.

**The fact.** The CT opening / key relation uses a challenge `c = ±X^k`
(`HashToMonomialChallenge`, `ct_proof.cpp:352-369`): one nonzero coefficient ∈ {+1,−1} at one of
128 positions ⇒ **challenge space = 2·128 = 256 ≈ 2^8 values**. This single monomial drives the
Schnorr-style masked openings `z0 = y0 + c0·s`, `z = y + c·r`, `z_coin = y + c·r`
(`ct_proof.cpp:2678-2696, 2711-2727`, verifier `3300, 3456`, framework uses `c_chal` and
`c_chal²` at `3656-3661`). By contrast the membership/balance/binary *relations* are checked with
**high-entropy slot challenges** (`alpha/beta/gamma`, `DeriveSlotChallenges` → ~q^4 per slot ≈
2^128, `ct_proof.cpp:3596,3603,3650,3663`). So the protocol is asymmetric: the *statement*
constraints are protected by 2^128-ish entropy, but the *commitment opening* — the step that an
extractor relies on to pull out the witness — has special-soundness knowledge error **≈ 1/256 =
2^-8 per opening**.

**Why this is the e1-class issue (unproven hardness assumption).** A 256-element challenge space
cannot give negligible soundness *by challenge entropy*. The only way the CT proof is sound is if
soundness **reduces to M-SIS binding of the BDLOP commitment** (a lucky monomial challenge still
cannot open one commitment to two messages without an M-SIS collision) rather than to challenge
entropy. That reduction is plausible and matches the SMILE/MatRiCT design philosophy, BUT:
- It is **NOT proven for BTX's exact transcript/relation surface**. The academic SMILE proof is
  for SMILE's own (larger / structured) challenge distribution and its specific relation; BTX
  swapped in a *single monomial* "because `||c·r||∞ ≤ 1` for ternary r enables small σ and compact
  7-bit responses" (`params.h:90-100`, `ct_proof.h:102-121`) — a performance-driven divergence
  from the paper, exactly the kind of "we changed the primitive's parameter and assumed the proof
  still goes through" that e1 found on the PoW side.
- **BTX's own audit caught this and then deferred it unresolved.**
  `doc/btx-audit-reassessment-remediation-plan-2026-03-23.md:274-294` (item **`C-002`**) states
  verbatim: *"The CT proof path uses a monomial Fiat-Shamir challenge with only 256 possible
  challenge values… so CT soundness may be materially weaker unless the protocol relies on M-SIS
  hardness rather than challenge entropy."* Its "Future assurance checklist" (lines 290-292) lists
  the work that was **never done**: *"Confirm whether soundness… relies on M-SIS hardness,
  challenge entropy, parallel repetition, or a combination"* and *"Produce a concrete security
  estimate using the live BTX parameters and current monomial challenge distribution."* It was
  closed as "a documented protocol choice," i.e. assumed-good without proof.

**Exploitability / why I rate it HIGH not CRIT.** The realistic attack on a 1/256-knowledge-error
FS proof is **transcript grinding**: if a prover holds a commitment they *cannot* validly open
(e.g. an unbalanced tx, or a spend of a note they don't own) but which would pass for some subset
of monomial challenges, they re-randomize the masking nonce and re-hash the transcript ~256 times
to land an accepting `c`. **Whether this actually yields a forgery depends entirely on the M-SIS
reduction holding**: if every accepting transcript still implies a valid M-SIS-bound opening, the
grinder gains nothing and the scheme is sound; if there exists a relation among the framework
terms that a lucky monomial collapses (e.g. `c·t − c·t' = 0` for a non-trivial `t−t'` that is *not*
an M-SIS solution because the monomial annihilates it), it is an **inflation / spend-forgery
CRIT**. I could not close that question from code alone (it requires the missing security proof),
and neither could BTX's auditors — which is itself the finding. **The active, consensus-critical
shielded value-soundness rests on an unproven, internally-flagged, un-audited assumption.**
Mitigants that *reduce* (not eliminate) grinding risk, observed in code: the monomial seed is
full-transcript Fiat-Shamir bound and the verifier RECOMPUTES it (`seed_c0_check`/`seed_c_check`
vs `proof.seed_*`, `ct_proof.cpp:3290-3296, 3445-3452`), so the challenge can't be chosen freely —
a grinder must redo the whole commit each attempt; and the high-entropy slot challenges guard the
balance/binary relations independently. These make a naive forge hard but do **not** substitute for
the missing reduction proof.

**Recommendation.** EXTERNAL crypto audit must complete exactly the `C-002` checklist: a tight
special-soundness/knowledge-extractor proof for BTX's monomial-CT transcript reducing to M-SIS,
and a concrete BKZ-cost security estimate for the *combined* (challenge × M-SIS) soundness. If the
reduction does not hold tightly, replace the monomial with a Hamming-weight polynomial challenge
(as was already done for the legacy ring-sig/range/balance proofs — see F3-2) under a height-gated
hard fork. Until then, treat shielded value-soundness as resting on an UNPROVEN assumption.

### F3-2 — **[context / precedent]** This codebase has a documented history of shipping under-sized challenge spaces into consensus crypto and only catching them in later audit rounds.

The legacy MatRiCT path shipped with **~23-bit scalar challenges in BOTH the balance proof and the
range-proof relation** (`doc/btx-shielded-pool-formal-audit-round3-2026-03-05.md:19,85`;
round4 §1.2). Those were *upgraded to >200-bit polynomial challenges in Round 3*
(`ChallengeFromTranscript`, `SampleChallenge`). This proves (a) the team did ship a genuinely
exploitable ~2^-23 soundness error in a confidential-tx proof and (b) the *fix pattern is known and
already implemented elsewhere in this very repo* — which is precisely why the active SMILE2 path
*still* running on a 2^-8 monomial challenge (F3-1) is a credible, not theoretical, concern: the
same class of bug, un-fixed, on the path that actually validates today's shielded sends.

### F3-3 — **[INFO, sound]** Rejection sampling / abort handling is sound; no leakage or FS soundness loss.

Lyubashevsky-style abort-and-retry is handled correctly:
- The accept/retry decision (`ComputeCtRejectionLogAccept` + `RejectionSample`,
  `ct_proof.cpp:43-64,297-305`) is **prover-side only**; the function carries an explicit audit
  note that "the verifier never calls this function" and floating-point non-determinism is
  consensus-irrelevant because the verifier checks only the final `z` vector
  (`ct_proof.cpp:37-42`). The verifier independently enforces hard `||z||₂` norm bounds
  (`β0² = σ_key²·2·KEY_COLS·d`, `ct_proof.cpp:3310-3326`). ⇒ a leaked/biased abort cannot create a
  soundness hole because soundness is decided entirely on the public final transcript.
- Retries are bounded (`MAX_PROVE_CT_REJECTION_RETRIES=256`, `ct_proof.cpp:131`); legacy path uses
  `MAX_REJECTION_ATTEMPTS=512`, P(all fail)≈2^-432 (`round5 R5-... §`). The rejection-coin uses
  full 64-bit entropy ("H2 audit fix", `ct_proof.cpp:302-304`).
- Timing: an earlier audit flagged that rejection-loop iteration count correlates with `||c·s||∞`
  (round4 §1.6); constant-time `SampleBoundedPolyCT` exists but is not yet wired into the consensus
  prover. This is a **wallet/sender-side** side-channel (the verifier is deterministic), not a
  consensus soundness issue — matches c3/c5's "ref code not constant-time, wallet-side only."

### F3-4 — **[INFO, sound]** No Frozen-Heart / Fiat-Shamir weak-transcript bug specific to the lattice path.

I specifically chased the lattice-specific FS surface (e2 owns Frozen-Heart generally):
- All three challenge seeds (`seed_c0`, `seed_c`, `seed_z`) are `TranscriptHash(transcript)` over
  the full appended transcript and are **recomputed by the verifier**; a present-but-wrong seed is
  rejected (`ct_proof.cpp:3291, 3447, 3719`). The `seed_c_final = present ? proof.seed_c :
  seed_c_check` ternary (`3446`) is cosmetic, NOT a Frozen-Heart hole: if `proof.seed_c` is present
  it must equal `seed_c_check` (else reject at 3447) so the effective value is `seed_c_check` in
  both branches — the prover cannot inject a free challenge.
- `public_fee`/`value_balance` and the statement are bound into the transcript (confirmed by c5
  §1/§6), so the challenge depends on the whole statement — no grind-r-then-place-witness style
  attack on the *statement* (that protection is the high-entropy slot challenges; the residual is
  the *opening* knowledge error of F3-1, a different axis).
- `InvertMonomialChallenge` (`ct_proof.cpp:66-100`) is mathematically correct: in `Z_q[X]/(X^d+1)`
  every monomial `a·X^k` is invertible (`a^{-1}` for k=0, `−a^{-1}·X^{d-k}` for k>0, since
  `X^d=−1`). Differences of distinct monomials are invertible w.h.p. ⇒ the special-soundness
  *extractor exists* — it just has the 1/256 knowledge error of F3-1.

### F3-5 — **[INFO]** Parameter / BKZ-cost assessment of the underlying lattice assumptions.

- **Active SMILE2 (d=128, q≈2^32, ranks α=β=10):** binding/hiding reduce to M-SIS/M-LWE in
  dimension ~10·128=1280, claimed root-Hermite δ≈1.004 (`spec:90`) ⇒ ~128-bit core-SVP. The
  *assumption* (M-SIS/M-LWE at these params) is standard and plausibly ~128-bit; I found **no
  evidence the lattice params themselves are weakened below the paper's regime.** The risk is NOT
  the lattice hardness — it is the **soundness reduction** (F3-1) that converts "break the proof"
  into "break M-SIS." If that reduction is loose/absent, the 128-bit M-SIS wall is bypassed via the
  2^-8 challenge, not via BKZ.
- **Legacy MatRiCT (d=256, q=8380417, rank 4):** MODULE_RANK=4 over d=256 (effective dim 1024),
  audited at "~128–192-bit MLWE/MSIS" (round4 §1.4 / round5 §). Challenges now polynomial (>200-bit,
  F3-2). Bespoke, no external CVE/audit history.

## 3. Bottom line

- **Lattice PARAMETERS are not obviously weakened** (no sub-paper modulus/rank/norm); a pure BKZ
  attack below the claimed bits is **not** the threat.
- **The threat is SOUNDNESS, not hardness:** the consensus-active SMILE2 CT proof secures
  commitment openings with a **256-value monomial challenge (2^-8 per-opening knowledge error)**,
  and the scheme is sound **only if** a tight M-SIS reduction holds for BTX's exact monomial
  transcript. That reduction is **unproven, internally flagged (`C-002`), and deferred without the
  required security estimate.** This is the shielded-side analogue of e1's finding: a
  performance-driven parameter change to a primitive, with the security proof assumed rather than
  established. ⇒ **HIGH, and a candidate CRIT (shielded inflation / spend-forgery) pending the
  external proof; cannot be downgraded to "robust" the way c5 downgraded the *implementation*
  layer, because the gap is in the primitive's *soundness assumption*, which c5 explicitly punted to
  e2/e3/f3 + external audit.**
- **Rejection sampling, abort handling, and the lattice-specific Fiat-Shamir transcript are sound**
  (F3-3, F3-4). The only residual side-channel is wallet-side prover timing (non-consensus).

### Action items for external crypto audit (the un-done `C-002` checklist)
1. Tight special-soundness extractor for the SMILE2 monomial-CT transcript → reduction to M-SIS
   (prove a lucky monomial cannot collapse any framework relation off the M-SIS-bound surface).
2. Concrete combined security estimate (challenge knowledge-error × M-SIS BKZ cost) at the LIVE
   BTX params (d=128, q=2^32−959, α=β=10, σ=55/31, monomial challenge).
3. If the reduction is not tight: height-gated hard-fork swap of the monomial challenge for a
   Hamming-weight polynomial challenge (reuse the already-implemented legacy `SampleChallenge`/
   `ChallengeFromTranscript` pattern from F3-2).
