# h2 — SOFT-FORK hardening bundle: regression + attack-closure verification (EMPIRICAL)

Goal: implement the soft-fork bundle (a5 fix, drift→1800, cumulative drift cap, ASERT slew guard,
height-gated mining clamp + enable-on-all-nets) in a /tmp pr-214 build variant and verify each fix
WORKS and does NOT regress / badly interact. Re-prove a1 grind / a5 halt / c1 lock-out are closed.

## What I implemented (diff vs pr-214, in /tmp/rtH2/src)
1. **a5 fix.** New `Consensus::Params::MaxMatMulFutureBlockTime(height, prev_mtp, prev_blocktime, enforce_timewarp)`
   returns `max(prev_mtp+drift, prev_blocktime − MAX_TIMEWARP)` when timewarp is enforced (helper
   `ReconcileWithTimewarpFloor`). Used IDENTICALLY in: consensus `ContextualCheckBlockHeader`
   (validation.cpp), miner `GetMaximumTime`→`UpdateTime` (node/miner.cpp, via
   `MatMulFutureBlockTimeLimitForMining`), submitSolution (node/interfaces.cpp uses `node::GetMaximumTime`),
   and `getmininginfo` policy (rpc/mining.cpp). The cap can never fall below the BIP94 lower bound ⇒ the
   legal window is never empty ⇒ no unmineable activation block, and `UpdateTime` can no longer emit
   `min_time>max_time` (the c1 self-reject).
2. **Drift tighten** wired as a configurable value (default stays 3600; PR sets 1800 @125k).
3. **Cumulative windowed future-drift cap** in `ContextualCheckBlockHeader` ("time-cum-drift-too-new"):
   Σ over last K blocks of `max(0, block.time − prev_blocktime − target_spacing)` ≤ cap.
4. **ASERT slew guard:** wired `ApplyDgwSlewGuard` into the MatMul ASERT path (`MatMulAsert`, pow.cpp) —
   it was previously ONLY in the DGW path (a no-op for MatMul). Gated on `nDgwSlewGuardHeight`; caps a
   single block's difficulty change to ×4 vs parent.
5. **Height-gate the mining clamp:** `GetMaximumTime` now computes next-height + enforce_timewarp and
   returns nullopt below activation (matches consensus; fixes b6-F1/c1 mismatch).

## RESULTS (each fix: works + no regression)

### Fix 1 — a5 boundary chain-halt — CLOSED (empirical, rtH2-s1)
Reproduced a5's exact inversion on the PATCHED build: activation @205, drift 1800, bip94.
Pre-activation block 204 stamped high (height-gate fix means the miner no longer clamps below
activation, so the high stamp is reachable): time=1781006500, prev_mtp=1781000034.
- INVERSION CONDITION TRUE: timewarp-floor (prev_blocktime−600 = 1781005900) > drift-cap
  (prev_mtp+1800 = 1781001834). On pr-214 the legal window for block 205 is EMPTY ⇒ permanent halt (a5/c1).
- PATCHED build `getmininginfo` for block 205: `mintime=1781005900`, **`maxtime=1781005900`** (cap raised
  to the timewarp floor), `curtime_clamped:true`, `recommended_action:clamp_time`. Window = a single valid
  point, NOT empty.
- **Block 205 MINED OK** at time 1781005900; chain advanced to 205 and then +5 more (height 210). NO HALT,
  no self-reject. Honest mining continues normally post-activation.
RESULT: a5 halt CLOSED. Fix applied identically across consensus/miner/submitSolution (the accepted block
proves consensus accepted exactly what the miner emitted).

### Fix 2 — tighten drift 3600→1800 — works, honest blocks NOT clipped (rtH2-s2, rtH2-s2b)
- Honest ~90s spacing, 40 blocks: max(block.time − prev_mtp) over the run = **540 s** ≪ 1800 cap; zero
  "time-mtp-too-new" rejects. (Live mainnet honest figure ≤ ~1143s; 1800 leaves headroom.)
- Single-block difficulty move (clean measurement off the floor, HL=3600):
  - drift=3600: one max-drift block eases difficulty **×1.97** (≈×2).
  - drift=1800: one max-drift block eases difficulty **×1.39** (≈×1.41, matches 2^((1800−90)/3600)).
- Boundary tests at 1800: a legit catch-up block at prev_mtp+1700 (under cap) ACCEPTED (not clipped);
  an over-stall (clock at prev_mtp+5000) ACCEPTED after the miner clamped time to the cap (liveness preserved).
RESULT: drift=1800 does not clip honest/catch-up blocks; cuts the per-block difficulty lever ~×2→~×1.41.

### Fix 3 — cumulative windowed future-drift cap — works, honest-safe (after fixing TWO interaction bugs I found, see below)
On the fully-patched build (rtH2-s4a/s4c/s4b/cf):
- (A) honest ~90s × 30 with cumcap=7200/6: ok=1, NO cum-drift reject.
- (C) TIGHT cumcap=3600/6, honest ~90s × 25: ok=1, NO false-positive.
- (D) legit catch-up after stall (gap ~1700s, under per-block cap), tight cumcap: ACCEPTED.
- (cf) POSITIVE proof the consensus check is LIVE: cumcap=500/3, a block with genuine ~1610s forward-drift
  excess (> cap) is REJECTED with `time-cum-drift-too-new` (debug.log confirms); a normal block right after
  is ACCEPTED (chain LIVE, not wedged).
- (B/jump) SIZING OBSERVATION (not a defect): with cumcap=7200/6, sustained max-stamping is NOT rejected —
  because the per-block cap (block.time ≤ prev_mtp+drift) + the slow MTP advance SELF-LIMIT forward drift:
  after one forward jump the per-block cap pins subsequent blocks back to ~target spacing until MTP catches
  up, so cumulative excess never exceeds ~one jump per window. ⇒ the cumulative cap only adds protection
  beyond the per-block cap when sized `cumcap/window < per-block-drift`; otherwise it is pure
  belt-and-suspenders (corroborates a2: the 2h clock bound + per-block cap already bound cumulative lead).
RESULT: cum-cap is wired, fires on genuine over-window drift, and never false-rejects honest/catch-up/recovery
blocks. PR guidance: size cumcap modestly above K×(realistic honest excess) but below K×(per-block drift).

  change to ×4 precisely. This also bounds the future-drift easing lever and the c1 min-gap spike rate.
RESULT: slew guard works (caps ×4), is honest-safe (no change to sub-×4 honest transitions, no oscillation),
and required a code fix (wiring into MatMulAsert) without which enabling the height did nothing for MatMul.

## ★ TWO INTERACTION BUGS FOUND & FIXED in the cumulative-cap (each a fail-CLOSED chain-halt)
Both were caught empirically; each rejected EVERY block from activation, halting the chain — i.e. a worse
version of the very a5 halt this bundle fixes. (Important lesson: a naive cumulative cap is dangerous.)

BUG 1 — MTP-lag metric (false-positive on every honest block). First impl summed `max(0, block.time −
prev_mtp)` per block. block.time − prev_mtp is dominated by the structural MTP lag (MTP = median of last 11
blocks, ~5–6 blocks behind the tip), so each honest term is several hundred seconds and the windowed sum
blows any sane cap ⇒ honest block at height 3 rejected (rtH2-s4 run 1). FIX: measure *excess inter-block
gap* `max(0, block.time − prev_blocktime − target_spacing)` (≈0 for honest at-/under-target blocks).

BUG 2 — genesis/activation-boundary fold-in. With the inter-block metric, the window walk still reached
across the genesis boundary; block_1.time − genesis.time is the entire gap from the (ancient hardcoded)
genesis epoch to the first real block (~hundreds of millions of s) ⇒ STILL rejected every early block
(debug.log: `time-cum-drift-too-new` at AcceptBlockHeader). FIX: only count a pair when BOTH the block and
its parent are at height ≥ activation height (never measure across the activation/genesis boundary), and
clamp each per-block term to the per-block drift cap.

After BOTH fixes: honest/catch-up/recovery blocks pass (Fix 3 A/C/D), the check still fires on genuine
over-window drift (Fix 3 cf), the chain is never wedged. PR guidance: define the cumulative cap on
inter-block gaps (not distance-above-MTP), gate the look-back at the activation height, size it modestly.
