# a7 — PR's new RPC / chain-guard / NTP surface (non-consensus)

Empirical A/B (pr-214 vs main), 3-node regtest, now torn down. Protected assets untouched.

## Vectors attempted
- **C5a getblocktemplate throws on chain-guard pause** (NEW in PR). main returns a USABLE template + informational `chain_guard{should_pause_mining:true}` (deliberate bypass). pr-214 instead **throws RPC -9 "mining paused by chain guard"** (src/rpc/mining.cpp ~7240). getmatmulchallenge: both branches error when paused; PR change there is cosmetic (categorized -9 vs -1).
  RESULT: PR-INTRODUCED REGRESSION for GBT pollers (BIP22/23): hard error vs readable template. Severity low-med (operational/compat, non-consensus).
- **C5b chain-guard griefing (eclipse → pause)** — REPRODUCED. `should_pause_mining = enabled && !healthy`. Unhealthy triggers (src/.../mining_guard.cpp): tip_uninitialized, IBD, network_inactive, `insufficient_peer_consensus` (counts **OUTBOUND only**, < `-miningchainguardminpeers` default 2; inbound skipped), local_tip_ahead/behind_peer_median (gap > `-miningchainguardmaxmediangap` default 6). Drove a healthy guard-on victim to paused by stalling/removing its OUTBOUND peers → both GBT and getmatmulchallenge threw -9, mining halted. Recovery automatic when a counted outbound peer returns.
  RESULT: cheap, sustained mining-halt DoS. Scope-limiters verified: guard is OFF by default on test chains (`default_enabled=!IsTestChain()`), ON by default mainnet; pause is mining-RPC-only — node keeps validating/relaying/syncing, so **NOT a chain split**, a local mining-availability DoS. Severity MED for self-hosted miners w/ thin/attacker-controlled outbound peering.
- **C6 NTP / wall-clock skew** — REPRODUCED thresholds (mocktime). MAX_FUTURE_BLOCK_TIME=2h (chain.h:30) vs NodeClock::now(). Backward skew: −1h50m OK; **−2h10m → node rejects incoming honest blocks as "time-too-new" and stalls**; fixed instantly on clock correction. Forward skew receive: +3h OK (blocks in its past). Forward skew mine: +3h miner's own blocks orphaned until real time catches up.
  RESULT: >2h backward = desync; >2h forward = self-orphan. **PRE-EXISTING upstream behavior, NOT a PR regression.** The PR's MTP rule is keyed to median-time-past (not local clock), so a skewed local clock does NOT trip "time-mtp-too-new"; only the pre-existing 2h wall-clock check is skew-sensitive.
