# c4 — Shielded/bridge POISON-BLOCK & crash surface (variant misuse, NDEBUG asserts, codec/tree crash)

Scope: the leads c1 flagged but did not finish — `std::get<>`/`std::variant` misuse on
shielded/bridge payloads (bad_variant_access → terminate), NDEBUG-stripped security asserts in
smile2 ct_proof/membership, codec/nullifier/commitment-tree crash (div0/unbounded/OOB/uncaught),
and bridge view-grant decrypt. c2 already cleared the network-deserialize layer (clean
ios_base::failure → ban). Method: exhaustive code reading on `pr-214` of src/shielded/{bundle.cpp,
v2_bundle.{h,cpp}, v2_send.cpp, v2_egress.cpp, v2_ingress.cpp, v2_proof.cpp, smile2/*}. Empirical
sanitizer build NOT performed — see "Empirical note" — but the safety arguments below are
by-construction (size-check-before-index, family-derived-from-active-alternative) and do not need a
crash repro to refute, because the alleged crash primitives are provably unreachable.

## Vectors attempted

### 1. std::variant misuse → std::bad_variant_access (c1's PRIMARY lead, bundle.cpp:510)
- **The whole `std::get<X>` family on shielded payloads is SAFE. REFUTED.** Audited EVERY
  `std::get<...>` on `FamilyPayload`/shielded payloads (bundle.cpp:65-95,510-870; v2_bundle.cpp
  :2354-2448,2925-2939,3599-3622,3690-3757,4000-4097,4186-4200; v2_send.cpp:189; v2_egress.cpp:593;
  v2_ingress.cpp:1643..2422; v2_proof.cpp:1248,1288,1609..2533,1815).
- **Why no bad_variant_access:** `FamilyPayload = std::variant<SendPayload, …, SettlementAnchorPayload>`
  (v2_bundle.h:1457). The "family tag" used to key every `std::get` is **derived FROM the active
  alternative**, not from wire bytes: `GetBundleSemanticFamily(b) → GetPayloadFamily(b.payload)`
  (v2_bundle.cpp:1144-1153,1183-1186) is a chain of `holds_alternative<T>` tests. So
  `switch(GetPayloadFamily(payload)){ case V2_SEND: std::get<SendPayload>(payload); … }` selects the
  case that matches the *actual* active alternative — the get can never disagree. This is
  tautological type-safety. c1's specific bundle.cpp:510 `std::get<SendPayload>` sits under
  `case TransactionFamily::V2_SEND` of exactly this switch ⇒ active alternative IS SendPayload there.
  RESULT: cannot throw.
- **The wire family tag CANNOT desync the variant.** On deserialize, the active alternative is set by
  `DeserializePayload(s, wire_header.family_id)` (v2_bundle.h:1595,1687-1730): the wire `family_id`
  only *chooses which alternative to construct*; the variant then holds exactly that type, and any
  unknown/`V2_GENERIC`/invalid family_id throws `ios_base::failure` (1727,1729) — a clean reject, not
  a get. `header.family_id` is thereafter used ONLY in bool gates (`WireFamilyMatchesPayload`,
  `IsGenericTransactionFamily`), **never** as a `std::get` key. So a crafted `family_id ≠ active
  alternative` is caught by `WireFamilyMatchesPayload` (returns false → bundle rejected) and still
  cannot mis-route a `std::get`. RESULT: no desync primitive.
- **Locally-guarded sites** (v2_send.cpp:185→189, v2_ingress.cpp:1639→1643/2416→2422,
  v2_proof.cpp:1605/1633/1864/1907/2078/2151/2203/2331/2415/2526) each have a matching
  `!std::holds_alternative<T>(payload) → reject` immediately before the `std::get<T>`. SAFE.
- **The one un-locally-guarded helper, BindV2SendProof (v2_proof.cpp:1815 `std::get<SendPayload>` when
  witness.use_smile) — SAFE:** its only two production callers (v2_proof.cpp:1876,1892) are inside
  `ParseV2SendProof`, which gates on `holds_alternative<SendPayload>` at 1864 first. (The other
  callers are unit tests, not consensus-reachable.) Confirmed by `grep` of all callers.
- **valueless_by_exception (the last theoretical bad_variant_access path) — IMPOSSIBLE.** Default
  `FamilyPayload` holds the first alternative `SendPayload` (never valueless). Each alternative
  (SendPayload et al., v2_bundle.h:623+) contains only PODs/`uint256`/`std::vector` ⇒ noexcept move ⇒
  variant assignment-from-complete-object is noexcept and cannot go valueless. Deserialize fills a
  *local* `payload_obj` and only then assigns it into the variant, so a mid-deserialize throw leaves
  the prior variant intact. RESULT: variant is never valueless ⇒ no get can throw on that ground.
- Builder-side sites (v2_egress.cpp:593, v2_bundle.cpp builders) operate on a *locally constructed*
  EgressBatchPayload (`bundle.payload = payload;` a few lines up) — not attacker-deserialized. SAFE.

### 2. NDEBUG-stripped asserts guarding a security/memory invariant on the verify path
- **REFUTED across smile2 ct_proof.cpp (15 asserts), membership.cpp (2), bdlop.cpp (2).** Every
  assert is one of: (a) PROVER-internal only (not reachable from any Verify entry), (b) a redundant
  duplicate of a real `throw`/`if-return` that runs in release too, or (c) a check on a
  VERIFIER-DERIVED quantity (Fiat-Shamir challenge counts, layout slot counts) that is deterministic
  and additionally pinned by an upstream runtime `if(...size()!=...) return false`.
- Classified each:
  - ct_proof.cpp:85-86 `InvertMonomialChallenge` — DUPLICATE of the `throw std::invalid_argument`
    at 82-83 (comment at l.68 explicitly: "Guard … in release builds as well as debug"). Release-safe.
  - ct_proof.cpp:853-869 `ComputeNumAuxMsg`, :2425 `TryProveCT`, :979-982 `ComputeCtWeakOpeningAccumulator`,
    :1487 `ComputeCtCompressedPublicX`, :1684 `SetAuxCommitmentSlot`, :2425/:3242 — all compare two
    quantities both computed from the same `(num_inputs,num_outputs,rec_levels)` triple, or are inside
    `TryProveCT` (the prover). Internal-consistency, attacker cannot violate.
  - **The genuinely attacker-facing indexing in VerifyCT is bounded by REAL checks, not the assert.**
    `VerifyCT` (ct_proof.cpp:3083) gates `num_inputs/outputs ≤ MAX_CT_INPUTS/OUTPUTS` (3114) and then
    `if (proof.aux_commitment.t_msg.size() != n_aux_msg) return false` (3247), `aux_residues.size()
    != n_aux_msg` (3252), `w0_residue_accs.size() != num_inputs` (3257), `z0[inp].size() != KEY_COLS`
    (3302), `aux_commitment.t0.size() < MSIS_RANK` (3169), serial/z0/input_tuples/output_coins size
    pins (3136-3168) — ALL before any layout-slot index. With `n_aux_msg == TotalSlots()` and every
    layout slot `< TotalSlots()`, the indexing is in-bounds by runtime check. The `assert(n_aux_msg ==
    TotalSlots())` is pure sanity (both sides derived identically).
  - membership.cpp:648 `ComputeFinalPublicXValues` `assert(gamma_final.size()>=2)` — `gamma_final` is
    verifier-derived `DeriveSlotChallenges(transcript, …, m+1)` (membership.cpp:1590) and is only
    reached inside `if (m>1)` ⇒ size = m+1 ≥ 3 by construction; not attacker-sized. And every
    `t_msg[slot]` access has an explicit `if (slot >= proof.commitment.t_msg.size()) return false`
    real bounds check (1569,1586,1595). membership.cpp:1150 is in a Prove function.
  - bdlop.cpp:224-225 — commit-side (prover) size sanity.
- RESULT: no assert found that is the SOLE guard of an attacker-violable memory/bounds invariant on a
  consensus verify path. The dominating pattern is "real `if/return false` (or `throw`) does the
  enforcement; the assert just restates it."

### 3. smile2 / codec / nullifier-commitment-tree crash (div0 / unbounded / OOB / uncaught)
- **Div-by-zero: REFUTED.** Every divisor/modulus in smile2/serialize.cpp & ntt.cpp is a compile-time
  constant (`POLY_DEGREE`, `SLOT_DEGREE`, `NUM_NTT_SLOTS`, `Q`, `8`) — no attacker-controlled divisor.
  NTT product/column sizing has explicit overflow guards (ntt.cpp:136,150 `> SIZE_MAX/NUM_NTT_SLOTS`).
- **OOB index in verify: REFUTED.** VerifyCT/VerifyMembership perform exhaustive structural size
  pinning (above) before indexing; commitment-tree `t_msg[slot]` reads are each preceded by a
  `slot >= size() → return false`. `proof.z0[inp].size()`, `coin_rings[inp].size()==N`, etc. all
  checked. anon_set bounded `N∈[1,NUM_NTT_SLOTS]`/`rec_levels==1` (3097-3108).
- **Unbounded loop/recursion / huge alloc: REFUTED on verify.** Loop bounds are min()-clamped to the
  key/layout (`min(aux_t0.size(), MSIS_RANK)`, `c < cols_next && c < sub_tensor.size()`) and
  `num_inputs/outputs` capped at MAX_CT_*. No attacker-set loop bound; allocations are tied to the
  capped, size-checked vectors. (Re-confirms c2's "bounded-before-alloc" at the smile layer.)
- **Uncaught exception other than the clean ios_base::failure: none found new.** Throws on the verify
  path are size/version `ios_base::failure` (caught at the tx-accept boundary, per c2) or the
  `invalid_argument` in InvertMonomialChallenge which is unreachable post the size/monomial checks.
- This domain also carries a large adversarial test corpus (test/smile2_*adversarial*, _extreme_,
  _realworld_attack_, shielded_v2_adversarial_proof_corpus) — consistent with the hardened reading.

### 4. Bridge view-grant / decrypt metadata path
- **DecryptViewGrant (bundle.cpp:231-280): REFUTED, and NOT consensus/relay-reachable.** Size-gates
  `encrypted_data` to `[EXPANSION, MAX_VIEW_GRANT_ENCRYPTED_DATA_SIZE]` (236-241) BEFORE the
  `size()-EXPANSION` subtraction (no underflow). `grant.nonce` is a fixed `array<uint8_t,12>`
  (ReadLE32/64 in-bounds). `mlkem::Decaps(kem_ct, sk)` and `AEAD::Decrypt` take fixed-size /
  span-bounded inputs; failure → `nullopt`. Critically this path requires the operator's
  `mlkem::SecretKey` ⇒ it runs only in a local wallet/operator decrypt context, never during
  block/tx validation ⇒ not a network poison vector. The consensus-side view-grant checks
  (bundle.cpp:470-473: size bounds only, no decrypt) are pure bounds gates.

## NEW / unmitigated findings
- **NONE of severity crit/high/med.** No reachable `std::bad_variant_access`, no NDEBUG-only security
  assert, no div0 / OOB / unbounded / uncaught-crash primitive on any consensus- or relay-reachable
  shielded/bridge validation path. The shielded poison-block / network-DoS surface c1 flagged is, on
  audit, ROBUST.
- **LOW / defense-in-depth (optional, NOT vulnerabilities), worth folding into the hardening PR:**
  1. Replace the consensus-path `std::get<T>` calls that are only *implicitly* safe (keyed on
     `GetPayloadFamily`) with `std::get_if<T>`/`holds_alternative` + explicit reject, OR add a
     `default:`/`assert(false)` arm, so the safety is local and refactor-proof. Today correctness
     depends on the global invariant "switch key == active alternative"; a future edit that keys a
     switch on `header.family_id` (the wire tag) instead of `GetPayloadFamily` WOULD introduce a
     reachable bad_variant_access. The `BindV2SendProof` (v2_proof.cpp:1815) helper especially should
     guard locally rather than relying on its callers.
  2. Convert the handful of release-stripped asserts that *restate* a security/size invariant
     (ct_proof.cpp:1684 `slot < t_msg.size()`, :979-982, membership.cpp:648) into real
     `if(!inv) return false;`/`throw` even though they are currently unreachable-with-bad-data — pure
     hardening so the invariant survives a future caller that doesn't pre-check. (Matches c1's and
     b1's "asserts compiled out under NDEBUG" defense-in-depth theme.)

## Refuted / robust (so others don't repeat)
- **bundle.cpp:510 `std::get<SendPayload>` (c1's primary lead) — REFUTED.** The family is derived from
  the active variant alternative (`GetPayloadFamily` = chain of `holds_alternative`), so every
  family-keyed `std::get` matches the active alternative tautologically; wire `family_id` desync is
  caught by `WireFamilyMatchesPayload` and never keys a get; the variant is never valueless
  (noexcept-movable POD/vector alternatives, local-object assignment). No bad_variant_access reachable.
- **smile2 ct_proof/membership NDEBUG asserts — REFUTED as poison.** Each is prover-only, a duplicate
  of a real throw, or a check on verifier-derived/upstream-runtime-pinned sizes. The real bounds
  enforcement on VerifyCT/VerifyMembership is `if(size()!=...) return false` and `if(slot>=size())
  return false`, present in release.
- **smile2 codec div0/OOB/unbounded/uncaught — REFUTED.** Constant divisors only; exhaustive size
  pins before indexing; min-clamped loops; overflow-guarded NTT sizing.
- **Bridge view-grant decrypt — REFUTED + out-of-band:** size/underflow-guarded and gated behind the
  operator secret key (not block/tx validation).
