## The unshield validation surface (one map, with file:line on pr-214)

An unshield is a `V2_SEND` with: shielded spends (≥1), ≥1 shielded output (the mandatory
change note), transparent `vout`, and `SendPayload.value_balance > 0` (value leaves the pool).
There is exactly **ONE** `value_balance` field and it is consumed identically by every check:

1. **Structure (`SendPayload::IsValid`, `v2_bundle.cpp:714-736`)** — run unconditionally via
   `CheckTransaction→CheckStructure` for every tx:
   - `MoneyRangeSigned(value_balance)` (720) ⇒ value_balance ∈ [−MAX_MONEY, MAX_MONEY]
     (MAX_MONEY overflow on the z-side is structurally impossible).
   - with spends: `value_balance < fee → reject` (729) ⇒ unshield value_balance ≥ fee ≥ 0
     (negative-fee / negative-value_balance inflation is structurally impossible).
   - `outputs.empty() → reject` (716) ⇒ an unshield MUST carry ≥1 shielded output (change).
   - intra-tx nullifier uniqueness `IsNonNullAndUnique` (744).
2. **SMILE2 proof balance (the TRUTH of value_balance)** — `VerifySmile2CTFromBytes(...,
   payload.value_balance, ...)` (`v2_proof.cpp:2510-2517`) feeds that SAME field as `public_fee`
   into the h2 balance relation `Σ input notes == Σ output notes + value_balance`. Lying about
   value_balance breaks the proof.
3. **Transparent conservation (`CheckTxInputs`, `tx_verify.cpp:203-228`)** —
   `adjusted_in = nValueIn + value_balance` (here nValueIn=0, vin empty); rejects
   `adjusted_in < value_out` (`bad-txns-in-belowout`); `CheckedAdd` + `MoneyRangeSigned` +
   `MoneyRange` guards. Reads `GetShieldedTxValueBalance` = same `payload.value_balance`
   (`bundle.cpp:1102`).
4. **Turnstile pool≥0 (`ShieldedPoolBalance::ApplyValueBalance`, `turnstile.cpp:11-26`)** —
   `new = m_balance − value_balance`; `new < 0 → reject` ("shielded-pool-balance-negative").
   Reads `TryGetShieldedStateValueBalance` = same `payload.value_balance` (`bundle.cpp:1050`).
   Applied at MEMPOOL (`validation.cpp:3302`) AND ConnectBlock (`6046`); persisted (`6624`).
5. **Whole-tx statement-digest binding** — `ComputeV2SendStatementDigest` hashes
   `TX_WITH_WITNESS(stripped tx)` incl. all transparent `vout` (`v2_proof.cpp:1140-1189`);
   verifier recomputes it (`validation.cpp` shielded `DescribeV2SendStatement` →
   `ParseV2SendProof` → `ProofEnvelopesMatch`, `v2_proof.cpp:1868`).

⇒ One field, five consistent consumers, no way to present different value_balances to
different checks. Breaking value conservation requires forging the SMILE2 proof (x1).

## Vectors attempted

### ATTACK 1 — Double-spend the same shielded note via unshield
- **Within one block (two unshields, same nullifier):** REJECTED.
  `ConnectBlock` collects `CollectShieldedNullifiers(bundle)` (family-agnostic,
  `validation.cpp:5937`) and `block_nullifiers.insert(nf).second` ⇒
  `bad-shielded-nullifier-duplicate` (5939-5942). Also caught earlier in mempool/package:
  `m_package_shielded_nullifiers.insert` ⇒ `bad-shielded-nullifier-package-conflict` (3197),
  and per-tx `invalid_in_tx` (`txmempool.cpp:684`).
- **Across blocks (unshield N, then unshield N again later):** REJECTED.
  `m_chainman.IsShieldedNullifierSpent(nf)` against the persisted NullifierSet ⇒
  `bad-shielded-nullifier-spent`, unconditional in ConnectBlock (5943-5946) AND at mempool
  (3328-3329). The set is written on connect (`Insert`, 6578).
- **Via reorg (unshield N on a fork, then N again on the main chain):** REJECTED as a
  double-spend; ANALYZED in detail — this is the sharpest case in my brief.
  - On `DisconnectBlock` the fork-block's nullifiers are `Remove`d from the persisted set
    (5445) and the pool is restored by `UndoValueBalance` per tx → `rolled_back_pool` →
    `WritePoolBalance(projected_pool)` (5374, 5513). This is CORRECT and NECESSARY: an
    orphaned-fork spend is no longer in the active chain, so the note must become spendable
    again — exactly once — on the new active chain.
  - The note can then be spent once on the main chain; ConnectBlock re-`Insert`s the nf (6578)
    after `IsShieldedNullifierSpent` passes. There is NO state where the same nullifier is
    active twice on the *active* chain ⇒ NO inflation. (A "double-spend" across an
    orphaned fork + main chain is not a double-spend: the fork is not in anyone's UTXO/pool
    state.) Disconnect/connect of value_balance is symmetric (`UndoValueBalance` ↔
    `ApplyValueBalance`), both pool≥0-guarded ⇒ no pool drift across the reorg.
- **Two DIFFERENT-looking nullifiers for one note:** REJECTED at impl level (crypto-bound).
  `CShieldedSpendAuthCheck(tx, spend_index, proof_bound_nullifiers[i])` requires the
  proof-bound nullifier == payload nullifier (`validation.cpp:6188-6192` block /
  3703-3708 mempool); `proof_bound_nullifiers` is extracted from the proof
  (`ExtractShieldedProofBoundNullifiers`). The serial→nullifier is a deterministic
  canonicalized (mod_q) hash of the note-derived key (e3). Producing a second valid nullifier
  for one note ⇒ a second valid lattice opening ⇒ the SMILE2 one-of-many soundness (x1 / f3),
  not a plumbing gap.

### ATTACK 2 — Value conservation / inflation on the z→t boundary
- **transparent_out + fee > value of spent note(s):** REJECTED. The proof's h2 balance forces
  `value_balance = Σ in − Σ shielded out`; `CheckTxInputs` then requires
  `value_balance ≥ value_out` (transparent out), and fee = value_balance − value_out ≥ 0.
  An over-claim breaks the SMILE2 balance (proof refuses / fails verify) OR, if value_balance
  is honestly bounded, the transparent fold rejects `bad-txns-in-belowout`.
- **mismatched / oversized value_balance:** REJECTED. `MoneyRangeSigned(value_balance)`
  structurally (`v2_bundle.cpp:720`) AND in `CheckTxInputs` (`tx_verify.cpp:206`,
  `bad-shielded-valuebalance-outofrange`). There is no second value_balance to mismatch
  against — proof, fold, and turnstile all read the one `payload.value_balance`.
- **negative fee / negative value_balance:** REJECTED structurally — `fee < 0 → reject`
  (`v2_bundle.cpp:721`); with spends `value_balance < fee → reject` (729) so value_balance ≥ 0.
- **MAX_MONEY overflow on the transparent side:** REJECTED. `CheckedAdd(nValueIn,
  value_balance)` + `MoneyRange` (`tx_verify.cpp:210-213`, `bad-shielded-adjusted-input-
  outofrange`); `tx.GetValueOut()` throws on per-output/sum overflow (classic, d1-confirmed);
  fee `MoneyRange` (223).
- **turnstile pool underflow (unshield more than was ever shielded):** REJECTED. The running
  pool is `Σ(shields − unshields)`; `ApplyValueBalance` rejects any unshield driving the pool
  < 0 (`turnstile.cpp:19`), at BOTH mempool (`validation.cpp:3302-3303`) and ConnectBlock
  (`6046-6047`). This is the hard ceiling: even a (hypothetically) forged over-unshield can
  never extract more than the legitimately-shielded total in aggregate — a forged single proof
  is bounded by the pool. INT64_MIN-negation UB is guarded by the preceding `MoneyRangeSigned`.
- ⇒ `CheckTxInputs` value_balance fold + SMILE2 h2 balance + turnstile pool≥0 catch ALL of
  these. The plumbing is airtight; the only residual is forging the proof (x1).

### ATTACK 3 — Replay / malleability
- **Lift an unshield's proof/nullifier into a different tx:** REJECTED. The proof statement =
  `ComputeV2SendStatementDigest(tx)` over `TX_WITH_WITNESS` of the whole stripped tx
  (`v2_proof.cpp:1140-1189`); the verifier RECOMPUTES it from the actual tx
  (`DescribeV2SendStatement` → `ParseV2SendProof` → `ProofEnvelopesMatch`, `v2_proof.cpp:1868`,
  reject `bad-shielded-proof`). A different tx ⇒ different digest ⇒ envelope mismatch.
- **Malleate the transparent outputs while keeping the proof valid (STEAL THE PAYOUT):**
  REJECTED — this is the marquee theft attempt and it fails by construction. The transparent
  `vout` (payout address + amount) is INSIDE the statement-digest preimage
  (`TX_WITH_WITNESS`). Changing the payout address/amount changes the recomputed digest ⇒
  `ProofEnvelopesMatch` fails ⇒ `bad-shielded-proof`. You cannot redirect an unshield's
  transparent payout to your own address while keeping the proof valid.
- **Replay an unshield across a reorg:** the nullifier is removed on disconnect (Attack 1) so
  the SAME tx can re-confirm on the new chain (this is correct re-org behavior, not a
  double-spend — the note is consumed once on the active chain). Cross-CHAIN replay is blocked
  by the chain-bound statement variant folding `hashGenesisBlock` + fork height
  (`v2_proof.cpp:1182-1186`).

### ATTACK 4 — Mempool / policy edge (does re-enabling expose relay of an invalid unshield?)
- REJECTED / NO new surface. The wallet gate is purely a CLIENT build-side `return fail(...)`
  (`shielded_wallet.cpp:2988-2991`); it is **never** consulted by mempool/relay/consensus.
  `MemPoolAccept` runs the FULL `CShieldedProofCheck` (`validation.cpp:3722-3753`, same
  verifier as ConnectBlock) PLUS the spend-auth nullifier-binding (3703-3708), turnstile
  pool≥0 (3302), nullifier double-spend vs chain (3328) and vs mempool/package (3178, 3197).
  `NeedsShieldedProofCheck` returns true for `V2_SEND` explicitly AND for any
  `HasShieldedInputs()` (`validation.cpp:1228-1237`) ⇒ an unshield ALWAYS gets its proof
  verified at relay — there is no "accept now, validate at block" window. An invalid unshield
  is rejected with `TX_CONSENSUS` before it can be relayed. Re-enabling the wallet path
  exposes nothing the node didn't already enforce on a hand-crafted raw tx.
