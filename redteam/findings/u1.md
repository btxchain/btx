# u1 — BTX build ergonomics for Docker / Alpine / Linux

Scope: 0.30.3 UX (non-consensus) build-quality items for the soft-fork PR. No
consensus code touched. Developed in an isolated `/tmp/rtU1-src` checkout of
`pr-214`; verified empirically by building in Docker (Alpine 3.22 / GCC 14.2.0,
musl) in an `rtU1-` namespace. The production node and all `pf3-*/bft-*/evx*/
btxn*/rt*`-not-mine containers were left untouched.

## Root-cause detail (fix #1)
The GCC 14.2 defect fires ONLY for block-scope automatic `const uint256 x{"…"}`
direct-init via the consteval `std::string_view` ctor. `static const`,
temporaries (`field = uint256{"…"}`), and `constexpr` locals are unaffected —
which is exactly why `chainparams.cpp`/`init.cpp` compiled in the daemon build
but `pow_tests.cpp` (`const uint256 digest{…}`, line 2766) did not. The
`std::string_view` indirection (rbegin/iterators over a view) is what confuses
GCC's "current evaluation" tracking; binding a `const char(&)[N]` literal
directly evaluates cleanly. Empirically minimised and confirmed against the real
patched header (patched: compiles; original: fails identically).

+## Fully-static, portable binary
+
+`Dockerfile.static` builds `btxd` + `btx-cli` with **no runtime shared-library
+dependencies**, so the binaries can be copied to any x86_64 Linux host. All the
+static-linking plumbing is behind the single `-DBUILD_STATIC=ON` CMake option
+(no manual `.so` removal or linker-flag fiddling required):
+
+```bash
+docker build -f contrib/docker/Dockerfile.static -t btxd:static --load .
+
+# Extract the static btxd:
+id=$(docker create btxd:static)
+docker cp "$id":/usr/local/bin/btxd ./btxd
+docker rm "$id"
+file ./btxd   # => ELF 64-bit ... statically linked
+```
+
+You can also pass `-DBUILD_STATIC=ON` to a plain CMake invocation, provided the
+static dependency archives are installed (on Alpine:
+`boost-static libevent-static sqlite-static zlib-static`):
+
+```bash
+cmake -B build -DBUILD_STATIC=ON && cmake --build build
+```
