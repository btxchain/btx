# c2 — P2P / networking memory-safety & remote-DoS review (MatMul / shielded / PQ)

Scope: deep code review of the BTX-specific P2P deserialization & verification surfaces
(MatMul block/Freivalds payload, shielded-tx bundle relay, custom `shieldedtx` /
`getshlddata` / `shieldeddata` messages, PQ sig verify), plus empirical malformed-message
fuzzing of a live regtest node (image `btx-node:pr214`, namespace `rtC2-`, /tmp/rtC2).
Goal = remote crash / OOM / hang / memory-safety/RCE primitive. Builds on a3 (relay guards)
and b2 (matmul verification-asymmetry/OOB) — did NOT repeat those; focused on the
deserializers and PQ/shielded paths b2 flagged as "the most likely place for buffer overflows."

## Vectors attempted
- **MatMul block payload OOB (matrix_a/b/c_data → Matrix)** — code review of
  pow.cpp CheckMatMulProofOfWork_{Phase2WithPayload,Freivalds,ProductCommitted} +
  primitives/block.h serialization.
  RESULT: ROBUST. `matmul_dim` is forced `== nMatMulDimension` EXACTLY in
  CheckMatMulProofOfWork_Phase1 (pow.cpp:2015) *before* any matrix alloc; payload size is
  checked `== n*n` (IsMatMulV2/FreivaldsPayloadSizeValid) before any indexing; per-element
  index is `row*n+col < n*n` by construction; `n*n` overflow guarded (uint16 dim, /n check,
  MATMUL_V2_MAX_PAYLOAD_WORDS cap). Freivalds Verify cross-checks A.cols==B.rows etc. and
  bails (no OOB) on mismatch. Matches b2 REFUTED. (code-only — can't drive valid-PoW blocks
  over P2P without mining; payload paths gated behind header PoW checks.)
- **Shielded v2 TransactionBundle deep deserialize (peer `shieldedtx`/`tx`+flag 2)** —
  reviewed bundle.h, v2_bundle.{h,cpp}, v2_types.h, account_registry_proof.h, v2_proof.cpp;
  EMPIRICAL: truncated v2-tag bundle, oversized legacy input/proof counts, view-grant
  oversize, compactsize bombs, 60x reconnect flood.
  RESULT: ROBUST. Every count goes through detail::UnserializeBoundedCompactSize /
  UnserializeBytes (cap check BEFORE resize/assign), or an explicit `> MAX_*` throw. The
  optional v2_bundle tag (=17) dispatch is bounded recursively the same way. proof-shard
  offset/size sliced only after `offset+size (uint64) <= proof_payload.size()` check
  (v2_proof.cpp:881-885, IsValid). Empirically: malformed bundles reach
  `TransactionBundle::Unserialize`, throw `ios_base::failure`, node catches → Misbehaving →
  disconnect/discourage, stays healthy. RSS flat 157→163 MB across all cases + flood.
- **Custom P2P messages `getshlddata` / `shieldeddata`** — net_processing.cpp:4922-5051.
  RESULT: ROBUST. getshlddata: NODE_SHIELDED-gated, null-hash rejected, per-peer request
  token bucket, depth-limited to tip-MAX_BLOCKTXN_DEPTH, response capped by
  MAX_SHIELDEDDATA_BUNDLES_PER_MSG(4096) + MAX_PROTOCOL_MESSAGE_LENGTH + relay token bucket +
  bounded FIFO cache (8 entries / 16MB). shieldeddata: unconditionally rejected pre-parse
  (no outstanding request), before any budget spend. Empirically no effect.
- **PQ (MLDSA44/Dilithium) verify from wire** — pqkey.cpp, libbitcoinpqc/bitcoinpqc.c,
  shielded address-lifecycle controls (v2_bundle.cpp:534-657), bridge auth (bridge.cpp:3511).
  RESULT: ROBUST. Every wire-reachable verify goes through `CPQPubKey::Verify`, which checks
  `m_data.size()==GetPQPubKeySize && sig.size()==GetPQSignatureSize` (pqkey.cpp:276) before
  calling the C lib, which re-checks both sizes (bitcoinpqc.c:294,302). Lifecycle-control
  pubkey/sig sizes are pinned to MLDSA44_{PUBKEY,SIGNATURE}_SIZE and the wire vectors are
  capped to those exact sizes on deserialize. No attacker-length memcpy into a fixed buffer
  found.
- **Single-tx allocation amplification (high-cap assign)** — MAX_BATCH_NULLIFIERS/LEAVES/
  EGRESS_OUTPUTS = 20000; MAX_PROOF_PAYLOAD_BYTES = 6MB; nMaxShieldedTxSize = 6.5MB.
  RESULT: bounded. `assign(count,{})` of ~150-byte default objects needs ≥65 wire bytes/elem,
  so a 6.5MB-capped tx yields ≤~3MB transient alloc (≈2-3x), transport-bounded. No remote OOM.
- **MatMul Phase1 genesis short-circuit** — pow.cpp:2010 `if (hashPrevBlock.IsNull()) return
  GetHash()==hashGenesisBlock;` before dim checks.
  RESULT: not a memory issue (header-hash compare only; no matrix code reached).

## Empirical environment
- Node: `docker run btx-node:pr214` (regtest, magic fabfb5da, port 19444), config mounted at
  /etc/btx/btx.conf, datadir /tmp/rtC2/data. Custom raw-socket Python P2P client
  (/tmp/rtC2/fuzz.py) — full version(NODE_SHIELDED)/verack handshake then malformed payloads.
- Mined 20 regtest blocks to exit IBD (tx path is `return`-skipped during IBD,
  net_processing.cpp:5397) so the deep shielded deserializer is actually exercised.
- Observed: handshake OK; all malformed messages → clean throw + peer ban, no crash/hang;
  node `healthy`, `getblockcount` responsive, VmRSS flat (~157–163 MB) before/after.
- Cleaned up: rtC2-node + rtC2-net removed. No prod/foreign assets touched.

## NEW / unmitigated findings
- NONE of severity crit/high. No remote crash, reachable assert, OOM, hang, or memory-safety
  primitive found on the reviewed P2P surfaces. The BTX-specific deserializers are
  consistently hardened (bounded-compactsize-before-alloc, exact-size pins, offset+size
  uint64 bounds checks) and backed by adversarial test corpora
  (test/fuzz/shielded_*, shielded_v2_adversarial_proof_corpus).

## Low / defense-in-depth observations (optional, not vulnerabilities)
- `ShieldedBlockData::Unserialize` (net_processing.cpp:240) does
  `bundles.reserve(bundle_count)` with bundle_count ≤ 4096 *before* reading bundles. Harmless
  (4096 empty CShieldedBundle ≈ a few hundred KB) and this path is only reachable as a
  response to a locally-issued request, which the node never issues → currently dead inbound
  (rejected pre-parse at net_processing.cpp:5050). No action needed; note if/when the node
  starts issuing getshlddata, keep the reject-unsolicited guard.
- `CShieldedInput::Unserialize` (bundle.h:170) `ring_positions.assign(count,0)` after a
  `MAX_RING_SIZE` cap — confirm MAX_RING_SIZE is small (it is; lattice param). Fine.
- Defensive consistency nit (matches b2): payload vectors (matrix_a/b/c_data) are hex/byte
  decoded up to the ~16MB transport cap before the cheap Phase1 dim check; a
  deserialization-side length bound to nMatMulMaxDimension^2 words would remove that
  transient. Not a vuln (transport-bounded, no amplification).

## Refuted / robust (so others don't repeat)
- Shielded-bundle truncation/oversize/type-confusion via `shieldedtx`/`tx` — REFUTED
  empirically: bounded deserialize → clean ios_base::failure → Misbehaving+disconnect, node
  healthy, RSS flat. (This is the direct memory-safety target b2 handed off; it holds.)
- `getshlddata`/`shieldeddata` custom messages — REFUTED: rate-limited, depth/size-capped,
  unsolicited-rejected-pre-parse. No remote memory growth.
- PQ MLDSA44 verify length-confusion → OOB — REFUTED: dual size-gate (C++ wrapper + C lib)
  on every wire path; no attacker-length buffer op.
- MatMul payload OOB / huge-alloc — REFUTED (re-confirms b2): dim pinned == nMatMulDimension
  pre-alloc, payload size == n*n pre-index.

## Open questions / leads for next wave
- Empirical MatMul-payload fuzzing requires a peer that can deliver a *header-PoW-valid* block
  carrying a malformed C'/A'/B' payload (the payload checks live behind Phase1 header PoW).
  Driving that needs an in-process/modified miner that mines a valid header then mutates the
  trailing payload vectors — worth a focused harness to empirically confirm b2's code-only
  OOB-robustness on the matrix indexing (I confirmed it by review; not yet by execution).
- Compact-block (BIP152) reconstruction with shielded bundles + the shielded-data cache
  interaction under reorg/many-tx blocks (cache eviction accounting m_shielded_data_cache_
  total_bytes) — not stressed here; bounded by 16MB cap but the byte-accounting on
  overlapping store/evict could be audited for a slow leak (low priority; would be DoS-soft
  not memory-safety).
- ASan/UBSan build deferred (shared 32-core budget); the deserializer hardening is uniform
  enough by review + black-box fuzz that a sanitizer run is lower marginal value than the
  payload-mutating miner harness above.
