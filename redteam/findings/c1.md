# c1 — Chain-halt / wedge / poison-block hunt (evil-miner, network-kill focus)

Scope: poison/crashing blocks (network-kill), a5 boundary-halt confirmation, difficulty lock-out,
validation/relay deadlock. Empirical on a self-built **correctly-rule-activated** regtest image
(`btx-node:rtC1-ruleon`, cap at h200 drift3600; `future_mtp_limit_height:200` verified via getmininginfo),
run with `-test=matmuldgw -test=bip94` (ASERT active @h2, HL=14400, spacing 90; BIP94 timewarp on).
Code reading on `pr-214`. Sanitizer build was NOT feasible (host has gcc only, no cmake/clang;
the project's `-DSANITIZERS=` path needs the in-container toolchain) — UB findings below are code-reasoned
and cross-checked against b1/b2; the runtime findings are black-box regtest repros.

## Vectors attempted

### 1. Poison / crashing block (MatMul validation paths beyond b2's dim check)
- **Transcript/ProductCommitted `throw std::runtime_error` reachability** (transcript.cpp:
  CanonicalMatMul, ComputeProductCommittedDigest, DeriveCompressionVector, ComputeProductCommittedDigestFromWords).
  These throw on bad dims (b==0, dim%b!=0, non-square). They run inside ContextualCheckBlock →
  Phase2/Freivalds/ProductCommitted with **no surrounding try/catch** (validation.cpp:8539-8591;
  an escaping exception from ContextualCheckBlock would propagate up ConnectBlock/AcceptBlock, not a clean reject).
  RESULT: NOT REACHABLE. Every throw is dominated by `CheckMatMulProofOfWork_Phase1` which pins
  `block.matmul_dim == params.nMatMulDimension` (consensus-global, 64) and `dim % nMatMulTranscriptBlockSize == 0`
  BEFORE any transcript call; b (=nMatMulTranscriptBlockSize) and noise_rank are consensus-global, not per-block.
  So attacker cannot drive b/dim into a throwing combination. Confirms+extends b2 (dim gate) for the throw paths.
- **C' payload OOB read** in CheckMatMulProofOfWork_Freivalds (pow.cpp:2204-2210) and _ProductCommitted
  (2286-2292): loop indexes `block.matrix_c_data[idx]`, idx∈[0,n*n). RESULT: BOUNDED — `IsMatMulFreivaldsPayloadSizeValid`
  requires `matrix_c_data.size()==n*n` with n pinned to consensus dim and `expected_words<=MATMUL_V2_MAX_PAYLOAD_WORDS`,
  and per-element `>= MODULUS` canonicality reject. matrix_a/b same shape-gated. No OOB, no OOM (16MB transport cap).
- **Matrix/field asserts** (matrix.cpp at()/block()/operator*, ConstMatrixView::at): all `assert()` (compiled
  out under NDEBUG). RESULT: not adversary-reachable — n is pinned; noise/transcript dims are derived from the
  pinned n and consensus b/rank, so no attacker-controlled index reaches them. `field::inv`'s `assert(a!=0)` is
  NOT on the consensus verify path (Verify/Freivalds/transcript use only add/sub/mul/dot). `from_oracle` is
  deterministic, bounded (256 retries + deterministic fallback), no UB. ReduceMersenne128 has a defensive 8-iter bound.
  RESULT: NO consensus-reachable abort/UB found in the MatMul validation paths.
- **Difficulty math (CalculateMatMulAsertTarget)**: __int128 with explicit int64 saturation of the exponent,
  net_shift handled for [<=-256, <0, >=256, >0] with SaturatingMultiply/Shift, ClampRetargetResult, guards
  half_life>0 & spacing>0. RESULT: no overflow/div-zero/assert (re-confirms b4). No poison via nBits.

### 2. a5 activation-boundary chain-halt — CONFIRMED reachable at boundary, unreachable post-activation
- Constraints on block B (parent P), all in ContextualCheckBlockHeader (validation.cpp:8437-8453):
  (i) time-too-old: B.time > P.MTP; (ii) BIP94 (always-on for MatMul): B.time >= P.blocktime - 600;
  (iii) PR cap (h>=200): B.time <= P.MTP + 3600; (iv) time-too-new: B.time <= clock + 7200.
  Inversion: lower bound (ii) > upper bound (iii)  ⟺  P.blocktime - 600 > P.MTP + 3600  ⟺  P.blocktime > P.MTP + 4200.
  Then NO timestamp is legal for B ⇒ next block unmineable by anyone ⇒ PERMANENT halt.
- Empirical, post-activation (capped blocks): mined past h200 under several clock regimes; every capped block
  clamps B.time to P.MTP+3600 (getblocktemplate `maxtime=prev.MTP+3600`, `curtime_clamped:true`); chain advanced
  h200→h205+ with no stall. Because capped B.time <= P.MTP+3600 and MTP is non-decreasing, a capped parent always
  has P.blocktime <= P.MTP+3600 < P.MTP+4200 ⇒ inversion CANNOT arise once the cap is active. CONFIRMS a5.
- Boundary reachability (code-confirmed; the misbehaving-miner step can't be produced by the honest wallet miner —
  see the asymmetry below): block 118481/h199 is the LAST *uncapped* block; consensus lets it be stamped up to
  clock+7200. A miner that stamps it to ~P.MTP+7200 makes P.blocktime > P.MTP+4200 ⇒ block 118482/h200 unmineable.
  Mainnet passed cleanly only because the honest reference miner self-clamped 118481 (UpdateTime).
- **UpdateTime does NOT save you (miner-clamp trace, miner.cpp:491-504):** when wall-clock would exceed the cap,
  it sets `nNewTime = std::max(min_time, *max_time)`. In the inversion `min_time > max_time`, so it emits
  `min_time`, which itself **violates** the consensus cap (B.time > P.MTP+3600) ⇒ the honest miner produces an
  unmineable/invalid block. So the clamp converts the empty window into a guaranteed self-reject, not a recovery.
- **MTP monotonicity (verified):** GetMedianTimePast = median of last 11 block times (chain.h:304); combined with
  the time-too-old rule (B.time > P.MTP) this makes MTP non-decreasing — observed empirically throughout (MTP rose
  monotonically across 870+ blocks). This is exactly why the post-activation inversion is unreachable, and it means
  there is no OTHER timestamp min>max deadlock: the four header-time constraints are the complete set and the only
  inversion is timewarp-floor ⊥ drift-cap (= a5).
- **a5 fix — verification status:** the inversion mechanism and post-activation safety are confirmed (code +
  empirical). The fix (upper bound = max(prev_MTP+drift, prev_blocktime−MAX_TIMEWARP), applied identically in the
  consensus check at validation.cpp:8446 and in GetMaximumTime/MatMulFutureBlockTimeLimit so UpdateTime can't emit
  an out-of-window block) is code-reasoned-correct: making upper≥lower removes the empty window by construction.
  NOTE the signatures must change to also take prev_blocktime (today MaxMatMulFutureBlockTime/MatMulFutureBlockTimeLimit
  take only prev_median_time_past). I did NOT build a fixed-variant image (would need a signature change across
  validation+miner + another full container build; marginal added confidence over the by-construction argument).

### 3. Difficulty lock-out (spike + recovery)
- ASERT (MatMulAsert) is stateless, anchored to a fixed anchor (h2 here / h50000 mainnet), NO per-block slew clamp
  active: `ApplyDgwSlewGuard` is gated on `nDgwSlewGuardHeight` which is **INT32_MAX (DISABLED) on every network
  including mainnet** (chainparams.cpp:185). So the only per-block difficulty bound is the ASERT half-life math.
- Min-gap attack empirically reproduced: forced every block to B.time = P.MTP+1 (set mocktime=MTP+1 so
  getblocktemplate curtime clamps to mintime). Once cumulative actual_elapsed fell below expected
  (height_diff*90), target began shrinking (difficulty rising):
  bits 2100ffff → 2100c252 (~h413) → 21009c8a → 207e1d01 (~h513) → 2022779d (~h813); see Spike/Recovery below.
- Per-block hardening ≈ 2^(90/HL) = 2^(1/160) ≈ ×1.0043/block at HL=14400; at mainnet HL=3600 it is
  2^(1/40) ≈ ×1.0175/block (×2 per ~40 min-gap blocks) — matches b4. Difficulty is driveable arbitrarily high
  with enough attacker blocks, but the attacker pays full PoW at the rising difficulty (self-funded).
- **Spike measured (HL=14400 regtest):** ~760 min-gap blocks (h53→h813) drove target 2100ffff → 2022779d,
  ≈ **×7.4 difficulty** (deficit D = expected−actual = 41,659 s; M = 2^(D/HL) = 2^(41659/14400) ≈ 7.4 ✓ formula).
  At mainnet HL=3600 the same deficit ⇒ M = 2^(41659/3600) ≈ ×3,000; deficit grows ≤90 s/attacker-block,
  so M is unbounded in attacker block-count (k min-gap blocks ⇒ up to ×2^(90k/HL)).
- **Recovery is SELF-CORRECTING, not a permanent freeze (measured).** ASERT difficulty depends on the deficit
  D=expected−actual. Honest mining at exactly target spacing keeps D constant (difficulty stays spiked), but at
  spiked difficulty ×M honest blocks take ~M× longer in REAL time, so each honest block's real gap ≫ 90 s and
  D drains. Empirically: resuming honest (slow) mining drained D=41,659→~0 and difficulty returned to baseline
  (2022779d → 21008403/2100xxxx) in ~50 honest blocks; chain stayed LIVE; a normal block was accepted afterward
  (h874) — no wedge. The transient-freeze *duration* ≈ the injected deficit in real seconds (~11.6 h here),
  independent of hashrate, and it heals automatically. A sustained MAJORITY attacker can extend the deficit
  (hence the freeze) arbitrarily, but pays full self-funded PoW and produces a valid chain the network follows.
  SEVERITY: MED (transient, self-healing, majority-only, self-funded) — consistent with a1/b4. Optional mitigation:
  re-enable the per-block slew guard on the ASERT path (`ApplyDgwSlewGuard` is wired but DISABLED via
  nDgwSlewGuardHeight=INT32_MAX on all nets) to cap per-block difficulty change to ×4 and blunt the spike rate.
- **Mining-policy clamp is NOT height-gated (NEW, see findings):** `GetMaximumTime`/`MatMulFutureBlockTimeLimit`
  clamp mining to P.MTP+drift whenever the drift is configured, with **no height check**, even below the
  consensus activation height (`future_mtp_consensus_active:false` while `future_mtp_mining_policy_active:true`
  at h<200). Observed directly: at h51 getblocktemplate `maxtime=prev.MTP+3600`.

## NEW / unmitigated findings
- **[HIGH, latent] a5 activation-boundary permanent halt — CONFIRMED (mechanism empirically + code).** repro:
  on a chain where the cap activates at height>0, let the last pre-activation block be stamped > prevMTP+4200
  (consensus-legal there, up to clock+7200); the activation block then has an EMPTY legal-time window
  (BIP94 floor prevblk-600 > cap prevMTP+3600) and is unmineable by anyone ⇒ permanent halt. Dormant on
  mainnet now (passed 118482), but re-armed on any network that activates the cap at height>0 (b4-1) and on
  any future re-activation. mitigation: consensus upper bound = `max(prev_MTP+drift, prev_blocktime - MAX_TIMEWARP)`
  AND make GetMaximumTime/UpdateTime use the same max() so the window is never empty and the honest miner never
  self-rejects; OR activate the cap from genesis on new networks. (Confirms a5; adds the UpdateTime self-reject detail.)
- **[LOW-MED, NEW] Mining-policy future-cap clamp is applied below the consensus activation height.**
  `Consensus::Params::MatMulFutureBlockTimeLimit` (used by GetMaximumTime) gates only on
  `nMatMulMaxFutureMtpDriftHeight != INT32_MAX`, not on `height >= that height` (contrast the consensus path
  `MaxMatMulFutureBlockTime`/`IsMatMulMaxFutureMtpDriftActive`, which IS height-gated). Effect: honest miners
  clamp block times to prevMTP+drift even for blocks BELOW activation, where consensus does not enforce it.
  Mostly benign (and it actually *protected* the mainnet activation block by keeping honest 118481 under the cap),
  but it is an inconsistency: it changes pre-activation honest timestamping and, if the proposed a5 fix changes
  GetMaximumTime, this height-gating must be made consistent with the consensus rule. mitigation: height-gate the
  mining clamp to match the consensus rule (or, with the a5 fix, fold both into one shared helper).

## Refuted / robust (so others don't repeat)
- MatMul Phase2/Freivalds/ProductCommitted/transcript validation: NO consensus-reachable assert/abort/throw/UB/OOB/OOM.
  All throwing/asserting paths are dominated by the Phase1 dim/b/rank pin and the payload size+canonicality gates.
  (extends b2's dim-check result to the throw/ProductCommitted/Freivalds paths explicitly).
- ASERT difficulty math: no overflow/assert/div-zero on extreme inputs (re-confirms b4).
- Post-activation a5 inversion: empirically unreachable (capped block.time <= prevMTP+3600 < +4200; MTP monotone).
