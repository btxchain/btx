# s2 — Supply-chain integrity of VENDORED in-tree libraries (MODIFIED-from-upstream audit)

Scope: for every in-tree vendored subtree forked from an upstream — **secp256k1, minisketch, univalue,
leveldb, crc32c, crypto/ (ctaes/sha/chacha/…), ml-kem-768, ethash/kawpow** — DIFF against the exact
upstream to answer the key question: **has it been MODIFIED, especially the crypto core?**
Read-only on repo; clones + diffs under /tmp/rtS2/. No production assets touched. No fabricated results.

Builds on c3 (which inventoried versions but did NOT diff against upstream). This finding closes that gap.

## METHOD
- BTX fork base = **Bitcoin Knots `v29.2.knots20251110`** (per d1). Cloned it (`bitcoinknots/bitcoin`)
  and diffed the vendored subtrees against it — this isolates *true BTX edits* from Knots-vs-Core noise.
- Also diffed secp256k1 against its canonical pinned release **`bitcoin-core/secp256k1 v0.6.0`**, minisketch
  against `bitcoin-core/minisketch` master (pinned the exact base commit), ml-kem-768 against **PQClean
  `crypto_kem/ml-kem-768/clean`**, univalue against Bitcoin Core v29 `src/univalue`.
- All clones verified byte-for-byte with `diff -rq` / per-file `diff -q`.

## Vectors attempted
- **secp256k1 (CROWN JEWEL) modified?**: full recursive diff of src/ + include/ vs upstream v0.6.0 AND vs
  Knots fork base.
  RESULT: **UNMODIFIED** — crypto core byte-identical. field_impl, field_5x52/10x26, scalar_4x64/8x32,
  ecmult/ecmult_gen/ecmult_const, ecdsa_impl, group_impl, secp256k1.c, **precomputed_ecmult(_gen).c
  (the generator tables)**, testrand, util, schnorrsig/recovery/ellswift main_impl, include/* — ALL IDENTICAL.
  Only diff anywhere = **5 comment lines `TODO`→`TO-DO`** (a Knots-wide lint substitution, present in the
  fork base too — NOT a BTX change). **0 non-cosmetic changed lines** vs Knots. No context/RNG/ecmult tamper.
- **crypto/ standard primitives modified?**: diff entire src/crypto vs Knots.
  RESULT: ChaCha20, ChaCha20Poly1305, Poly1305, AES, **ctaes**, HMAC-SHA256/512, RIPEMD160, SHA512, SHA1,
  SHA3/Keccak, **SipHash**, common.h — ALL BYTE-IDENTICAL. Only sha256.* and hkdf_sha256_32.* differ (below).
- **SHA256 (consensus hash) modified?**: diff vs Knots.
  RESULT: **NO algorithmic change.** Adds (1) a `~CSHA256()` destructor that `memory_cleanse()`s internal
  state s/buf/bytes; (2) `ENABLE_SSE41` + `_M_X64/_M_IX86` preprocessor guards around x86 SIMD backend
  selection (MSVC/portability, makes SSE41 opt-in); (3) drops an unused `#include <sys/auxv.h>`. The
  `Transform()` compression fn and all SIMD backends are untouched ⇒ **SHA256 output unchanged.** Same
  memory_cleanse-destructor addition in hkdf_sha256_32.* (scrubs the PRK). Defense-in-depth, security-positive.
- **minisketch modified?**: diff src/+include vs upstream master; pinned exact base commit.
  RESULT: **UNMODIFIED.** Base commit = **`d1e6bb8`** (`Merge #92: doc rename sipa→bitcoin-core`); in-tree
  src+include matches it exactly and matches Knots byte-for-byte. The only deltas vs *current* master are
  upstream's LATER changes BTX hasn't pulled (staleness, see below) — not BTX edits.
- **univalue modified?**: diff vs Bitcoin Core v29 src/univalue (the real upstream; the standalone
  bitcoin-core/univalue repo is archived/diverged).
  RESULT: **One additive method** — BTX added `UniValue::reserve(size_t)` (wraps `values.reserve()` /
  `keys.reserve()`), plus its header declaration. The JSON parser state machine (univalue_read.cpp),
  getters, and writer are otherwise IDENTICAL (0 logic changes after excluding the added method). Benign
  perf helper for large matmul/shielded payloads.
- **leveldb (chainstate) modified?**: diff full tree vs Knots.
  RESULT: **Only db_impl.{cc,h} modified** — see finding S2-1. All of db/table/util/port/include/helpers
  otherwise identical. Data path (compaction merge, CRC, recovery) untouched.
- **crc32c modified?**: diff vs Knots.
  RESULT: **BYTE-IDENTICAL** (the `Process`→`Proccess` comment typo seen vs Core is in Knots itself).
- **ml-kem-768 (Kyber, shielded note encryption — CONSENSUS-reachable) modified?**: identified upstream as
  **PQClean `mlkem768/clean`** (symbol prefix `PQCLEAN_MLKEM768_CLEAN_`, K=3/N=256/Q=3329) and diffed.
  RESULT: cbd/indcpa/kem/ntt/polyvec/reduce/symmetric-shake **and fips202.c (vs PQClean common/fips202.c)**
  IDENTICAL. poly.c + verify.c differ **only by constant-time hardening**: a `PQCLEAN_PREVENT_BRANCH_HACK(b)`
  barrier (`__asm__("":"+r"(b):)`) inserted before the `cmov_int16` in `poly_frommsg` and in `verify`. Same
  logic/values (`(KYBER_Q+1)/2`, `(msg>>j)&1`) — a security-POSITIVE anti-branch optimizer barrier (matches
  PQClean's own later compat.h). Not a backdoor.
- **ethash/progpow (kawpow) modified/active?**: chfast/Pawel-Bylica `ETHASH_VERSION 0.5.1-alpha.1`
  (Apache-2.0), wrapped by BTX `kawpow.cpp` (Copyright 2026 BTX).
  RESULT: **DISABLED on every network** — `fKAWPOW=false`, `nKAWPOWHeight=INT_MAX` for mainnet/test/signet/
  regtest (chainparams.cpp). Legacy/era-compat dead consensus code. LOW priority.

## NEW / unmitigated findings
- **S2-1 — BTX patched vendored leveldb engine; broke compaction `bytes_written` stat | severity LOW |
  repro:** `diff knots/src/leveldb/db/db_impl.cc src/leveldb/db/db_impl.cc`. BTX **removed** the loop
  `for(...) stats.bytes_written += compact->outputs[i].file_size;` in `DoCompactionWork()` (so compaction
  outputs no longer count toward `bytes_written`) while **adding** a test hook
  `int64_t DBImpl::TEST_TotalCompactionBytesWritten()` (sums `stats_[level].bytes_written`) used by
  `db_test.cc:1065/1067`. Self-inconsistent: a getter was added for a value they stopped populating, so it
  under-reports. **Impact: cosmetic only** — `bytes_written` feeds nothing but a `Log()` line
  (db_impl.cc:1425) + the new test helper; it does NOT drive compaction triggering, file selection, CRC, or
  any data-correctness path. No consensus/integrity effect. **Mitigation:** restore the removed accumulation
  loop (one-liner) or drop the orphan stat; keep test instrumentation in test-only translation units rather
  than patching the vendored engine.
- **S2-2 — minisketch is STALE: missing upstream default-ctor segfault hardening | severity LOW |**
  base commit `d1e6bb8` predates upstream `5cc7d20` (2024-06-11) which changed
  `Minisketch() noexcept = default;` → `= delete` because "trying to use one of its methods will cause a
  segfault." BTX still ships the `= default` (segfault-prone) wrapper ctor. Relay/erlay-side (not consensus
  block validation), and only triggers on internal misuse of a default-constructed object. **Mitigation:**
  re-sync minisketch subtree to current `bitcoin-core/minisketch` master.
- **S2-3 — ml-kem-768 / ethash carry NO recorded upstream pin (provenance gap) | severity LOW |**
  unlike libbitcoinpqc (which records dilithium/sphincs commits in README), the in-tree `crypto/ml-kem-768/`
  and `crypto/ethash/` have **no SHA256SUMS / upstream-commit anchor**. I established them empirically
  (PQClean mlkem768/clean; chfast ethash v0.5.1-alpha.1) but a reviewer cannot pin them from the repo.
  ml-kem-768 is **consensus-reachable** (shielded note_encryption→bundle/v2_egress) so this matters.
  **Mitigation:** record upstream repo+commit+SHA256SUMS for ml-kem-768 and ethash (parity with dilithium).
  (Extends c3's LOW item #6.)

## Refuted / robust (so others don't repeat — HIGH-VALUE NEGATIVES)
- **★ secp256k1 is UNMODIFIED — no crypto-core backdoor.** The single highest-value question in this mission.
  Byte-identical to canonical `bitcoin-core/secp256k1 v0.6.0` and to the Knots fork base across the ENTIRE
  src/ + include/ (field/scalar/ecmult/ecdsa/group/context/RNG/**generator tables**). The only diff is 5
  cosmetic `TODO→TO-DO` comment lines that originate in Knots, not BTX. **0 non-cosmetic changed lines.**
- **No tampering of any standard Bitcoin crypto primitive.** ctaes/AES, ChaCha20(+Poly1305), Poly1305,
  HMAC-SHA256/512, RIPEMD160, SHA512, SHA1, SHA3, SipHash, common.h = byte-identical to Knots. SHA256/HKDF
  differ ONLY by added memory_cleanse destructors + build guards (no output change). crc32c byte-identical.
- **univalue / leveldb data paths unmodified** — only an additive `reserve()` (univalue) and a cosmetic
  stat regression + test hook (leveldb). JSON parser & DB compaction/CRC logic untouched.
- **ml-kem-768 (Kyber) edits are constant-time HARDENING, not a backdoor** — barrier macro only; FO-transform
  cmov values/logic identical to PQClean.
- No git submodules ⇒ no submodule-takeover (re-confirms c3). All vendored code is in-tree copies.

## Remediation (re-sync / record-pin / restore)
1. **[LOW] leveldb (S2-1):** restore the removed `bytes_written` accumulation loop in
   `DoCompactionWork`; move the `TEST_TotalCompactionBytesWritten` hook out of the vendored engine.
2. **[LOW] minisketch (S2-2):** re-sync subtree to current `bitcoin-core/minisketch` master (picks up the
   `=delete` default-ctor segfault hardening + whitespace cleanup).
3. **[LOW] provenance (S2-3):** record upstream repo+commit+SHA256SUMS for `crypto/ml-kem-768`
   (PQClean mlkem768/clean) and `crypto/ethash` (chfast v0.5.1-alpha.1) — ml-kem is consensus-reachable.
4. **[info] keep** the sha256/hkdf memory_cleanse destructors and the ml-kem branch-barrier (good hygiene).

## Staleness check (resolved)
- **secp256k1 v0.6.0 pin is NOT behind any security fix.** Upstream has since shipped v0.7.0 (2025-07-21) and
  v0.7.1 (2026-01-26); I reviewed both CHANGELOG entries — they contain **only** build/CMake/test-framework
  changes, API cleanups (removed deprecated `_privkey_*` aliases; `const` static-context pointers), and a
  best-effort "increase cases where secrets are cleared from stack" (defense-in-depth, not a vuln fix). **No
  correctness/security CVE.** Bumping is hygiene, not required. (ABI note: 0.7.0 removed deprecated symbols —
  re-sync would need those call sites updated, but BTX doesn't use the removed aliases.)
- **ml-kem-768 fully verified** vs PQClean mlkem768/clean: all core files + fips202.c byte-identical; only
  the two constant-time barrier additions (security-positive).
