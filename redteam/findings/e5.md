## 4. Fix validation — e1's `seed = SHA256(prev‖height‖nonce‖which)` WORKS, with caveats

**Does it defeat the amortization? YES.** Folding nNonce64 into the seeds makes A,B (hence A·B)
change every nonce ⇒ the dense product is nonce-dependent and cannot be precomputed/reused ⇒ honest
Θ(n³) per attempt restored. The low-rank trick only ever amortized the NOISE terms (already O(n²r));
killing the AB reuse is exactly the right lever. Validated in principle by the model: if A,B vary
per nonce there is no constant AB to factor out. ✅

**Honest cost impact — acceptable, and SMALLER than e1 implies, because of the eps pre-gate:**
- The mining grind applies the cheap pre-hash eps gate FIRST (`BuildMatMulNonceBatchWindow`
  pow.cpp:1074-1080: compute `DeriveSigma`, reject if > shifted target) before any matmul. On
  mainnet eps=18 ⇒ ~1 in 2^18 nonces reaches the expensive path.
- `DeriveSigma` already needs seed_a/seed_b (they're in `ComputeMatMulHeaderHash`, matmul_pow.cpp:239).
  Under the fix, every grind nonce computes 2 extra SHA256 (seed_a,seed_b) before the existing
  2-SHA256 sigma gate ⇒ per-grind-nonce overhead goes from ~2 to ~4 SHA256. Trivial.
- The EXPENSIVE part of the fix — rebuilding A,B from seed (2n² = 524,288 SHA256/`from_oracle` on
  mainnet, because `SharedFromSeed`'s cache `matrix.cpp:34-41` now always misses) — happens ONLY for
  the ~2^-18 of nonces that pass the eps gate, i.e. once per actual matmul, where it sits next to the
  134M-mult dense product it accompanies (same order as the existing 4·n·r noise-gen SHA, dwarfed by
  the matmul). So the fix's amortized honest overhead is negligible; it merely RESTORES the intended
  per-nonce O(n³). ✅ (e1 hand-waved "cheap O(n²) SHA"; the eps-gate ordering is what actually makes
  it cheap — worth stating in the PR.)

**NEW issues the fix introduces / must handle:**
1. **No circularity** with sigma/digest: seed = SHA256(prev‖height‖nonce‖which) consumes nonce (an
   independent header field) and is computed BEFORE `ComputeMatMulHeaderHash` consumes the seeds.
   One-directional. Fine. (Do NOT, however, derive the seed from `sigma` as e1's alt-wording #1
   suggested — that WOULD be circular since sigma depends on the seeds. Use nNonce64 directly, as in
   his primary wording.)
2. **Pre-hash gate interaction:** still sound — the eps gate keys off sigma, which now transitively
   depends on the per-nonce seeds; nothing breaks, and the gate keeps the A,B-rebuild cost off the
   hot path (above). No new grind surface (b1's Fiat-Shamir binding of the Freivalds r to C' is
   untouched).
3. **Base-matrix caching becomes dead weight:** `g_from_seed_cache` and the CUDA
   `cached_matrix_a_key`/uploaded-base cache will thrash (always miss). Not a correctness problem,
   but the PR should drop/repurpose those caches to avoid wasted memory + cache-management overhead.
4. **Seeds are explicit header fields** today (`block.seed_a/seed_b`, in GetHash) AND
   consensus-recomputed. The fix changes the recompute formula in BOTH `validation.cpp:8385` and the
   miner; they must agree exactly (height + nNonce64 both available at validation). Straightforward
   but it's a two-site change — a divergence would split the chain.
