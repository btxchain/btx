Scope: NEW bugs/edges introduced by PR #214 (`git diff main...pr-214`) in
consensus/params.h, node/miner.cpp (UpdateTime clamp), validation.cpp (new
"time-mtp-too-new" ordering + pre-hash gate relocation), rpc/mining.cpp

Environment: built a variant `btx-node:rtB6-mtprule` from pr-214 + a minimal
regtest activation arg (`-regtestmatmulmaxfuturemtpdriftheight`,
`-regtestmatmulmaxfuturemtpdrift`) because **the shipped PR provides NO way to

## Vectors attempted

- F0 / "ruleon image does not actually turn the rule on": started
  `btx-node:pr214-regtest-ruleon`, mined to height 201.
  RESULT: NO-EFFECT — `getmininginfo.time_policy.future_mtp_limit_height` stays
  `2147483647` (INT32_MAX) at every height; `future_mtp_consensus_active=false`,
  `future_mtp_mining_policy_active=false`. The image exposes none of the
  `-regtestmatmul*` args for the drift rule (`btxd -regtest -help-debug` lists
  binding/productdigest/aserthalflife/prehashepsilon only). PR's regtest
  chainparams hardcodes `nMatMulMaxFutureMtpDriftHeight = INT32_MAX`
  (asserted by the PR's own `matmul_params_regtest` test). Mainnet value is
  118482 (asserted by `matmul_params_defaults_mainnet`). ⇒ The consensus path
  is untestable on regtest as shipped; the "ruleon" image is mislabeled.

- Height-gating mismatch consensus-vs-mining (params.h / miner.cpp): code review
  + variant repro at next-heights 199/200/201, drift 3600.
  RESULT: CONFIRMED mismatch (conservative direction). `GetMaximumTime` →
  `MatMulFutureBlockTimeLimit` is NOT height-gated (only checks height!=INT32_MAX
  && drift>0), while consensus `MaxMatMulFutureBlockTime` IS height-gated via
  `IsMatMulMaxFutureMtpDriftActive`. Observed at next_height=199 (one below
  activation 200): `consensus_active=False` but `mining_active=True` with
  `maxtime` set; the template `curtime` was clamped to `prev_mtp+3600`
  (1296814602) even though `node_time` was `prev_mtp+6000` (1296817002) and
  consensus (inactive) would accept up to `node_time+7200`. See F1.

- UpdateTime BIP113 lower-bound violation / nTime ≤ prev MTP: code review.
  RESULT: REJECTED(safe). `nNewTime = max(min_time, …)` and the clamp uses
  `std::max(min_time, *max_time)`, so output is always ≥ `min_time ≥ prev_mtp+1`.
  Cannot emit nTime ≤ prev MTP.

- UpdateTime inversion (min_time > max_time) producing a self-rejecting template
  at activation: code review + bip94 stress repro (mine 30 blocks with clock
  jumping `prev_mtp+10000` each time, drift cap active at height 200).
  RESULT: REJECTED(not reachable in practice). Inversion needs
  `prev_blocktime - MAX_TIMEWARP(600) > prev_mtp + 3600`, i.e.
  `prev_blocktime > prev_mtp + 4200`. Because the mining clamp is ALWAYS on when
  the rule is configured, the chain can never build `prev_blocktime` far above
  MTP; all 30 future-clocked blocks mined fine, none exceeded the cap, no
  self-rejection. Latent code edge only (see F3).

- Mining clamp at activation produces self-rejecting / cap-violating template:
  variant repro, set mocktime `prev_mtp+100000`, mine across height 200.
  RESULT: REJECTED(safe). Template `curtime` clamped to exactly
  `maxtime=prev_mtp+3600`; the mined block landed exactly at the cap and was
  accepted (tip_time == maxtime; consensus check is `> cap` so boundary is
  inclusive-safe). No off-by-one at the cap.

- Activation off-by-one (199 vs 200 vs 201): variant repro.
  RESULT: clean. `IsMatMulMaxFutureMtpDriftActive(199)=False`,
  `(200)=True`, `(201)=True` (matches PR pow_tests for 118481/118482). Consensus
  flips exactly at the configured height; no off-by-one.

- params.h INT64 overflow guard (`prev_mtp > INT64_MAX-drift → INT64_MAX`): review.
  RESULT: REJECTED(correct). Guard is present and shared by both
  `MatMulFutureBlockTimeLimit` and `MaxMatMulFutureBlockTime` (latter delegates).
  With drift=3600 the guard only triggers near 9.2e18; unreachable for real
  timestamps. No overflow.

- rpc/mining.cpp crash via CHECK_NONFATAL(pindex_prev) with null / curtime
  overflow / RPC assertion: review + live probes (default + variant nodes).
  RESULT: REJECTED(robust). `MiningTimePolicyToJSON` is only reached with a
  non-null tip (getmininginfo: `*CHECK_NONFATAL(active_chain.Tip())`;
  getblocktemplate TemplateToJSON: non-null pindexPrev; getmatmulchallenge:
  header sourced from `block_template->getBlockHeader()` (tip), guarded by an
  earlier `CHECK_NONFATAL`). `CHECK_NONFATAL` throws a catchable RPC exception,
  not abort. `curtime_clamped` is pure int64 comparison, no arithmetic.
  getmatmulchallenge/getmininginfo/getblocktemplate all return cleanly with the
  rule ON; no crash/assert observed.

- submitblock / getblocktemplate-proposal / submitheader hex guards: live probes.
  RESULT: REJECTED(robust). Odd-length hex → RPC -22 "must be even-length hex";
  garbage → "Block decode failed"; oversized handled by `EnforceBlockHexSizeLimit`
  (MAX_BLOCK_SERIALIZED_SIZE*2 chars). `submitheader` on MatMul chain → -8
  "not supported … use submitblock". All clean, no crash.

- Pre-hash gate relocation into ContextualCheckBlockHeader (validation.cpp):
  review.
  RESULT: REJECTED(safe/consistent). `CheckMatMulPreHashGate` is cheap (two
  SHA256 via `DeriveSigma`, header-only fields) — no verification-asymmetry DoS.
  It returns true immediately when `pre_hash_epsilon_bits==0` (regtest). It is
  ALSO still run in the Phase2/Freivalds/ProductCommitted paths, so no block
  that should be valid is newly rejected and no invalid block is newly accepted;
  the change only moves the rejection earlier (header time) with reason
  "high-hash"/BLOCK_INVALID_HEADER. Net DoS-reduction, not a regression.

- "time-mtp-too-new" ordering vs BIP94 timewarp vs MAX_FUTURE_BLOCK_TIME: review.
  RESULT: REJECTED(safe). The three are independent reject predicates;
  ordering only affects which error string is returned, not validity. New check
  uses `pindexPrev->GetMedianTimePast()` — same basis as the miner's
  `GetMaximumTime`, so mining and consensus agree at/above activation. pindexPrev
  null/genesis handled by the existing `bad-prevblk-null` /
  `bad-blk-height-overflow` guards at the top of ContextualCheckBlockHeader.

## NEW / unmitigated findings

- F1 | Mining/submitSolution future-MTP clamp is active BELOW the consensus
  activation height (height-gating mismatch) | severity low (liveness/correctness,
  conservative direction) |
  repro: variant node, rule configured at height 200, `-mocktime`.
    next_height=199 → getmininginfo.time_policy:
      future_mtp_consensus_active=False, future_mtp_mining_policy_active=True,
      maxtime=prev_mtp+3600. With clock at prev_mtp+6000, gbt curtime is clamped
      to prev_mtp+3600 (curtime_clamped=True) though consensus (inactive) and
      MAX_FUTURE_BLOCK_TIME would accept up to node_time+7200.
    The same ungated `GetMaximumTime` gates node/interfaces.cpp `submitSolution`
    (line ~1045): a solved block with a consensus-VALID timestamp in
    `(prev_mtp+3600, clock+7200]` is rejected ("rejecting MatMul solution with
    timestamp above maximum") below the activation height. submitblock is NOT
    affected (it does not call GetMaximumTime).
  why it matters: on mainnet the rule is configured at 118482 but the clamp/reject
    is live at ALL prior heights. In a genuine pre-activation hashrate-stall (real
    clock legitimately > prev_mtp+3600), the standard mining path (getblocktemplate
    curtime + submitSolution) refuses to use/accept consensus-valid timestamps,
    while a peer block carrying such a timestamp is still accepted by consensus
    (and via submitblock). Impact is bounded and conservative (no invalid block is
    produced; worst case the local template lags real time and submitSolution
    rejects an otherwise-valid solution), so liveness-leaning, not a security hole.
  proposed mitigation: height-gate the mining/submitSolution path too — make
    `GetMaximumTime` (or its callers) consult `MaxMatMulFutureBlockTime(next_height,
    prev_mtp)` instead of the ungated `MatMulFutureBlockTimeLimit`, so mining
    policy and submitSolution match consensus activation exactly. (If the intent
    is "advise miners early," keep advising via time_policy but do not hard-clamp
    curtime or hard-reject in submitSolution below activation.)

- F3 | UpdateTime clamp can invert (min_time > max_time) and emit a block the
  node's own consensus rejects at/above activation | severity low (latent;
  not reachable under the always-on clamp) |
  repro: code-path only. In `UpdateTime`, if `min_time = prev_blocktime - 600`
    (BIP94, active for MatMul whenever -test=bip94) exceeds
    `max_time = prev_mtp + 3600`, then `nNewTime = std::max(min_time, *max_time)
    = min_time > max_time`, and the `nOldTime > *max_time` branch writes it. At
    height ≥ activation, consensus then rejects with "time-mtp-too-new" — a
    self-rejecting template. Requires `prev_blocktime > prev_mtp + 4200`.
  empirical: REFUTED as reachable — 30 future-clocked blocks under bip94+rule
    never inverted, because the always-on clamp keeps prev_blocktime ≈ prev_mtp
    +3600 so prev_blocktime-600 < prev_mtp+3600. Only reachable if a block with
    prev_blocktime far above MTP is somehow accepted (e.g. injected pre-activation
    via submitblock, then rule activates) — narrow and operational.
  proposed mitigation: make the clamp explicit about precedence, e.g.
    `nNewTime = std::min(std::max(min_time, clock_or_min), max_time_if_present)`
    only when `min_time <= max_time`, and if `min_time > max_time` prefer the
    consensus-valid lower bound and log; or document that drift cap must exceed
    MAX_TIMEWARP by the expected block-time spread.

## Refuted / robust (so others don't repeat)

- INT64 overflow in MaxMatMulFutureBlockTime / MatMulFutureBlockTimeLimit:
  guarded correctly, unreachable with drift=3600 and real timestamps.
- RPC crash/assert via getmininginfo/getblocktemplate/getmatmulchallenge +
  time_policy / CHECK_NONFATAL: not reachable; tip always non-null on those
  paths, CHECK_NONFATAL throws (catchable), no node abort. Hex-size + submitheader
  guards reject malformed input cleanly.
- Pre-hash gate relocation: cheap (SHA256-only), still enforced in full-block
  paths, only moves rejection earlier; no valid block newly rejected, no invalid
  block newly accepted. No DoS amplification.
- Clamp at the cap boundary: inclusive-safe (consensus uses `> cap`); miner emits
  exactly `prev_mtp+3600` and it is accepted. No off-by-one.
- Activation height boundary (199/200/201): consensus flips exactly at the
  configured height; matches PR unit tests. No off-by-one.
- BIP113 lower bound: UpdateTime output always ≥ prev_mtp+1; cannot violate.
