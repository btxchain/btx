## VERDICT on the e1 fix (nonce-in-seed): CORRECT and SUFFICIENT to restore
## per-attempt hardness — but the naive form has TWO design caveats f1 quantifies.

**Does `seed_{a,b}=SHA256(prev‖height‖nonce‖which)` restore Θ(n³)/attempt? YES.**
I audited every axis along which a dense-product amortization could survive once A,B
depend on the nonce (`/tmp/rtF1/fix_analysis.py`, n=32):
- **A·B distinct per nonce:** all AB matrices distinct (4/4), bit-different digests.
- **No per-matrix structure:** rank(A)=rank(B)=32/32 (full) — from-oracle entries are
  full-entropy; no low-rank/Strassen-beating structure on a single AB.
- **No cross-nonce linear relation:** rank(A(k)−A(0))=32 for k=1..4 (full) ⇒ the
  "difference" between two nonces' A is itself dense ⇒ no shared dense work; any
  incremental update costs the full O(n³) again.
- **No shared sub-blocks:** 0 repeated b×b blocks across 8 nonces ⇒ block-product
  caching yields nothing.
Result (`/tmp/rtF1/floors_and_fix.py`): under the fix, **best-attacker == honest =
n³+n² field-mults/attempt ⇒ 1.00× advantage.** The e1 amortization required AB to be
*shared* across the grind; making AB per-nonce removes the only thing to share. There is
**no residual dense-product amortization axis.** Independent bit-exact reproduction of
the e1 break (unfixed) and its collapse under the fix in `/tmp/rtF1/verify_identity.py`
(amortized digest == honest, 3/3 nonces; AB(nonce0)≠AB(nonce1) under fix).

### CAVEAT A (must address in the fix) — naive nonce-in-seed makes the PoW SHA-bound
on small-n networks. Under the fix, A,B (n² elements each) are regenerated PER NONCE via
`field::from_oracle` = **1 SHA256 per element**. Per-nonce input-gen jumps:
| net | SHA/nonce UNFIXED (A,B cached once/tip) | SHA/nonce FIXED (A,B per-nonce) |
|-----|---:|---:|
| mainnet | 16,640 | **540,928** |
| regtest | 1,088 | **9,280** |
Treating 1 SHA256 ≈ 100 field-mults (`/tmp/rtF1/sha_bottleneck.py`): the SHA generation
becomes **28.7% of per-nonce cost on mainnet but 77.7% on regtest (n=64).** So the naive
fix shifts a large share of PoW weight from field-matmul (the work the chain *wants*)
onto **SHA256 — the single most ASIC-optimized primitive on Earth.** On mainnet matmul
still dominates (~71%), so the fix is acceptable there; but the fix should be specified
to keep matmul ≫ generation (e.g. derive A,B from a *cheap stream expansion of one
per-nonce seed* rather than one SHA256 per element, or only fold the nonce into a small
per-nonce **delta** added to a cached base — see "stronger fix" below). No incremental-SHA
shortcut exists across nonces (the 32-byte seed prefix is itself a hash of the nonce, so
no SHA midstate is shared) — confirmed, so generation is irreducible Θ(n²) SHA/nonce.

### CAVEAT B (must extend the fix) — the service-challenge subsystem has the SAME hole.
`getmatmulservicechallenge` (rpc/mining.cpp:1551-1555) derives seeds with the identical
nonce-free `DeterministicMatMulSeed(prevhash,height,which)`. The same e1 amortization
applies to any nonce-grind graded against a service challenge. Worse, the RPC literally
exposes a `next_block_seed_access` profile advertising `winner_knows_next_seeds_first`
and `publicly_precomputable_before_parent_seen` (mining.cpp:1469-1482) — the designers
KNEW seeds are precomputable and treated it as benign. **The nonce-in-seed fix must apply
here too**, or service-challenge solving remains amortizable independently of the block PoW.

## (1) Sigma pre-hash lottery — quantified; NO asymmetry, but it is a work-DILUTION knob
`CheckMatMulPreHashGate` (pow.cpp:2029-2037): sigma=DeriveSigma(header) (2×SHA256 over a
~146-byte header incl. nonce); gate passes iff `sigma ≤ target<<eps`. Only gate-winners run
the O(n³) matmul (pow.cpp:1074-1080) — for BOTH honest and attacker.
- **Pass rate = 2^−(work_bits−eps); E[matmuls per BLOCK found] = 2^eps = 2^18 = 262,144.**
  i.e. 99.9996% of nonces are filtered by 2 SHA256 and never touch the matmul.
- **No asymmetry:** the honest reference also gates (pow.cpp:1074-1080), so both pay 2^18
  matmuls/block. The gate sets *how many* Stage-B evaluations happen; the e1 speedup
  multiplies each Stage-B eval and is **unchanged** by the gate (`/tmp/rtF1/sigma_lottery.py`).
  Stage-A (2×SHA/nonce) is identical and non-amortizable for both (sigma depends on nonce in
  both unfixed and fixed designs). A "smart miner" cannot skip Stage-B work the honest miner
  does — both skip the same losers; the gate is the *intended* cost shape, not a shortcut.
- Block-level mainnet work with the gate folded in: honest 35.25e12 vs e1 3.37e12 field-mult
  ⇒ **10.47× block-level edge** (corroborates e1's 12.8×; difference is op-accounting of the
  low-rank term — same conclusion). The gate does not close it.

## (2) Other amortizable per-nonce quantities / memory-hardness — audited
- **Noise E,F:** rank-r, generated per nonce from sigma; computed via associativity in
  O(n²r). This is the honest cost AND the attacker cost — not an asymmetry, it's the floor.
- **Compress / digest:** `Compress` is a **linear** dot with v (transcript.cpp:204-230) ⇒
  `acc=Compress(AB)+Compress(Mnoise)` separates — this LINEARITY is precisely what let e1
  pull AB out. Under the fix AB is per-nonce so separability no longer helps the attacker.
- **FromSeed cache (matrix.cpp:32, capacity 8):** in the UNFIXED design this cache hands the
  attacker A,B for free for the whole grind (seeds nonce-independent ⇒ ~100% hit) — it is an
  *enabler* of e1. **Memory-hardness: NONE** — working set is 8·n² words ≈ 8 MB at n=512,
  fits L3/RAM trivially; no enforced large random-access table. The fix incidentally neuters
  the cache (hit-rate ~0 ⇒ A,B regen per nonce as intended).
- **Incremental / time-memory tradeoff:** none found. Cross-nonce A differences are full-rank
  (no incremental update), no SHA midstate sharing, no precomputable table.

## (3) Verification-side asymmetry beyond b1/e1 — the structural root, closed by the fix
Product-committed path (mainnet ≥61000, pow.cpp:2253-2305) and Freivalds path
(2222-2227): the verifier does **only O(k·n²) Freivalds + O(n²) digest over the SUPPLIED
C'** — it **NEVER recomputes A'·B' densely.** It *assumes* obtaining C'=A'·B' cost the
prover O(n³). That assumption is the entire asymmetry e1 exploits: Freivalds (b1: sound,
2^−62) proves C'=A'B' but says **nothing about HOW MUCH WORK** produced C'; and with AB
shared across nonces the prover gets C' in O(n²r). **No NEW verification asymmetry beyond
this** — the digest/Freivalds binding is tight (b1). The fix (per-nonce AB) is the correct
closure: it forces the prover to actually spend the O(n³) the verifier assumes.

## NEW / unmitigated findings
- **f1-A | high (fix-design) | the e1 `nonce-in-seed` fix, in its naive form, partially
  converts the PoW from field-matmul to SHA256.** Regenerating A,B per nonce = Θ(n²) SHA256;
  on regtest (n=64) this is **77.7%** of per-nonce cost, on mainnet **28.7%**. SHA256 is the
  most ASIC-favorable primitive, undermining the "matmul is the work" thesis on small-n /
  high-r networks. repro: `python3 /tmp/rtF1/sha_bottleneck.py`.
  **Mitigation (stronger fix):** keep matmul ≫ generation. Option (i): derive A,B from a
  *cheap counter-mode stream* keyed by ONE per-nonce seed (amortize SHA setup), not one
  SHA256/element. Option (ii) "delta fix": A=A_base+δ_A(nonce) where A_base is cached
  (O(n²) once/tip) and δ_A is a *full-rank* per-nonce perturbation that still forces a fresh
  dense product — but δ_A must be full-rank (NOT low-rank, else e1 re-applies to the
  base·base term). Verify any chosen scheme leaves **zero shared dense work** (f1's axis
  tests) AND matmul-mults ≥ ~3× generation-SHA at the smallest live n.
- **f1-B | high | the service-challenge subsystem (`getmatmulservicechallenge`,
  mining.cpp:1551-1555) uses the SAME nonce-free seed derivation ⇒ e1 amortization applies
  there independently.** The fix MUST extend to it (and the `next_block_seed_access` profile
  fields that advertise precomputability should be re-evaluated). repro: code (seeds =
  `DeterministicMatMulSeed(prevhash,next_height,{0,1})`, no nonce).
- (info) The pre-hash gate is a work-dilution knob (2^eps matmuls/block), NOT a hardness or
  asymmetry lever; it does not mitigate e1. eps tuning is orthogonal to the amortization.

## Refuted / robust (so others don't repeat)
- **`nonce-in-seed` DOES restore per-attempt hardness** — no residual dense-product
  amortization on any axis (per-matrix structure, cross-nonce linearity, sub-block reuse all
  full-rank/zero-reuse; best-attacker==honest, 1.00×). The e1 fix is *directionally correct*;
  only the two caveats above need folding in. (`/tmp/rtF1/fix_analysis.py`, `floors_and_fix.py`)
- **No verification asymmetry beyond e1/b1.** Verifier legitimately does O(n²); the gap is the
  prover-work assumption that e1 attacks, closed by the fix. Freivalds binding tight (b1).
- **No memory-hardness exists and none is newly creatable** by the fix; working set ~MBs.
- **Pre-hash sigma lottery is symmetric** (honest path gates identically); not a shortcut.

## Open questions / leads
- The "delta fix" (cached A_base + full-rank per-nonce δ) could keep matmul-dominant AND
  cheap to generate, but needs a proof that base·base never becomes a shareable term (i.e.
  δ must enter multiplicatively/full-rank, not as an additive low-rank like E,F). Worth a
  dedicated design pass before the hard-fork flag-day.
- C++ wall-clock PoC (build a regtest miner doing per-nonce A,B-regen) would confirm the
  SHA-vs-mult balance with real constant factors; f1 quantified it in the field-exact model
  (stated per the "say so" rule — no C++ submitblock PoC built).
