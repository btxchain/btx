## 1. ARCHITECTURE MAP — how unshield works on-chain today (two distinct mechanisms)

There are **two completely different** shielded→transparent mechanisms in-tree. The community ask
conflates them; separating them is the whole insight.

### (A) The OPERATOR/BRIDGE egress path (what "use bridge unshield" points at)
This is an L2/netting settlement, NOT a per-user shielded spend:
- A user's notes enter the pool via **V2_INGRESS_BATCH** and are netted **off-chain** into an
  aggregated `BridgeBatchStatement` (the canonical `statement_hex`).
- The statement is attested off-chain by a **verifier set** (threshold ML-DSA signatures) and/or a
  **proof-receipt set**, producing `BridgeBatchReceipt` / `BridgeProofReceipt` artifacts. A
  **V2_SETTLEMENT_ANCHOR** tx commits the attested statement on-chain; it must mature and is
  single-use (`src/validation.cpp:6299-6326`, `IsSettlementAnchorMature`, `UseSingleUseSettlement
  Anchors`).
- A **V2_EGRESS_BATCH** tx then debits the pool against that matured anchor and mints transparent
  outputs. CRUCIAL: the egress pool debit is taken **directly from the operator-attested statement**,
  not from a shielded-note spend in the egress tx:
  `value_balance = -witness->statement.total_amount` (`src/shielded/bundle.cpp:1067-1072`). The
  egress tx carries NO shielded spends and NO SMILE2 proof — only a `SettlementWitness`
  (`VerifySettlementContext`, `src/shielded/v2_proof.cpp:2616-2746`).
- The verifier-set / proof-receipt requirement is enforced inside `VerifySettlementContext`: the
  threshold-signature branch fires when a verification bundle is present (`v2_proof.cpp:2695-2730`,
  rejects `bad-v2-settlement-signed-threshold` / `…-verifier-set` / `…-verification-bundle`); the
  proof-receipt-only branch requires a valid external anchor built from receipts
  (`v2_proof.cpp:2740-2745`).
- The wallet RPCs that drive this (`bridge_buildegressbatchtx`, `bridge_buildproofanchor`,
  `bridge_signbatchreceipt`, `bridge_buildunshieldtx`) all take a `statement_hex` and pre-existing
  receipts/plan as **inputs** (`src/wallet/rpc/wallet.cpp:1208-1386`; help signature of
  `bridge_buildegressbatchtx "statement_hex" [{proof_system_id,verifier_key_hash}…]
  ["proof_receipt_hex"…] [{address,amount}…]`). **No node RPC PRODUCES a validly-attested
  statement_hex+receipts** — that is exactly the off-chain operator/L2 software the community
  flagged. `bridge_buildunshieldtx` (`shielded_rpc.cpp:14844-14900`) merely settles an
  already-funded bridge-out CTV plan output.

**Why this leaks linkage:** the off-chain operator computes the netting and therefore learns which
shielded inputs map to which transparent egress outputs (and the timing). That is the privacy + trust
problem in the ask.

### (B) The DIRECT V2_SEND path (the real self-service unshield — already consensus-valid)
A `V2_SEND` tx can carry shielded spends, shielded outputs, AND transparent `vout`. Its
`value_balance = transparent_out + fee` (positive ⇒ value leaves the pool to the transparent side),
and it carries a real **SMILE2 DIRECT_SMILE proof** over the spent notes. This is a genuine
per-user, self-contained shielded→transparent spend with no operator. **It is fully wired and
consensus-accepted post-fork; only wallet policy disables it.**

Evidence the builder already does it (`src/shielded/v2_send.cpp:479-510`):
```
if (has_shielded_spends) {
    total_public_value = transparent_value_out + fee;
    require total_input_value == total_output_value + total_public_value;   // value conservation
    payload.value_balance = total_public_value;          // positive = unshield
}
```
Evidence the consensus accepts it:
- **No fork gate rejects a V2_SEND with transparent outputs / positive value_balance.**
  `RejectDirectSendEncodingForFork` (`src/shielded/validation.cpp:211-239`) post-fork ONLY requires
  spends use `SMILE_COMPACT_POSTFORK` encoding and forbids lifecycle controls — it does NOT touch
  `value_balance` sign or `vout`. I grepped the whole consensus path for a mixed-transparent /
  positive-value_balance V2_SEND rejection: **none exists.**
- The SMILE2 balance check binds it: `payload.value_balance` is passed as `public_fee` into
  `VerifySmile2CTFromBytes` (`src/shielded/v2_proof.cpp:2510-2517`) → `VerifyCT` h2 balance check
  enforces `Σ inputs == Σ shielded outputs + value_balance` (c5 §1, `ct_proof.cpp:3122-3133`).
- The transparent-side conservation closes the loop: `CheckTxInputs` folds `value_balance` into
  Bitcoin's input/output equation (c5 §1, `tx_verify.cpp:203-227`), and the turnstile pool delta =
  `value_balance` applied with pool≥0 (`validation.cpp:1075-1086`, `bundle.cpp:1067 V2_SEND →
  payload.value_balance`). `CShieldedBundle::IsUnshieldOnly()` / `turnstile.h:25` document
  `value_balance > 0 means value leaves pool (unshield)`.
- `z_sendmany` already parses transparent recipients and calls `CreateV2Send` with them
  (`shielded_rpc.cpp:8570-8604, 8646-8654`); `CreateV2Send` already appends them as
  `tx_template.vout` and builds the full proof (`shielded_wallet.cpp:3425-3435`).

### Why the wallet z→t was disabled post-fork (the actual reason)
NOT a consensus security reason and NOT an incomplete proof builder. It is a **privacy-redesign
policy decision**: `AllowMixedTransparentShieldedSendAtHeight(h) = !UseShieldedPrivacyRedesignAt
Height(h)` (`src/wallet/shielded_privacy.cpp:61-64`), gated on `nShieldedMatRiCTDisableHeight`. The
redesign pushed users toward the bridge for unshield (the error string literally says "use bridge
unshield"). The builder/proof/consensus for direct z→t were left intact — the gate is a single
early-return:
`src/wallet/shielded_wallet.cpp:2988-2991`:
```
if (!transparent_recipients.empty() && !AllowMixedTransparentShieldedSendAtHeight(validation_height))
    return fail("post-fork mixed shielded-to-transparent direct sends are disabled; use bridge unshield");
```
(Same gate also short-circuits the other `CreateV2Send` overload around 2807-2944, reached via the
mixed routing in 4617-4681.)

## 4. SECURITY ANALYSIS

### 4.1 Does re-enabling re-open the post-fork-disable risk? NO consensus risk; it was a policy choice.
The disable was a privacy-redesign *policy* nudge (§1), not a consensus fix. There is no consensus
rule that a direct z→t was violating. So re-enabling does not re-open a consensus hole. The thing it
*does* re-expose is the privacy/UX intent the redesign had (push to bridge) — which is precisely
what we WANT to reverse for the self-service privacy win.

### 4.2 f3 / SMILE2 monomial-challenge soundness — THE security gate (interaction is real).
The unshield proof IS a SMILE2 `DIRECT_SMILE` CT proof, whose commitment-opening soundness rests on
the **unproven 2^-8 monomial-FS → M-SIS reduction** that BTX's own auditors flagged as **C-002** and
deferred (f3-1; `doc/btx-audit-reassessment-remediation-plan-2026-03-23.md:274-294`). Self-service
unshield WIDENS exposure of this gap in three concrete ways:
- **Permissionless + high-volume.** Today the only consensus-active SMILE2-spend surface to the
  transparent side is gated behind wallet policy and the (operator-mediated) bridge. Self-service
  unshield makes the SMILE2 spend→extract-value-to-transparent path open to every holder, at OTC-desk
  volume. If the monomial reduction is loose, the payoff of a forged spend is direct BTX exfiltration
  to a transparent address (no operator to net it out / catch it).
- **Grinding economics improve for the attacker.** f3's residual attack is transcript grinding
  (~256 attempts to hit an accepting monomial for a commitment the prover can't validly open). A
  self-service builder is a clean, scriptable harness to do exactly that grind locally and then
  broadcast — no operator in the loop to refuse.
- **It does NOT create the gap.** If the reduction holds (likely, per f3), the feature is sound.
**REQUIREMENT:** ship self-service unshield ONLY with the C-002 external-audit verdict resolved, OR
behind the PR3 height-gated monomial→Hamming-weight-polynomial challenge swap. Do not headline an
operator-removed, permissionless value-extraction path on an *internally-flagged-unproven* soundness
assumption. This is the single most important security caveat.

### 4.3 The prover SIGSEGV (c5/e3) MUST be fixed first — it is the feature's prover.
Confirmed 4× (exit 139). `CreateV2Send` for the unshield calls `CreateSmileProof → TryProveCT`, the
crashing path. Shipping the feature on a prover that segfaults = unusable + dangerous (a
memory-safety bug in consensus-adjacent bespoke crypto; e3 notes a prover bug often shadows a
matching verifier bug that WOULD be remote-reachable). PR2 #15 (ASan/UBSan root-cause + fix + prover
AND verifier fuzz targets) is a hard precondition, on the critical path for this headline.

### 4.4 Value-conservation / no-inflation on the z→t boundary — CONFIRMED present (reuses c5).
The unshield path uses the exact checks c5 validated, with positive `value_balance`:
- SMILE2 h2 balance: `Σ inputs == Σ shielded outputs + value_balance` (`ct_proof.cpp:3122-3133`,
  via `VerifySmile2CTFromBytes(... payload.value_balance ...)` `v2_proof.cpp:2510-2517`).
- Transparent conservation: `CheckTxInputs` folds `value_balance` (`tx_verify.cpp:203-227`).
- Turnstile pool≥0: `ApplyValueBalance(value_balance)` for V2_SEND (`validation.cpp:1075-1086`;
  `bundle.cpp:1049-1050,1067` derives V2_SEND delta from `payload.value_balance`).
- Range / nullifier / membership / proof-tx-binding: all unconditional in ConnectBlock (c5 §2-§6),
  family-agnostic ⇒ apply equally to an unshielding V2_SEND. No new inflation surface beyond the
  pre-existing f3 soundness unknown.
- Structural: `SendPayload::IsValid` requires `value_balance >= fee` when spends present and
  `outputs` non-empty (`v2_bundle.cpp:714-731`) ⇒ a *pure* unshield (zero shielded change) is not
  representable; the builder must always include ≥1 shielded output (in practice a change note;
  zero-value change note if exact). Design accounts for this (§5).

## 5. PRIVACY ANALYSIS

### 5.1 Big win vs the operator path.
Node-built unshield removes the operator entirely, so **no third party learns the
shielded-source↔transparent-dest linkage** or the timing/amounts. The bridge/egress path inherently
exposes that mapping to the off-chain netting operator (§1A). This is the headline privacy benefit.

### 5.2 Residual on-chain linkability (same as any shielded spend; already mitigated in code).
- **Anonymity set / ring:** SMILE path proves membership against the FULL tree snapshot (no decoy
  selection to deanon; e3). The wallet enforces a minimum privacy tree size and ring diversity before
  allowing the send (`shielded_wallet.cpp:2993-3008`,
  `GetShieldedMinimumPrivacyTreeSizeForHeight`), and a post-fork dust threshold. Keep these for
  unshield (do NOT relax for OTC convenience).
- **Amount correlation:** the transparent output amount is visible and equals the unshielded value
  (minus fee). This is intrinsic to any unshield (Zcash t-side has the same property). Mitigate by:
  canonical fee buckets (already enforced for V2_SEND, `IsCanonicalShieldedFee`); recommend
  round/standard denominations in wallet UX; allow splitting across multiple t-outputs.
- **Change handling:** the mandatory shielded change note keeps the remainder private; output order
  is shuffled (`ComputeShieldedOutputOrder` Fisher-Yates over a build_nonce,
  `shielded_privacy.cpp:199-212`) and the shared ring seed is build-nonce-randomized
  (`DeriveShieldedSharedRingSeed`, 180-197). Good — reuse for unshield unchanged.
- **Timing:** to avoid coupling the unshield to a just-prior receive, the wallet should support
  **decoupled / user-scheduled** broadcast (optional delay or "broadcast later"), and must not auto-
  trigger an unshield on receipt. No operator metadata is emitted (no statement_hex, no receipts).
- **Anchor reuse:** the direct path references the live merkle/account-registry anchors (tracked
  historical roots, c5 §4); no per-unshield single-use settlement anchor to correlate. Use the
  freshest valid anchor to avoid pinning to an old root.

### 5.3 New linkability introduced? None beyond the above. The direct path emits strictly LESS
metadata than the bridge path (no statement, no verifier set, no settlement anchor, no CTV plan).

## 6. CONCRETE DESIGN + IMPLEMENTATION PLAN

### 6.1 RPC surface
- **Primary: re-enable z→t in `z_sendmany`** behind a height/feature gate, OR
- **Cleaner: a dedicated `z_unshield <fromzaddr|*> [{address,amount},…] fee` RPC** that internally
  calls `CreateV2Send(shielded_recipients={}, transparent_recipients=…)`, plus a mandatory shielded
  change note. A distinct RPC name (a) makes the privacy semantics explicit, (b) lets us keep
  `z_sendmany`'s historical "no z→t" contract for compatibility, (c) gives a natural home for the
  privacy knobs (delay, denomination splitting, min-anonset enforcement).

### 6.2 The one-line unblock + supporting changes
1. Replace the early-return at `shielded_wallet.cpp:2988-2991` with a gate keyed to a new policy
   `AllowSelfServiceUnshieldAtHeight(h)` (default ON at/after the 0.30.3 activation), instead of
   `!AllowMixedTransparentShieldedSendAtHeight`. Mirror in the second overload (2807-2944) and the
   routing in 4617-4681. (~1 screen.)
2. Ensure the builder always adds a shielded change output for a pure unshield (so `outputs` is
   non-empty per `v2_bundle.cpp:716`); if change==0, emit a zero/dust-bucket shielded note (it
   already prefers exact-balance change reserves, `PreferExactBalanceShieldedChangeReserve`).
3. Keep all anonymity gates (min privacy tree size, ring diversity, dust threshold, canonical fee
   bucket) — do NOT bypass for OTC.
4. **No node RPC for `statement_hex` is needed** — the direct path doesn't use it. (If we ever wanted
   self-service *egress*, we'd have to build a node-side statement+self-attestation, which WOULD be a
   bigger consensus change — avoid; the V2_SEND path is strictly simpler and already valid.)

### 6.3 Statement/proof builder to bring into the node
**Nothing to import** — `CreateSmileProof`/`TryProveCT` (`src/shielded/smile2/wallet_bridge.cpp`,
`ct_proof.cpp`) is already the node's prover for V2_SEND. The feature's only "new" builder work is
the SIGSEGV fix in that same prover.

### 6.4 Hard dependencies / sequencing (the critical path)
1. **FIX the SMILE2 prover SIGSEGV (PR2 #15)** — ASan/UBSan root-cause; the intermittent crash points

2. **RESOLVE f3 / C-002** — external-audit verdict on the monomial-FS soundness, OR land PR3's
   monomial→Hamming-weight challenge swap first. *Gates whether it's safe to headline.*
3. Then land the wallet gate + `z_unshield` RPC + UX.

### 6.5 Test plan (once the prover is fixed)
- **Functional:** regtest, post-fork active; build pool ≥ min privacy tree size; `z_unshield` 5 BTX
