# h1 — VERIFY the e1 PoW-amortization fix (fold nonce into MatMul seeds) for regressions

**Verdict: the fix WORKS (defeats e1, empirically confirmed end-to-end on a live patched
regtest node) but is NOT regression-free as proposed. The one-line "patch pow.cpp:47 +
validation + miner + genesis" scope is INCOMPLETE and has real perf + API + test regressions.**
With the additional surface patched (below), it is a sound HARD FORK. A strictly BETTER fix
exists (keep the cheap pre-hash gate seed-independent) that removes the largest perf regression.

Build: `git archive pr-214` → patched → `docker build` = **clean compile, btxd links** (build.log).
Empirical node: `rth1-pr214-e1fix`, regtest, `-test=matmulstrict` (seed-binding + full Phase2 enforced).

## What I implemented (the fix, in /tmp/rtH1 build variant)
- `DeterministicMatMulSeed(prev,height,nonce,which)` — folds `uint64 nonce` into the SHA (pow.cpp:47, pow.h:190).
- Consensus enforcement now nonce-aware: validation.cpp:8385 passes `block.nNonce64` → "bad-matmul-seeds".
- Miner template (miner.cpp:693), getmatmulchallenge (mining.cpp:1552), GenerateBlock (mining.cpp:4484),
  genesis tool (btx-genesis.cpp:375 + per-nonce A,B in the grind loop), test util (mining.cpp:45).
- **The hard part the proposal omits — the mining HOT PATH**: `SolveMatMul` (pow.cpp) and its
  batch pipeline hoist `A=SharedFromSeed(seed_a)` ONCE and reuse it across the whole nonce batch
  (pow.cpp:2776 + `SubmitMatMulDigestPreparedBatchForMining(headers,*A,*B,…)`). That hoist IS the
  honest-miner embodiment of e1. I had to: (a) re-derive per-nonce seeds in
  `BuildMatMulNonceBatchWindow` (added `seed_height` param), (b) **force `configured_batch_size=1`**
  (the cross-nonce batch shares one A,B — incompatible with per-nonce seeds), (c) recompute A,B from
  each window header's seed before the digest, (d) persist the winning nonce's seeds onto the block.
- Updated the `getmatmulchallengeprofile` `cross_nonce_reuse` metadata (no longer advertises reuse).

## 1. Does it actually defeat e1?  YES — confirmed in math AND on the live node.
- **Algebra (`/tmp/rtH1/defeat_e1.py`, field-exact mod 2^31-1, n=32):** OLD scheme → A·B IDENTICAL
  across nonces 0..3 (amortizable, e1's "precompute A·B once"); FIX → A·B DISTINCT every nonce
  (4/4 distinct hashes) ⇒ the dense O(n³) product cannot be precomputed/amortized. e1's decisive
  premise (e1.md:29 "A and B are FIXED for the whole grind") is directly negated.
- **Live node:** mined under `-test=matmulstrict` (full Phase2 + seed-binding). Block @h4 nonce64=2
  carried seed_a=eccd2a97…; I independently computed `SHA256(prev_le‖h4‖nonce2‖0)` = eccd2a97… (EXACT
  match) ⇒ consensus binds seeds to the nonce. **Rejection proof:** resubmitting that block with the
  OLD nonce-independent seeds `SHA256(prev‖h‖which)` → **`bad-matmul-seeds` REJECT**. So amortized/
  fixed-seed blocks are now consensus-invalid. Honest cost == attacker cost (both pay full n³/nonce).
- Genesis remains valid WITHOUT re-mining: seed-binding is skipped for null-prevhash (validation.cpp:8384),
  so the existing hardcoded mainnet/regtest genesis blocks need no change (good compatibility result).

## 2. Mining performance regression — REAL, quantified.
- **Per-nonce seed-regen (the added O(n²) SHA fill)** is NOT "negligible" — the e1 claim glossed the
  constant factor (each matrix entry = one full SHA256). Microbench (`/tmp/rtH1/bench_seedfill.cpp`,
  real from_oracle cost):
  - mainnet n=512: FromSeed(A)+FromSeed(B)=283ms vs dense n³ product=1003ms ⇒ **+22% per accepted
    nonce** (honest miner now ~1.28× its prior digest cost). Acceptable but not negligible.
  - regtest n=64: seed-fill 4.37ms vs product 0.96ms ⇒ seed-fill is **4.6×** the product (SHA dominates
    at small n). Benign for regtest, but shows the O(n²) SHA term is NOT free below the crossover.
- **Pre-hash prefilter regression (the BIG one, mainnet):** the cheap sigma gate
  (`DeriveSigma` hashes seed_a/seed_b) now needs the per-nonce seeds derived FIRST ⇒ +2 SHA256 per
  scanned nonce. `/tmp/rtH1/bench_prehash.cpp`: prefilter 2125→3275 ns/nonce = **1.54× slowdown**.
  At mainnet eps=18 the prefilter runs ~2^18×/hit and DOMINATES the grind ⇒ ~1.5× honest hashrate
  loss on top of the 1.28× digest cost. ★ This is AVOIDABLE — see "better fix" below.
- **Caching / pipelining BREAKS by construction:**
  - `g_from_seed_cache` (cap 8, matrix.cpp:32) becomes useless — every nonce is a fresh seed ⇒ 100%
    miss + per-nonce mutex traffic. Should be bypassed for the matmul path.
  - The whole **batched digest pipeline** (`SubmitMatMulDigestPreparedBatchForMining`, the CUDA/Metal
    `matrix_a_cache_key=&blocks.front().seed_a` batch APIs) is built on "one A,B per nonce batch" —
    exactly the e1 amortization. The fix forces batch_size=1, **disabling cross-nonce batching/GPU
    amortization entirely.** (Note e1's own lead: the amortized approach is MORE GPU-friendly; the fix
    therefore makes honest mining LESS batch/GPU-efficient — arguably good for ASIC-resistance, but a
    real honest-throughput regression.)
  - **Empirical end-to-end:** 30 blocks strict regtest n=16: baseline pr214 31.4s vs patched 36.2s =
    **~15% slower** (regtest has eps=0 so this is digest-path+batch-loss only; mainnet adds the 1.54×
    prefilter). `verifychain 4 100` = true (all blocks revalidate deterministically).
- Pre-hash sigma GATE BEHAVIOR otherwise unchanged: still a strict relaxation; I placed the per-nonce
  seed derivation BEFORE the sigma check so sigma reflects the correct per-nonce seed (consensus side
  is consistent because validation enforces seed==derive(nonce) before the gate runs).

## 3. Consensus / validation correctness — one MISSED consensus-adjacent subsystem.
- **d2 backend determinism: PRESERVED.** Seeds stay pure-integer SHA256; A,B from from_oracle integer
  oracle; digest recomputed on CPU from the block's stored seed. batch=1 makes the CPU path MORE
  canonical. No float, no reassociation change. `verifychain` deterministic over 34 blocks.
- **Header relay / serialization: UNCHANGED** — seed_a/seed_b were already header fields (BTX_HEADER_SIZE
  182). GetHash/sigma layout untouched. seed is now redundant-but-committed (derivable from prev,height,
  nonce); no new grinding freedom (all seed inputs are fixed or the nonce itself).
- **★ getmatmulchallenge RPC contract BROKEN (API regression, MED).** The challenge returns a SINGLE
  `matmul.seed_a/seed_b` + `header_context.seed_a/seed_b` derived at `nonce64_start=0` over the full
  `noncerange 0..ffff…`. An external miner/pool/ASIC following this contract uses those FIXED seeds for
  every nonce ⇒ every nonce≠0 block is `bad-matmul-seeds` REJECTED. The RPC must stop pinning a fixed
  seed and instead publish the per-nonce derivation rule (I patched `seed_derivation_rule`, but the
  returned seed VALUES still mislead). Internal generatetoaddress path works (I fixed SolveMatMul).
- **★ getmatmulservicechallenge subsystem NOT fixed (scope gap, MED).** `solvematmulservicechallenge`
  derives seeds via `DeriveMatMulServiceSeed(challenge_id, anchor_hash, label)` = `sha256(challenge_id‖
  anchor‖label)` (mining.cpp:1654, 3146, 3907) — **nonce-independent** — then `BuildMatMulServicePowState`
  fixes `state.seed_a` and grinds `state.nonce` up to max_tries=500k against it. ⇒ **identical e1
  amortization still applies** to the service-challenge PoW. Not block consensus, but it's the same
  algorithm + same break; the fix as proposed leaves it open. Must fold the attempt-nonce into
  `DeriveMatMulServiceSeed` and re-derive per nonce in the service solver too.
- The codebase EXPLICITLY models the e1 reuse: `getmatmulchallengeprofile.cross_nonce_reuse` computed
  `fixed_clean_product_field_muladds_upper_bound = transcript_field_muladds` and an
  `amortization_advantage_upper_bound` (observed = 5 on the pre-fix image). I corrected these to
  no-reuse (seed_scope→per_nonce, fixed_*→0, advantage→~1). (Source-complete; the empirical image
  predated this cosmetic edit — consensus parts were in and proven.)

## 4. New attack surface — none from per-nonce seeds.
- Seed = deterministic SHA of already-committed (prev,height,nonce); attacker gains no new grinding
  lever (nonce was already the grind variable; sigma already varied per nonce via nNonce64). No new
  circularity (seed depends on nonce, sigma on seed — acyclic). Difficulty/ASERT/target derivation
  untouched ⇒ no timing/difficulty change. No precompute horizon: a miner can no longer precompute the
  next tip's A·B before seeing the parent (it now also needs the nonce) — strictly REMOVES the
  precompute/ASIC advantage e1 flagged.

## NEW / unmitigated findings
- **Incomplete-fix: service-challenge PoW still e1-amortizable** | MED | repro: seeds via
  DeriveMatMulServiceSeed are nonce-independent, solver grinds nonce vs fixed seed | mitigation: fold
  attempt-nonce into DeriveMatMulServiceSeed + re-derive per nonce in the service solver.
- **getmatmulchallenge RPC pins a nonce-0 seed over the whole noncerange** | MED (breaks external
  miners) | mitigation: don't return a fixed seed; publish the per-nonce derivation rule/algorithm.
- **Pre-hash prefilter 1.54× slowdown** | MED perf | the per-nonce seed feeds the CHEAP sigma gate
  unnecessarily.

## ★ BETTER FIX (recommended)
The damaging amortization is ONLY in the expensive dense product A·B. So make ONLY A,B depend on the
nonce, and keep the cheap pre-hash sigma gate seed-INDEPENDENT:
- Fold the nonce into seed_a/seed_b (as here) — defeats e1 in the matmul. ✔
- But compute the pre-hash sigma over a FIXED placeholder seed (or exclude seed_a/seed_b from
  `ComputeMatMulHeaderHash` on the pre-hash path), so the 2^eps prefilter stays at 2 SHA/nonce and the
  seed (2 more SHA + the n² fill) is derived ONLY for nonces that PASS the prefilter. This removes the
  1.54× prefilter regression — the dominant honest cost at mainnet eps=18 — while still forcing a fresh
  O(n³) A·B per surviving nonce. (Verify this doesn't open a grind: sigma must still bind nNonce64, which
  it does; the seed is a deterministic function of the same nonce, so prefilter-then-derive is sound.)
- Also: bypass `g_from_seed_cache` for the matmul path; redesign the digest pipeline to overlap
  per-nonce A,B generation across threads (recover some throughput) WITHOUT sharing A·B; fix the two
  subsystems above; rewrite the 16 batch-pipeline tests to assert per-nonce A,B regeneration instead.

## Refuted / robust (so others don't repeat)
- The fix does NOT regress d2 determinism, header relay, serialization, difficulty/ASERT, or genesis
  validity (genesis seed-check is null-prevhash-exempt). It introduces no new grinding/precompute.
- "FromSeed is negligible O(n²)" — TRUE asymptotically, but +22% (mainnet) / 4.6× (regtest n=64) by
  constant factor; budget for it.
