## NEW / unmitigated findings

### ★ w3-1 | NULLIFIER FORGE — spend-while-presenting-any-nullifier ⇒ DOUBLE-SPEND | severity CRITICAL
- **Repro (verifier-level, DEFINITIVE):** build a real spend of a note at `secret_index`, but set
  `serial_numbers[0]` to ANY non-null attacker value instead of `ComputeSerialNumber(sn_ck, sk) =
  ⟨b_sn,s⟩`; build the rest of the proof normally. `VerifyCT(proof, 1,1, pub, fee=0, bind=true)`
  → **returns true** (`[rtW3-inv2a] verify=1`, `[rtW3-inv2b]` two distinct serials both verify=1).
- **Why it works (root cause — x1c's class, applied to the nullifier):** in the LIVE m=1 consensus
  path (`UsesLiveM1Layout() == rec_levels==1`, the only path `VerifyCT` accepts, ct_proof.cpp:3112),
  the verifier uses `proof.serial_numbers` in **exactly three places**: a size check
  (ct_proof.cpp:3146), an **append into the FS transcript** (3277), and a log line. It is **NEVER**
  compared to an independently-derived value, and **NEVER** tied by an arithmetic equation to the
  key opening `z0`. The serial→key relation the prover comments (`⟨b_sn,z_0⟩ = ⟨b_sn,y_0⟩ + c_0·sn`,
  ct_proof.cpp:2817-2818) is **NOT closed**: the verifier computes `⟨b_sn,z_0⟩` and appends it to
  the transcript (3434-3441) but never checks `⟨b_sn,z_0⟩ − c_0·serial_numbers[i] == w_sn` against a
  committed `w_sn`. The serial-number garbage commitment `t'_1` exists ONLY in the legacy
  multi-level `else` branch (ct_proof.cpp:2592); the live m=1 aux layout commits ONLY `GSlot`/`PsiSlot`
  (2587-2589) — **there is no `w_sn` and no serial relation on the consensus path.** So `serial_numbers`
  is a free, prover-chosen field whose only constraints are non-null + intra-proof-unique
  (verify_dispatch.cpp:187-221).
- **End-to-end (DEFINITIVE by code, no extra run needed):** the on-chain nullifier is computed
  VERBATIM from this field — `ExtractSmile2SerialNumbers` does `serial_numbers = proof.serial_numbers`
  (verify_dispatch.cpp:223), then `ComputeSmileSerialHash(serial)` becomes the `Nullifier`
  (v2_proof.cpp:2003). The spend-auth check (`CShieldedSpendAuthCheck`) only requires
  `proof_bound_nullifier == payload_nullifier` — **both derive from the SAME forged serial** — so it
  is satisfied by construction. e3's claim "a different valid serial would require a different valid
  `s`" is **empirically FALSE on the live path**: the serial is structurally unconstrained.
- **Exploit:**
  1. **Spend a note, present an unspent note's nullifier:** spend your note but emit the serial of a
     DIFFERENT (still-unspent) note. The chain records that other note's nullifier as spent; YOUR
     note's true nullifier is never recorded ⇒ you can spend your note AGAIN later (its real nullifier
     is unused) ⇒ double-spend. Or grief: burn a victim's note's nullifier so the victim's later spend
     is rejected as "already spent."
  2. **Two distinct nullifiers for one note (inv2b):** spend the same note twice in two txs, each with
     a different forged serial. Both proofs verify; the two nullifiers differ ⇒ the
     within/cross-block nullifier-set uniqueness (validation.cpp:5938-5946) does NOT fire ⇒ direct
     double-spend of the same shielded note.
- **Permissionless at consensus** (per x2): `VerifyCT`/`CShieldedProofCheck` runs at MEMPOOL admission;
  attacker hand-builds the raw tx with their own patched prover (the `g_forge_serial` 6-line change).
- **FIX (same shape as x1c's balance fix — "add the MISSING equation"):** make `VerifyCT`
  independently bind the serial: commit `w_sn = ⟨b_sn, y_0⟩` in round 1 (restore the `t'_1` garbage on
  the live m=1 path), and on verify require `⟨b_sn, z_0[i]⟩ − c_0·serial_numbers[i] == w_sn[i]` (the
  same relation the prover comments but the verifier never checks). This forces `serial_numbers[i] =
  ⟨b_sn, s⟩` for the SAME `s` whose `A·s` opens the key/ring ⇒ the nullifier is then note-unique and
  cannot be decoupled. Consensus change; gate @126400; ties to the same external audit (C-002) as x1c.

## Invariant 4 — does the turnstile/pool provide ANY independent backstop? (important for x1c severity)

**At the proof/verifier level: NO.** `VerifyCT`'s signature is `(proof, num_inputs, num_outputs, pub,
public_fee, bind)` — it carries **no running-pool, no total-backing, no UTXO context**. `[rtW3-inv4]`
shows it accepts a forged mint regardless of any pool state (it has none). So the verifier cannot catch
either the x1c balance forge or the w3-1 nullifier forge.

**At the OUTER tx level: a PARTIAL, AGGREGATE backstop only (confirms x1c's severity framing).**
- For the **x1c balance / w3-2 bridge** forge: the outer turnstile `ApplyValueBalance` rejects any tx
  driving the running pool `< 0` (turnstile.cpp:19; validation.cpp:6046 connect + 3302 mempool). This
  **bounds aggregate extraction to the legitimately-shielded pool total** (drained from honest
  depositors) — it does NOT re-derive shielded balance, so it does NOT PREVENT the forge; it only caps
  the blast radius to the pool size. ⇒ x1c is end-to-end exploitable up to the pool balance (a
  self-funded-looking but actually-inflationary unshield), exactly as x1c assessed. The pool is a
  ceiling, not a soundness check.
- For the **w3-1 nullifier** forge: the turnstile does **NOT** bound it at all — a double-spend doesn't
  push the pool negative (each spend is individually value-balanced); it just consumes the same note
  twice. The ONLY backstop for double-spend is the nullifier-set uniqueness, which w3-1 **defeats** by
  presenting a different (forged) serial each time. ⇒ no aggregate ceiling protects against w3-1; the
  bound is "the value of every spendable note an attacker can reach," i.e. the whole pool, repeatedly.

⇒ **Net: there is NO independent invariant anywhere that recomputes the shielded balance OR the
nullifier; the turnstile pool≥0 is a one-sided aggregate ceiling that limits balance-forge extraction
to the pool total but provides ZERO protection against the nullifier double-spend.**

## Refuted / robust (so others don't repeat)
- The MEMBERSHIP-vs-ANCHOR check is NOT a `VerifyCT` responsibility and is NOT forgeable at the verifier:
  `VerifyCT` has no anchor argument; the node reconstructs the rings from its own tree snapshot
  (`BuildV2SendSmileRingMembers`, validation.cpp:1409) and `IsShieldedAnchorValid` pins the anchor to a
  tracked historical root (c5 §4, e3). The forgeable part is the INNER one-of-many binding — and the
  sharpest break there is w3-1 (serial not bound to the ring member's key).
- The bridge ingress proof-skip (zero-consumed-spend) is closed by `IsValid` `consumed_spends≥1` (c5);
  NOT a new hole. The new hole is that the proof's AMOUNT is forgeable (= x1c), not that the proof is skipped.
