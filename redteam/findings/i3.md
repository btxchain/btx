# i3 — z_sendmany (z→z) prover SIGSEGV: ROOT-CAUSED + FIXED (stack overflow in SMILE2 CT prover)

Implementation agent i3. AUTHORIZED. Developed in ISOLATED `/tmp/rtI3-src` (`git archive pr-214`),
built/tested only in docker `rti3-*`, regtest `rtI3-*` namespace. No production/foreign assets touched
(no btxd.service/.btx/.local/bin; no pf3-*/bft-*/evx*/btxn*/other rt* containers/images).

## Headline
- **Root cause: STACK OVERFLOW (not OOB/UAF/inflation) in `smile2::TryProveCT`** — the SMILE2 CT prover
  has a single stack frame too large for the thread it runs on. The frame holds dozens of by-value
  `SmilePoly` (1 KB each) / `NttForm` (1 KB each) temporaries live simultaneously.
- **The trigger is a musl-vs-glibc thread-stack divergence.** BTX ships on **Alpine/musl**
  (`btx-node:pr214` links `ld-musl-x86_64.so.1`), where the default *non-main* thread stack is only
  **128 KB**. `z_sendmany` (z→z) runs the prover on an **HTTP/RPC worker thread** (`b-httpworker.N`),
  whose 128 KB musl stack the prover frame overflows → SIGSEGV (the exit-139 the 3 agents saw).
- **Severity: MED — sender/wallet-local ONLY. NOT remote-reachable.** The mirrored consensus/network
  verifier `smile2::VerifyCT` has a frame **half the size** (~64 KB ASan vs ~128 KB ASan for the prover)
  and stays within the 128 KB worker stack, so a poison block does NOT reproduce this. The crash is
  confined to the local prover (the sender builds its own proof). Re-confirms c4's "verify path robust".
- **FIXED**: run the prover on a dedicated thread with an ample (8 MB) explicit stack. Minimal, confined
  to one call site (`CreateSmileProof`), touches no crypto logic, fixes the bug for ALL prover callers
  regardless of calling-thread stack/libc. **VERIFIED**: z→z now succeeds (note spent + new note
  created, tx mined), node stays up, ASan-clean (0 errors), `verifychain` = true. **Unblocks v1 unshield
  + the c5/e3 empirical tamper tests.**

Key facts in the trace:
- Bug class = **stack overflow**. The fault `pc` is at the **prologue** of `TryProveCT`
  (`ct_proof.cpp:2215` = the function's opening brace), where the compiler's stack-probe loop touches
  the guard page — the classic signature of *"this function's frame is bigger than the thread stack"*,
  NOT an out-of-bounds index, use-after-free, or uninitialized read.
- The faulting thread is **T8 = `b-httpworker.5`**, created by `StartHTTPServer()` (httpserver.cpp:610).
  HTTP workers are plain `std::thread`s → **OS-default stack** → 128 KB on musl.

### Frame-size evidence (disassembly of the ASan binary)
`TryProveCT` prologue does a page-by-page stack-clash probe:
```
sub  $0x20000,%r11     ; probe target = current rsp − 0x20000 (131072 = 128 KB)
.L:  sub $0x1000,%rsp ; movq $0,(%rsp) ; cmp %r11,%rsp ; jne .L
sub  $0x2c0,%rsp       ; + 704 bytes  ⇒ total frame ≈ 131,776 B ≈ 128.6 KB
```
`smile2::VerifyCT` (the consensus/verify mirror, reachable from `VerifySmile2CTFromBytes` on the network
path) probes only `sub $0x10000,%r11` ⇒ **~64 KB** ASan frame — half the prover, comfortably inside the
128 KB worker stack. `CreateSmileProof` itself is ~480 B (no probe loop). This is why the **prover**
overflows but the **verifier** does not → local-only severity.

## Root cause (file:line + bug class)
- **`src/shielded/smile2/ct_proof.cpp` : `smile2::TryProveCT` (def starts line 2207; fault attributed to
  its prologue at line 2215).** Bug class = **stack-frame-size / stack-overflow** (a memory-safety bug,
  but of the "frame too big for the thread stack" kind — not OOB/UAF/init).
- The frame is large because `TryProveCT` is a single ~840-line function that holds many large
  by-value lattice temporaries simultaneously: `SmilePoly` = `std::array<int64_t,128>` = 1032 B,
  `NttForm` = `std::array<NttSlot,32>` = 1024 B, and a `SmileCTProof proof;` aggregate, plus scattered
  `SmilePoly`/`NttForm` results of `NttMul`/`NttForward`. At `-O1` (and worse under ASan redzones) the
  compiler does not coalesce these across the function's many scopes ⇒ ~128 KB frame.
- **Why z_shieldfunds works but z→z crashes (confirms c5/e3):** `CreateSmileProof` → `TryProveCT` is
  reached **only** when `has_shielded_spends` is true (`v2_send.cpp:559`, inside `if (use_smile)`).
  `z_shieldfunds` shields transparent→shielded with **no shielded spends** (`SendProofMode::NONE`,
  v2_send.cpp:529) and never calls the prover. Only a real shielded **spend** (z→z) invokes the
  giant-frame prover, on the HTTP worker thread.

## Severity (local vs remote-reachable) — MED, local only
- **Sender/wallet-local prover only.** The sender builds its own proof; nothing about a peer's data
  reaches `TryProveCT`. No inflation/forgery/consensus effect.
- **NOT a remote poison-block crash.** The mirrored verifier `VerifyCT` (consensus path, reachable from
  network via `VerifySmile2CTFromBytes`) has a ~64 KB frame that fits the 128 KB worker stack. I
  re-checked the mirror specifically (per brief): it does NOT share the prover's overflow. c4's "verify
  path robust by construction" holds. (Note: had VerifyCT's frame been larger, this WOULD have been a
  remote-reachable CRIT — worth keeping the verifier frame bounded as a standing invariant.)
- Net effect on the shipped musl/Alpine build: shielded **sends are non-functional** and the node
  **crashes** (DoS-of-self) on any z→z — exactly what c5/e3/e3 reported (exit 139, not OOM). On a glibc
  build (8 MB worker stacks) it would not crash, but BTX ships the musl image.

## The fix (unified diff)
Run the heavy-frame prover on a dedicated thread with an explicit 8 MB stack. One helper +
one wrapped call in `CreateSmileProof` (the single entry that calls `TryProveCT`). No crypto/logic
change; fixes every prover caller (RPC worker, wallet, scheduler) regardless of libc/stack.

+template <typename Fn>
+auto RunOnAmpleStack(Fn&& fn) -> std::invoke_result_t<Fn>
+{
+    using Result = std::invoke_result_t<Fn>;
+
+    struct Ctx {
+        Fn* fn;
+        std::optional<Result> result;
+        std::exception_ptr error;
+    } ctx{&fn, std::nullopt, nullptr};

+        auto* c = static_cast<Ctx*>(arg);
+        try {
+            c->result.emplace((*c->fn)());

+            c->error = std::current_exception();

+        // glibc the caller stack is large enough anyway.

Notes on correctness: the helper joins before returning (no concurrent access to the by-ref
`prove_error`/captured state), forwards exceptions across the thread boundary, and falls back to inline
execution if thread creation fails (correct on glibc). `VerifyCT` is intentionally left on the caller
thread (its ~64 KB frame fits 128 KB), so no extra thread is spawned for the verify step.

### Alternative fixes considered (and why this one)
- Enlarging the HTTP worker stacks in `StartHTTPServer()` would fix z_sendmany but not other prover
  callers and requires raw-pthread plumbing into Core's worker creation; it also leaves the latent
  large-frame hazard for any future caller. The chosen wrapper fixes ALL prover entry points centrally.
- Reducing `TryProveCT`'s frame (heap-allocating the many temporaries) is the "purest" reduction but is
  diffuse (dozens of sites) and risks perturbing the consensus-mirrored crypto — higher risk for no
  extra benefit. The wrapper is the minimal, low-risk, fully-effective fix.

## Refuted / robust (so others don't repeat)
- It is NOT an OOB read/write, use-after-free, or uninitialized read, and NOT OOM — it's a pure stack
  overflow from an oversized function frame on a small (musl-default 128 KB) thread stack. Extensive
  static review of the prover/serializer/BDLOP/wallet-assembly (all heavily bounds-guarded) found no
  OOB — consistent with this being a frame-size, not an indexing, bug.
- The consensus VerifyCT path does NOT share this crash (frame fits) ⇒ no remote poison-block CRIT here.

## Open questions / leads
- Standing invariant worth a guard/CI check: keep `VerifyCT`'s (and any consensus-reachable lattice
  verifier's) stack frame well under 128 KB so the musl worker stack never becomes a remote crash
  surface; consider `-Werror=frame-larger-than=N` on the shielded TU as a tripwire.
- Optional defense-in-depth (orthogonal to this fix): also reduce `TryProveCT`'s frame (heap the big
  temporaries) so it fits a 128 KB stack natively — would protect against any future caller that bypasses
  the wrapper. Not required given the wrapper covers the one prover entry point.
