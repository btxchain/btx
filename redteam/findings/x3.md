## 1. PRIVACY / LINKABILITY (vector 1)

### 1.1 The structural blocker (THE finding) ‚Äî direct unshield is unrepresentable post-fork
Code chain (all in the isolated pr-214 tree, identical to working tree):
- `RejectDirectSendEncodingForFork` (`shielded/validation.cpp`): post-fork, a payload with
  `!spends.empty()` MUST use `SendOutputEncoding::SMILE_COMPACT_POSTFORK` else
  `bad-shielded-v2-send-encoding`.
- `SendOutputEncodingElidesValueBalance` (`v2_bundle.h:82-85`): SMILE_COMPACT_POSTFORK ‚áí elide.
- Serialize (`v2_bundle.h:687-690`): `if (!elide_value_balance) Serialize(value_balance)` ‚Äî for
  the compact encoding value_balance is **NOT written**.
- Unserialize (`v2_bundle.h:739-751`): elide ‚áí `value_balance = fee` (set equal to the parsed fee).
- `SendPayload::IsValid` (`v2_bundle.cpp:749-752`): `if (elides_value_balance && (!has_spends ||
  value_balance != fee)) return false;` ‚Äî **hard-requires value_balance == fee** for the compact
  encoding.
- Builder (`v2_send.cpp:421-438`): post-fork + has_shielded_spends ‚áí
  `use_postfork_compact_send_encoding=true` ‚áí `output_encoding=SMILE_COMPACT_POSTFORK`, while
  setting `payload.value_balance = transparent_out + fee` (`v2_send.cpp:507`).
‚áí The builder's in-memory `value_balance = transparent_out+fee` is **dropped on serialize**; the
  relayed/mined tx has `value_balance = fee`, so `transparent_out` must be 0 for the tx to be
  self-consistent. Any nonzero transparent output ‚áí the deserialized tx violates conservation
  (`CheckTxInputs` folds value_balance=fee, but vout sums to >0) OR the SMILE balance proof (built
  with public_fee = transparent_out+fee) mismatches the wire value_balance=fee ‚áí `bad-shielded-*`.
‚áí **A post-fork self-service unshield with a real transparent payout cannot be encoded.** The
  feature is not a wallet-policy unblock; it needs a fork. (Empirical confirmation in ¬ß4.)

### 1.2 IF an unshield encoding existed (fork) ‚Äî the de-anon residuals, quantified
The SMILE balance proof requires the TRUE public fee/value_balance in the clear (it is bound into
the FS transcript, `ct_proof.cpp:3211-3215`), so an unshield encoding MUST serialize value_balance.
Then, as a chain observer:
- **Amount fingerprinting ‚Äî exact, intrinsic, ~deterministic link for non-round values.**
  `value_balance` (= total unshielded) and each transparent `vout.nValue` are plaintext. For an
  OTC-style unshield of an unusual amount (e.g. 137.42119 BTX), the transparent payout exactly
  equals value_balance‚àífee; matching that value against the (public) set of recent shielded-pool
  inflows / a counterparty invoice de-anonymizes the unshielder with probability ‚âà1 when the amount
  is unique in the relevant window. Mitigations present but weak: fee is only **bucketed to a 1000-
  sat quantum** (`SHIELDED_PRIVACY_FEE_QUANTUM=1000`, `bundle.cpp:1136`) ‚Äî that hides the fee's low
  bits, NOT the payout. The shielded CHANGE note is hidden (good), but the *released* amount is not.
- **Timing correlation.** The unshield references the live tree root `m_tree.Root()` as its anchor
  (`shielded_wallet.cpp:4857,4888`), pinning it to "tree state ‚â• height H". If a wallet unshields
  shortly after receiving a z‚Üíz (the c5/e3/v1 z‚Üíz prover works intermittently), the two appear in
  adjacent blocks; the anchor + block proximity narrow the source set. Standard Zcash-class residual;
  the wallet has NO enforced broadcast delay, so the naive "unshield right after receive" pattern is
  fully exposed.
- **Anchor / tree-root reuse.** All sends built at the same tip share the same anchor ‚Äî does not
  uniquely fingerprint a spender (every honest spend does the same), LOW value.
- **Ring / decoy.** SMILE proves membership against the FULL tree snapshot (no decoy selection to
  attack ‚Äî e3); ring strength is fine. NOT a de-anon axis here.
- **Change-output linkage.** A pure unshield still REQUIRES ‚â•1 shielded output (`v2_bundle.cpp:716`
  `outputs.empty()` forbidden) ‚áí a mandatory shielded change note; its value is hidden, but its
  EXISTENCE + the spent nullifiers are visible. Linking change‚Üínext-spend is the usual shielded
  chain-analysis, not unshield-specific.
- **Transparent output's later spend.** Once unshielded, the t-output is a normal UTXO; its later
  spend links to whatever it funds ‚Äî intrinsic to ANY unshield (Zcash t-side identical), unfixable.
- **NET vs operator path:** the operator/bridge path leaks source‚Üîdest to the off-chain operator
  (v1 ¬ß1A). The *direct* path would leak the EXACT released amount + timing to the WHOLE WORLD on
  chain. For a unique-amount OTC unshield, public amount-fingerprinting can be a *stronger* de-anon
  than a single semi-trusted operator who at least doesn't publish. ‚áí v1's "clear privacy win" is
  overstated even in the hypothetical fork case: it trades operator-trust for permanent public
  amount/timing disclosure.

**De-anon rate estimate (hypothetical fork case):** for unshields of non-round amounts that match
a recent unique pool inflow or a known invoice ‚Äî **~100% linkable by amount alone**; for round/
bucketed amounts shared by many users ‚Äî drops toward 1/k of the same-amount anonymity set. No
mechanism in-tree forces denomination bucketing of the *payout* (only the fee), so an attacker
controls nothing ‚Äî the *unshielder's own amount choice* determines their anonymity. Practically:
**high de-anon for any non-standard amount; the feature provides no payout-denomination defense.**

## 2. DoS / AMPLIFICATION (vector 2)

### 2.1 The O(chain) rebuild trigger is on the unshield path, now permissionless
- Unshield = V2_SEND with spends; each spend carries `ring_positions` (attacker-chosen,
  `v2_send.cpp:516`, `V2SendSpendWitness::ring_positions`).
- At MemPoolAccept, `CShieldedProofCheck` ring assembly calls `tree.CommitmentAt(pos)`; an
  out-of-range `pos` ‚áí `bad-shielded-ring-member-position` (`v2_proof.cpp:2048-2056`).
- That reject is in `IsRecoverableShieldedProofReject` (`validation.cpp:1255-1264`, UNCHANGED from
  d3's analysis) ‚áí `validation.cpp:3731-3751` fires `TryAutoRebuildShieldedState(... "MemPoolAccept"
  ...)` = `RebuildShieldedState` = genesis‚Üítip block replay of every shielded bundle (the ~25-min
  production op d3 measured), **with NO PoW** ‚Äî just a mempool tx that clears min-relay + the
  shielded premium.
- Dedup `MarkShieldedAutoRepairAttempt` keys on <tree.Root, tree.Size, registry.Root> ‚áí 1 rebuild
  per distinct shielded state, but RE-ARMS on every new shielded-bearing block ‚áí ~1 free O(chain)
  CPU/IO stall per shielded block, each driven by one cheap unshield-shaped tx.

### 2.2 Verifier cost vs attacker cost ‚Äî amplification ratio
- Attacker cost per trigger tx: a min-relay-fee tx + `MIN_SHIELDED_RELAY_FEE_PREMIUM = 5000` sat
  (`policy.h:121`). No PoW, no valid proof needed past the cheap structural ring-position check.
  (Caveat from d3: the tx must pass the spend-auth check first if it carries shielded inputs ‚Äî a
  self-service unshield builder produces exactly such a structurally-valid-but-state-mismatched
  bundle, which is the point: the feature gives attackers a clean builder for this shape.)
- Verifier cost: a full `RebuildShieldedState` ‚âà d3's observed ~25 min on a production-length chain
  (O(chain_len √ó bundles)); on regtest it is small (short chain) ‚Äî quantified in ¬ß4 once measured.
- Amplification (production-scale): ~5000 sat ‚áí ~1500 s of single-core CPU + full block re-read IO.
  At ~0.0001 BTX min fee + 5000 sat premium the **cost/stall ratio is effectively unbounded**
  (a few cents ‚Üí a 25-minute node stall, repeatable once per shielded block). This is HIGH and is
  the *most exploitable* property the self-service feature would add (it removes the operator
  gatekeeper that today throttles who can submit such bundles).
- Mempool flooding: each distinct shielded-state allows one rebuild; an attacker who also mines/
  observes shielded blocks can land one trigger per shielded block ‚áí sustained ~100% stall on a
  chain with frequent shielded activity. Even without that, a single trigger per state is a cheap
  per-block griefing primitive.
- Note: the per-block *block-inclusion* shielded verify budget (`DEFAULT_MAX_BLOCK_SHIELDED_VERIFY
  _COST=240000`, ~2400 spends/block at 100 units/spend) does NOT bound the rebuild ‚Äî the rebuild is
  triggered at mempool-accept and is not charged against verify_units. So the existing DoS budget
  does not mitigate this lever.

### 2.3 Mitigation (inherits d3)
Restrict self-repair to block-connect/startup (already-PoW'd) only; bounded incremental re-sync
from the last verified checkpoint; drop "position absent" rejects from the mempool recoverable set
(a non-existent ring position is peer misbehavior, not local-state corruption). MUST land before
any unshield feature (which would otherwise hand this trigger to every user).

- ‚òÖ FOLLOW-UP NEEDED to fully resolve: an **MSan or Valgrind** run (catches uninitialized in-bounds

  Under ASan alone, the bug presents as latent.

### 3.0 Static (live path, for the auditor)
The live (rec_levels==1, N‚â§32) prover path in `TryProveCT`/`CreateSmileProof` has consistent vector
sizing (`ComputeNumAuxMsg` == `CtAuxLayout::TotalSlots()` for the m1 layout; aux-slot writes are
guarded; ring builders all produce N rows; `secret_index<N` checked). No deterministic OOB on the
live path. Candidate uninitialized-read sites for MSan: `BuildCtCombinedSelectedAccountRows`
(`ct_proof.cpp:1448-1459`); the per-input recursion/aux assembly; the wallet anon-set/witness

### 3.2 Verifier (VerifyCT) ‚Äî re-checked given the confirmed prover bug
`VerifyCT` (`ct_proof.cpp:3140-3200`) explicitly validates EVERY attacker-supplied dimension before
indexing: `z0.size()==num_inputs`, `input_tuples.size()==num_inputs`,
`output_coins.size()==num_outputs`, `aux_commitment.t0.size()>=MSIS_RANK`,
`coin_rings.size()==num_inputs`, per-ring `size()==N`, `account_rings` match. These are the
by-construction guards c4 cited. So the prover bug (operating on locally-trusted, well-formed data)
does NOT obviously mirror to a remote-reachable verifier OOB ‚Äî UNLESS ASan reveals the prover bug is
an uninitialized read of a field that the verifier path ALSO reads before its size guard. [To be
