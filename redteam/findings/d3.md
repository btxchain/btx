# d3 — Verification-cost amplification & resource exhaustion (catalog G4): MULTI/repeat dimension

Scope: go BEYOND the single-block DoS that b2/c1/c2 already cleared (Phase1 dim-pin gates
all matrix work; worst single-block reject ~1.3–3.8 ms at n=64, amplification <1). Target the
*repeat/state* dimension: (1) MatMul verification amplification across many headers/tips/IBD;
(2) index/state exhaustion incl. the shielded-commitment "rebuild from tip" (~25 min observed);
(3) mempool/orphan/relay; (4) disk/IBD/reorg verification cost. Method = code analysis on
`pr-214` + empirical regtest on a self-built isolated node (host static binary
`/tmp/btx-pr214-bin/btxd`, namespace `rtD3-`, ports 29472/29473, /tmp/rtD3, n=64 regtest).
Authoritative n=512 mainnet verify costs taken from the repo's own design benchmark
(`doc/btx-matmul-benchmarks.md`). No production/foreign assets touched; node stopped+datadir removed.

## Ground-truth cost model (sources, pinned)
- **Per-header gate is O(1).** Header relay (`HasValidProofOfWork`, validation.cpp:8259-8263) and
  `CheckBlockHeader` (8990-8004) call ONLY `CheckMatMulProofOfWork_Phase1` (pow.cpp:2006-2027):
  pure field comparisons + `DeriveTarget` + one `matmul_digest <= target` compare. **No
  `DeriveSigma`, no `noise::Generate`, no `SharedFromSeed`, no matrix alloc.** `DeriveSigma` itself
  (matmul_pow.cpp:247) is 2×SHA-256 (µs); it is reached only by the Phase2 *pre-hash gate*, never
  by header processing. ⇒ junk-header flooding does O(1)/header. (re-confirms a3/b2 for the header path.)
- **Expensive work is Phase2-only**, in `ContextualCheckBlock` (validation.cpp:8539-8590), for a
  FULL block: `SharedFromSeed`×2 + `noise::Generate` + transcript `CanonicalMatMul` (O(n³)) OR the
  Freivalds/ProductCommitted O(n²) path. Authoritative timings (design benchmark, n=512 = mainnet
  dim): **transcript verify 622 ms/block; Freivalds/ProductCommitted O(n²) ≈ few ms; noise::Generate
  3.4 ms; working set 3.2 MiB.** (regtest n=64: transcript ~2.9 ms, Freivalds 1.3–3.8 ms per b2.)
- **Matmul verification runs ONCE per block at AcceptBlock-time.** `ConnectBlock`
  (validation.cpp:5653-5665) explicitly does NOT re-run `ContextualCheckBlock`/Phase2 — confirmed by
  the in-code comment. ⇒ reorg/reconnect does NOT re-verify matmul. No reorg-driven matmul re-amplification.

## Vectors attempted

### 1. MatMul verification amplification across many headers / tips / IBD
- **Junk-header flood (many competing low-work tips, each "forcing" verification).**
  RESULT: REFUTED. Per-header cost is O(1) Phase1 (above). On TOP of that, header processing
  charges a designed **per-peer + global verification budget** for the Phase2 work the headers
  *imply*: `CountMatMulPhase2Checks` (pow.cpp:2385) → `ReserveMatMulVerificationSlot` (cap
  `nMatMulMaxPendingVerifications=16`) + `ConsumeMatMulVerificationBudgetForPeer` (net_processing.cpp
  3663-3694, 2080-2137). Over-budget ⇒ peer disconnected. Steady-state caps (mainnet, struct
  defaults consensus/params.h:146,147,164): **per-peer 32 Phase2/min, global 512 Phase2/min,
  ≤16 concurrent pending.** ⇒ a peer cannot drive unbounded verifications via headers.
- **Force the O(n³) transcript FALLBACK at mainnet n=512 (134M ops, 622 ms) cheaply.**
  RESULT: NOT REACHABLE at the tip on mainnet; reachable ONLY for honest historical h<61000 during
  IBD (no amplification). Dispatch trace (validation.cpp:8539-8590): the transcript fallback (8584)
  is reached only when `should_run_matmul_validation && !payload_required && no product payload &&
  not already verified`. `payload_required = fMatMulRequireProductPayload || product_digest_active`
  (params.h:307). At mainnet h≥61000 `product_digest_active` is true ⇒ `payload_required` true ⇒ a
  no-payload block is rejected CHEAPLY at 8565 ("missing-product-payload"); a payload block runs the
  O(n²) ProductCommitted/Freivalds and a bad one is rejected at 8572/8577 — **the 622 ms transcript
  is never reached for tip blocks.** Below h<61000 the transcript path runs, but a block can only
  reach `ContextualCheckBlock` if it passes Phase1 (`matmul_digest ≤ target`, i.e. REAL PoW on
  mainnet) AND clears AcceptBlock's anti-DoS gate (8852-8861: unrequested low-work / too-far-ahead /
  `< MinimumChainWork()` blocks are dropped BEFORE ContextualCheckBlock). ⇒ the 622 ms is paid once
  per legitimately-PoW'd block = same work the original miner did; defender:attacker ratio ≈ 1, not >>1.
- **IBD/catch-up budget relaxation.** NOTED (bounded). During IBD (or "catch-up": active tip >10
  behind best header, or peer-announced height > active+max(32,budget), net_processing.cpp 3644-3658)
  the per-peer budget jumps to **200,000/min** (pow.cpp:2462-2468) and the **global budget is bypassed**
  (net_processing.cpp:2126 `if (!is_ibd && !in_fast_phase)`). This is necessary so honest 2000-header
  IBD batches don't self-disconnect. It is NOT a cheap-attacker amplification because the implied
  Phase2 work still only executes on blocks that pass Phase1 PoW + the AcceptBlock work-gate; a peer
  feeding fake headers gets the headers rejected at Phase1 (junk digest > target) or never delivers a
  connectable block. The budget protects CPU; PoW protects the actual transcript execution.

### 2. Index / state exhaustion (shielded-commitment rebuild, txindex/coinstats/blockfilter)
- **Shielded full-chain "rebuild from tip" (`RebuildShieldedState`, validation.cpp:982-1031) — the
  ~25-min production operation — triggerable by a cheap unconfirmed tx.** PARTIAL FINDING (see below).
  `RebuildShieldedState` walks genesis→tip, `ReadBlock` + replays every shielded bundle ⇒
  O(chain_len × bundles). It is auto-invoked as a "self-repair" when a shielded proof check fails with
  a *recoverable* reason (`IsRecoverableShieldedProofReject`, validation.cpp:1255-1264:
  bad-shielded-proof / bad-shielded-ring-member-position / bad-shielded-ring-tree-unavailable /
  bad-smile2-ring-member-account[-leaf|-public-account] / bad-smile2-shared-ring). Trigger sites:
  **MemPoolAccept (3731-3751) — unconfirmed tx, NO PoW needed** — and ConnectBlock (6497-6520, needs a
  mined block). The `bad-shielded-ring-member-position` reject (v2_proof.cpp:2048-2056) fires on a
  CHEAP structural check of attacker-controlled `ring_positions` (`tree.CommitmentAt(pos)`==nullopt)
  that happens during ring assembly. ⇒ an attacker tx referencing a non-existent commitment position
  triggers a full-chain rebuild.
  DEDUP GUARD limits it: `MarkShieldedAutoRepairAttempt` (validation.cpp:11014-11040) keys on
  `<merkle_tree.Root, tree.Size, account_registry.Root>` and allows ONE rebuild per distinct shielded
  state. So repeated trigger txs at an unchanged state are skipped — but the guard **re-arms on every
  shielded-state change** (i.e. every new block that adds a shielded commitment). ⇒ amplification is
  ~1 free O(chain) rebuild per shielded-state-changing block, each driven by one cheap mempool tx.
  CAVEATS that lower severity / need a build to fully confirm: (a) the trigger tx must first pass
  min-relay + shielded-relay-premium fee (validation.cpp:3649-3660) and, if it carries shielded
  *inputs*, the spend-auth check (3704-3708) BEFORE reaching the ring-position reject — so a pure
  garbage-proof spend may be stopped earlier; reaching the recoverable reject likely needs a
  structurally-valid-but-state-mismatched bundle. (b) I did NOT execute the malicious-bundle repro
  (crafting a v2 bundle with out-of-range ring positions + passing earlier gates is deep shielded-
  format work owned by c4/c5; I confirmed the trigger/cost/dedup by code, not by a live rebuild).
  Severity assessed MED pending that confirmation; if the recoverable reject is reachable post-fee
  without expensive proof work, it is HIGH (cheap tx ⇒ 25-min CPU+IO stall, repeatable per shielded block).
- **Reorg-driven shielded recompute.** Empirical: built a shielded note (z_shieldcoinbase, pool
  39.999), then `invalidateblock` the shield block (3-block disconnect). DisconnectBlock shielded
  path (validation.cpp:5314-5424) uses the **fast `DetachToMemoryOnly`+`Truncate`** path
  (merkle_tree.cpp:510-578, rebuilds from nearest frontier checkpoint), NOT the O(chain) rebuild.
  Measured **25 ms** for the disconnect (tiny tree); shielded balance correctly 39.999→0→39.999 on
  reconsider. The O(chain) `RebuildShieldedState` is only the FALLBACK when Truncate/Detach fails
  (degraded/indexless in-memory state). ⇒ normal reorgs are bounded. NOTE (low): `DetachToMemoryOnly`
  reads the WHOLE tree into RAM (`ReadRange(0,size)`, merkle_tree.cpp:588-594) per disconnected
  shielded block — O(tree_size) RAM+IO per block; a deep reorg over a large shielded tree pays
  K×O(N). Bounded by needing a real higher-work competing chain (PoW), so not a cheap lever.
- **txindex / coinstatsindex / blockfilterindex.** Standard upstream incremental `CustomRewind`
  using block-undo data, O(reorg-depth) (index/base.cpp:247-265; coinstatsindex.cpp:268+ reads
  CBlockUndo). No O(chain) recompute on reorg, no attacker-driven unbounded growth beyond block
  content already weight-capped. Not the novel surface.

### 3. Mempool / orphan / relay
- Not separately re-fuzzed (c2 covered the deserializers/relay budgets; bounded). The one mempool
  amplification of interest is the shielded auto-rebuild trigger in #2 (a cheap mempool tx → O(chain)
  CPU/IO), which is the genuinely novel mempool→state amplification lever and is dedup-gated as above.

### 4. Disk / IBD / reorg verification cost
- Matmul verified once per block (no reconnect re-verify, §Ground-truth). IBD transcript cost
  (h<61000, 622 ms/block at n=512) is intrinsic honest-sync cost, not attacker-amplifiable beyond the
  budget+PoW gates (#1). Block disk size is weight-capped (b2: payload counts toward 24 MB; transport
  16 MB). No cheap disk-spam amplification found.

## NEW / unmitigated findings
- **[MED, needs build-confirm] Shielded self-repair rebuild is an attacker-triggerable O(chain)
  amplification.** mechanism: a recoverable shielded-proof reject (esp. `bad-shielded-ring-member-
  position`, a cheap structural check on attacker ring_positions) in **MemPoolAccept** auto-invokes
  `RebuildShieldedState` = full genesis→tip block replay (~25 min observed on production). repro
  (code-confirmed; live malicious-bundle repro not executed): submit a shielded tx whose ring
  references a commitment position absent from the tree; the node walks the whole chain before
  rejecting. dedup `MarkShieldedAutoRepairAttempt` caps it to 1 rebuild per shielded-state, but
  **re-arms on every new shielded-bearing block** ⇒ ~1 free O(chain) rebuild per shielded block via
  1 cheap tx. proposed mitigation: (a) NEVER run the O(chain) rebuild from an unauthenticated mempool
  tx path — restrict self-repair to block-connection (already-PoW'd) or startup only; (b) make the
  recoverable-reject self-repair a rate-limited, bounded *incremental* re-sync (replay only from the
  last verified checkpoint, not genesis), with a hard wall-clock/op budget; (c) treat a mempool tx
  that references a non-existent ring position as a plain peer-misbehavior reject (it cannot be the
  NODE's state that is wrong for a position the chain never created) — i.e. drop these reject reasons
  from `IsRecoverableShieldedProofReject` for the mempool trigger.

## Refuted / robust (so others don't repeat)
- **Junk-header / many-low-work-tip flood forcing matmul work** — REFUTED. Per-header = O(1) Phase1
  (no DeriveSigma/noise/matrix); implied Phase2 work is per-peer+global budget-capped (32/512 per min,
  16 pending) with disconnect on overflow.
- **O(n³) transcript fallback (622 ms @ n=512) reached cheaply** — REFUTED on mainnet tip: payload-
  required at h≥61000 rejects no-payload blocks cheaply (validation.cpp:8565); transcript only runs on
  honest historical h<61000 blocks that already carry real PoW + clear AcceptBlock work-gate
  (8852-8861). Defender:attacker work ≈ 1.
- **Reorg/IBD matmul RE-verification amplification** — REFUTED: matmul Phase2 runs once at
  AcceptBlock; ConnectBlock does not re-run it (validation.cpp:5653 comment).
- **Normal reorg forcing O(chain) shielded rebuild** — REFUTED empirically: DisconnectBlock uses the
  fast Truncate/frontier-checkpoint path (25 ms measured for a 3-block shielded disconnect); O(chain)
  rebuild is only a degraded-state fallback.
- **txindex/coinstats/blockfilter reorg recompute** — bounded (incremental undo-based rewind).
