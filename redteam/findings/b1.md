# b1 — MatMul PoW soundness / Freivalds false-positive forgery (crown jewel B1/B4)

Scope: can an attacker produce a block that PASSES MatMul verification WITHOUT doing
the real O(n^3) matrix multiply (counterfeit PoW at ~zero cost)? Analyzed the full
derivation chain in src/pow.cpp + src/matmul/{freivalds,noise,transcript,matmul_pow,field,matrix}.cpp
and ran an empirical model of the consensus verification (pure-Python reimplementation,
/tmp/rtB1/freivalds_model.py + forge_fast.py).

## Bottom line
**Forgery is NOT feasible. The design is sound.** The Freivalds challenge vector r is
bound to the claimed product C' (Fiat-Shamir), so the "grind nonce -> pick fake C'"
attack the catalog hypothesizes does not work: the attacker cannot see r before
committing C', and any C' != A'B' fails each round with probability 1/p = 2^-31, giving
~2^-62 forgery resistance with k=2 rounds. Independently, C' is committed in the header
via matmul_digest (which is in GetHash). No working forgery; documenting WHY + residual
hardening ideas.

## Vectors attempted
- **Grind-nonce -> fake-C' Freivalds forgery (the hypothesized B1 attack)**: model the
  regtest consensus path (n=64, b=8, noise_rank=4, k=2 rounds, p=2^31-1). Fix a cheap fake
  C' (single-entry flip of the honest product, OR all-zero = zero matmul work), grind a
  free header field (nonce -> sigma) and run the real Freivalds verify each attempt.
  Commands: `cd /tmp/rtB1 && python3 -u forge_fast.py`
  RESULT: REJECTED — 0 forged passes / 100000 grind attempts (flip-1), 0 / 100000 (all-zero).
  Each grind attempt is a *fresh independent* (A',B',C',r) instance because sigma reseeds
  A',B' (noise) AND C' is hashed into the challenge seed. Grinding gives the attacker no
  algebraic advantage; it is brute-forcing a ~2^-62 event per attempt.
- **Fiat-Shamir binding probe** (the crux): take honest C'=A'B', flip ONE entry, observe
  whether the challenge seed / r change.
  RESULT: REJECTED forgery — challenge seed CHANGES and r CHANGES when C' flips; honest C'
  passes k=2, flipped C' fails k=2. (`/tmp/rtB1` inline test). The attacker cannot choose
  r first and then place C'-A'B' in r's kernel: doing so re-derives a different r.
- **Per-round false-positive measurement**: 300000 random nonzero linear forms <d,r> over F.
  RESULT: 0/300000 hits == 0 (matches theory 1/p = 4.66e-10 = 2^-31). Confirms per-round FP
  bound. r entries are uniform over [0,p): Pr[r_i==0] measured 0/1,280,000 (~1/p).
- **Reduced-work / low-rank shortcut (B4)**: examined whether the rank-r noise lets A'B' be
  computed in < O(n^3). A'B' = AB + AF + EB + EF. A,B are full-entropy from FromSeed
  (field::from_oracle per entry => full rank w.h.p.), so AB is a genuine dense×dense product
  requiring O(n^3). The AF/EB/EF terms are all rank<=r => O(n^2·r). The irreducible work is
  AB; the low-rank `Denoise` routine only recovers AB from a *known* A'B' (helps a verifier
  who already has the product), it does NOT let a miner produce A'B' cheaply.
  RESULT: REJECTED — no sub-cubic shortcut from the noise structure.

## Derivation chain of the challenge (exact)
1. `ComputeMatMulHeaderHash(header)` = SHA256 over { nVersion, hashPrevBlock, hashMerkleRoot,
   nTime, nBits, nNonce64(8), matmul_dim(2), seed_a(32), seed_b(32) }.
   NOTE: matmul_digest is **deliberately excluded** here (so the digest can be set to match
   C' without perturbing sigma — see "Is C' bound").
2. `DeriveSigma(header)` = SHA256(header_hash). Fully determined by grindable fields
   (nNonce64, seeds, time, dim, etc.).
3. `noise::Generate(sigma, n, r)` -> E_L,E_R,F_L,F_R via DeriveNoiseSeed(tag,sigma) +
   from_oracle. A' = SharedFromSeed(seed_a) + E_L·E_R ; B' = SharedFromSeed(seed_b) + F_L·F_R.
4. **Freivalds challenge (the key defense)** — freivalds.cpp:
   `challenge_seed = SHA256( 0xF4 || sigma || H(A') || H(B') || H(C') )`  (DeriveChallengeSeed
   hashes the FULL contents of A', B', AND C').
   `r_round = from_oracle( SHA256(0xF4||challenge_seed||round||n), i )`.
   Each round multiplies B'·r, A'·(B'r), C'·r and checks A'(B'r) == C'r.

The catalog's worry was "is r FULLY determined by grindable block data such that the
attacker can search (nonce -> r) so a chosen fake C' passes." Answer: r IS determined by
block data, but **r is also a function of C' itself** (step 4 hashes C'). So the attacker
cannot decouple "choose C'" from "know r": any change to C' re-randomizes r. This is the
standard Fiat-Shamir transform and it closes the attack. (Confirmed: only one r-derivation
exists in consensus code; DeriveRandomVector is only ever called with the C'-bound
challenge_seed — grep in src/, no path derives r from raw sigma.)

## Is C' bound? (two independent locks)
1. **Freivalds binding (algebraic):** A'B'==C' checked probabilistically; C' is hashed into r.
   Error <= (1/p)^k. Regtest/mainnet k=2, p=2^31-1 => **error <= 2^-62**.
2. **Header commitment (digest):** when product-digest is active (regtest height 0; mainnet
   height 61,000), `ComputeProductCommittedDigest(C', b, sigma)` must equal block.matmul_digest,
   and matmul_digest IS serialized in CBlockHeader / GetHash (block.h:53). So C' is committed
   to chain identity — a relay attacker cannot swap C' post-mining without changing GetHash
   (and thus PoW linkage). The digest also binds sigma and dim/b (PRODUCT_DIGEST_TAG v3).
   Below the digest height, payload-carrying blocks must additionally pass the full O(n^3)
   Phase2 transcript check (pow.cpp:2229-2241), so C' is bound there too.

## Effective bits of forgery resistance
- **2^-62** per block-creation attempt (k=2 rounds, p=2^31-1). To forge ONE block by luck an
  attacker needs ~2^62 fresh attempts; each attempt still requires constructing A',B',C' and
  running the verify, i.e. it is strictly more expensive than honest mining of the same block.
  There is no shortcut, no grindable bias, no domain-separation collision (tags 0xF4/0xA1/0xB2/
  0xC3 separate the matrix hashes; noise/compress/digest use distinct string tags).
- Domain separation checked: noise tags "matmul_noise_{EL,ER,FL,FR}_v1", compress
  "matmul-compress-v1", digest "matmul-product-digest-v3", Freivalds 0xF4 — all distinct; no
  cross-protocol reuse of sigma that would let one oracle's output be replayed as another's.

## NEW / unmitigated findings
- **(low / defense-in-depth) k=2 rounds gives 2^-62, not a comfortable 2^-80+ margin.**
  2^-62 is cryptographically safe for single-block forgery, but it is a *per-attempt* bound and
  the constant is fixed network-wide. Mitigation: bump nMatMulFreivaldsRounds to 3 (=> 2^-93)
  for headroom; cost is one extra O(n^2) pass, negligible vs O(n^3) mining. Not urgent.
- **(info) sigma excludes matmul_digest by design.** Correct and necessary (otherwise digest and
  sigma would be circular). Flagged only so future changes don't "fix" it and break mining. No action.

## Refuted / robust (so others don't repeat)
- **B1 Freivalds false-positive forgery via nonce grind — DOES NOT WORK.** C' is hashed into the
  challenge seed (Fiat-Shamir); flipping any C' entry re-derives r (empirically confirmed:
  honest C' passes, single-flip C' fails). 0 forged passes in 200k+ grind trials. Resistance 2^-62.
- **B4 low-rank / sub-cubic shortcut — DOES NOT WORK.** A,B are full-rank from per-entry oracle;
  A'B' decomposes as dense AB (O(n^3), irreducible) + 3 low-rank terms. Denoise helps a verifier,
  not a forger.
- **C' swap on the wire — DOES NOT WORK** when digest active: matmul_digest commits C' and is in GetHash.

## Open questions / leads for next wave
- b2 (verification-asymmetry DoS): the forgery is closed, but `IsMatMulFreivaldsPayloadSizeValid`
  accepts any matmul_dim in [min,max] with an n^2 payload; a malformed-but-cheap large-dim block
  could still cost the verifier the noise generation + Freivalds n^2 work before rejection. That's a
  DoS/asymmetry question (b2's vector), not a soundness break — hand off.
- Whether mainnet pre-61000 history has any payload-carrying blocks that relied only on Freivalds
  (without the transcript binding) is a historical-audit question; in code the pre-digest path forces
  Phase2 for payload-carrying blocks, so soundness holds. (reason-from-code only; not regtest-tested.)

## Method note
Empirical results are from a faithful Python reimplementation of the consensus math
(/tmp/rtB1/freivalds_model.py mirrors field::from_oracle, DeriveChallengeSeed, DeriveRandomVector,
Verify) rather than the C++ node, because the decisive property (C'->r binding, per-round 1/p) is in
the math, not the node plumbing. The reimplementation reproduces the exact byte layout
(uint256 internal-LE handling, LE32 element encoding, SHA256 tags). A full submitblock forgery
against btx-node:pr214 regtest was not attempted because the analysis shows the crafted block would
be rejected at Freivalds verify with probability 1-2^-62 (no point grinding 2^62 attempts); stated
explicitly per the "say so" rule. No production node / off-limits container touched; all work in
/tmp/rtB1.
